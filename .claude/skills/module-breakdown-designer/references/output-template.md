## 列定义

### 模块编号
- 格式：`[类别前缀][两位数字]`
- 前缀规则：2-4 个大写字母，代表功能域。
- 示例：`USR-01`（用户）、`BILL-05`（计费）、`INF-03`（基础设施）。
- 必须在整份文档中唯一。

### 模块名称
- 仅限名词短语。不含动词，不含连词。
- 良好示例：`用户画像`、`订单退款`、`Token 路由`
- 不良示例：`用户注册与登录`（应拆分为 `用户注册` 和 `用户登录`）、`处理订单并通知`（含动词和连词）

### 核心功能
- 严格一句话。
- 描述**做什么**，而非**怎么做**。
- 良好示例：`管理用户在系统中的身份凭证与登录会话。`
- 不良示例：`使用 JWT 实现用户登录。`（描述的是怎么做，而非做什么）

### 颗粒度描述
- 1-3 条具体、可测试的行为或约束。
- 必须包含至少一个可量化元素（数字、阈值、上限、超时时间）。
- 良好示例：`支持密码/OTP/SSO 三种方式；会话有效期 24h；失败 5 次触发锁定。`
- 不良示例：`提供多种登录方式。`（不可测试）

### 模块类型
- 取值：`🔴 核心` 或 `🟢 一般`
- **`🔴 核心`**：项目特色或差异化功能；设计文档描述不够清晰、存在多种实现可能；涉及复杂业务规则、状态机或关键算法决策；位于用户关键路径上，出错成本高；与多个模块紧密耦合的枢纽模块；技术选型对项目架构有重大影响。
- **`🟢 一般`**：行业内常见标准实现（如标准 CRUD、常见认证流程）；设计文档描述清晰，实现路径唯一；存在成熟开源方案或框架可直接使用；不涉及项目核心差异化竞争力。
- **占比约束**：`🔴 核心` 模块通常应占总模块数的 **30-40%**。若超过 50%，需重新审视判断标准是否过松；若低于 15%，需检查是否遗漏了真正需要严格对齐的模块。

### 设计文档溯源
- 格式：`[文档名] §[章节编号或标题]`
- 如果源自多份文档，全部列出：`[文档 A] §3.1；[文档 B] §5.2`
- 如果是行业补充，使用：`🔄 行业补充 — [简要理由]`
- 如果是从代码推断而非文档，使用：`🔧 代码推断 — [文件路径]`

## 章节标题规则

1. **H2 (`##`) 用于功能域**：**必须**使用 `## [两位数字]-[分组名]` 格式（如 `## 01-用户域`、`## 05-管线容错与恢复`）。两位数字从 `01` 开始递增，直接对应 `directory-convention.md` 的单模块目录路径。
2. **H3 (`###`) 用于子域**：仅当功能域包含 >8 个模块时使用。子域名称保持简短。
3. **分隔线**：在每个 H2 功能域之间插入 `---` 以增强视觉清晰度。
4. **排序**：业务功能域在前，基础设施/运维域在后。

> **硬性规则**：禁止使用中文数字（`一、二`）、罗马数字（`I、II`）或单位数（`1、2`）作为章节序号。违反此规则的产物将被 directory-convention.md 判定为不合格。

## 边界清晰规则

1. **禁止子集关系**：如果模块 A 的全部职责都被模块 B 覆盖，将 A 合并入 B，或拆分 B 直到 A 成为独立项。
2. **禁止共享所有权**：每个用户可见的结果必须有且仅有一个归属模块。如果两个模块都声称负责，重绘边界。
3. **禁止名称含连词**：将 `A与B` 拆为两个模块。例外：如果 A 和 B 紧密耦合到永远无法独立实现或独立部署，则保留为一个模块，使用单一概念名称。
4. **独立可估算性**：每个模块应当是一个工程师或一对工程师在计划会议上能够独立估算的，无需理解另一个模块的内部细节。
5. **逻辑/表现分离**：任何模块不得同时承担"逻辑处理"（数据获取、状态管理、业务计算、API 调用）和"视觉表现"（UI 渲染、样式、布局、动画）的职责。若模块的核心功能描述涉及两者，必须拆分为独立的逻辑层模块和表现层模块。前端项目中，hooks/services/stores 属于逻辑层，components/pages 属于表现层。

## 附录规则

### 附录 A（排除项）
- 仅列出该项目类型通常会有、但本项目明确排除的模块。
- 每项需附一句话理由。

### 附录 B（补充项）
- 列出每个标记为 `🔄 行业补充` 的模块。
- 每项需附一句话理由，解释为什么设计文档遗漏了它。

### 附录 C（边界决策）
- 记录每个非显而易见的边界选择。
- 防止未来的审阅者试图重新合并或重新拆分模块。

### 附录 D（模块类型判断依据）
- **仅列出 `🔴 核心` 模块**。一般模块的判断标准统一（行业常见实现、设计文档清晰、有成熟方案），无需逐条赘述。
- 每条核心模块的理由必须引用判断标准中的具体条目，并结合该模块的特征说明，禁止空泛描述。
- 良好示例：`对应检查清单第 2 条：实现方式可能产生分歧（设计文档仅描述"智能推荐"，未明确是基于协同过滤还是内容推荐，两种方案差异显著）。`
- 不良示例：`对应检查清单第 1 条：去掉后项目核心价值受损（...）。`（空泛，未结合模块特征说明具体受损点）
