## 1. 前端框架

| 维度 | React | Vue | Angular | Svelte |
|------|-------|-----|---------|--------|
| 生态与招聘 | ★★★★★ 最广泛 | ★★★★☆ 国内极多 | ★★★☆☆ 企业级为主 | ★★☆☆☆ 小众 |
| 学习曲线 | 中等（JSX + Hooks） | 平缓（模板语法） | 陡峭（TypeScript 强制 + RxJS） | 平缓（编译时优化）|
| SSR/SSG 方案 | Next.js / Remix | Nuxt.js | Angular Universal | SvelteKit |
| 类型安全 | 强（TS 首选） | 良好 | 极强（TS 内置） | 良好 |
| 适用场景 | 大型应用、生态复杂 | 中小型、快速交付 | 企业级、强规范 | 性能敏感、精简项目 |

**决策要点**：
- 团队已有 React 经验 → 优先 React + Next.js（需 SSR 时）
- 交付周期紧、团队 junior 多 → Vue 3 + Vite
- 强类型约束、银行/政企 → Angular

## 2. 后端语言与框架

| 维度 | Python/FastAPI | Node.js/NestJS | Go/Gin | Java/Spring Boot |
|------|---------------|----------------|--------|------------------|
| 开发效率 | ★★★★★ | ★★★★★ | ★★★☆☆ | ★★★☆☆ |
| 运行时性能 | ★★★☆☆ | ★★★☆☆ | ★★★★★ | ★★★★☆ |
| 异步原生 | ★★★★★ asyncio | ★★★★★ EventLoop | ★★★★★ goroutine | ★★★☆☆ WebFlux |
| 类型安全 | 良好（Pydantic） | 良好（TS） | 极强（静态编译） | 极强 |
| 部署体积 | 大（含运行时） | 中 | 极小（单二进制） | 大（JVM）|
| AI/数据生态 | ★★★★★ | ★★★☆☆ | ★★☆☆☆ | ★★★☆☆ |
| 团队招聘 | 广泛（AI/数据） | 广泛（全栈） | 中等（云原生） | 极广（企业） |

**决策要点**：
- 需要 AI 集成（LLM、向量、ML）→ Python/FastAPI
- 前后端同构、快速 MVP → Node.js/NestJS
- 高并发、低延迟、微服务 → Go/Gin
- 企业级、复杂领域模型、长期维护 → Java/Spring Boot

## 3. 关系型数据库

| 维度 | PostgreSQL | MySQL | SQLite |
|------|-----------|-------|--------|
| 高级特性 | JSONB、全文检索、GIS | 主从成熟、社区广 | 零配置、嵌入式 |
| 事务与一致性 | ★★★★★ | ★★★★☆ | ★★★☆☆ |
| 扩展性 | 良好（分片需 Citus） | 良好（TiDB 兼容） | 极差 |
| 运维复杂度 | 中等 | 较低 | 无 |
| 适用场景 | 复杂查询、JSON 字段、地理信息 | 通用 Web、国内云厂商首选 | 测试、嵌入式、极小应用 |

**决策要点**：
- 默认推荐 PostgreSQL（功能全面、开源友好）
- 国内云环境强依赖 RDS → MySQL 8.x
- 嵌入式/单元测试 → SQLite

## 4. 缓存与 NoSQL

| 维度 | Redis | MongoDB | Elasticsearch |
|------|-------|---------|---------------|
| 核心能力 | KV / 队列 / 分布式锁 | 文档型 / 灵活模式 | 全文检索 / 日志分析 |
| 持久化 | AOF/RDB（非主力持久化） | 原生持久化 | 原生持久化 |
| 查询复杂度 | 简单（K/V、ZSet） | 中等（聚合管道） | 复杂（DSL） |
| 运维成本 | 低（单实例/哨兵/集群） | 中等（副本集/分片） | 高（集群调优） |

**决策要点**：
- 会话、缓存、限流、排行榜 → Redis（几乎必用）
- 非结构化数据、快速迭代 schema → MongoDB
- 搜索、日志、监控 → Elasticsearch（或轻量替代 Meilisearch）

## 5. 消息队列与异步

| 维度 | Redis Stream | RabbitMQ | Kafka | MQ 云服务 |
|------|-------------|----------|-------|-----------|
| 吞吐量 | 中 | 中 | ★★★★★ | 依厂商 |
| 可靠性 | 中 | ★★★★★ | ★★★★★ | ★★★★★ |
| 运维复杂度 | 极低 | 中等 | 高 | 低（托管）|
| 延迟 | 低 | 低 | 中（高吞吐优先）| 低 |
| 适用规模 | 小型/中型 | 中型 | 大型/日志流 | 全规模 |

**决策要点**：
- 项目初期、吞吐 < 1k/s → Redis Stream（复用已有 Redis）
- 需要路由、死信、延迟队列 → RabbitMQ
- 大数据、日志流、事件溯源 → Kafka
- 无专职运维 → 云厂商托管 MQ（阿里云 RabbitMQ、AWS SQS 等）

## 6. 实时通信

| 方案 | 复杂度 | 兼容性 | 适用场景 |
|------|--------|--------|----------|
| WebSocket | 中（需连接管理） | 现代浏览器 | 双向实时（聊天、协作）|
| SSE (Server-Sent Events) | 低 | 良好 | 服务端单向推送（通知、流式 AI）|
| 长轮询 | 低 | 极佳 | 兼容性要求极高的 legacy 场景 |
| MQTT | 中 | IoT 设备 | IoT、移动端弱网 |

**决策要点**：
- 服务端推送为主（AI 流式输出、通知）→ SSE（实现简单、自动重连）
- 双向实时（聊天、协同编辑）→ WebSocket
- 不需要实时 → 普通 HTTP 轮询即可

## 7. 认证鉴权方案

| 方案 | 实现成本 | 安全等级 | 用户体验 | 适用场景 |
|------|----------|----------|----------|----------|
| JWT (自研) | 低 | 中 | 好 | 独立应用、无外部集成 |
| Session + Cookie | 低 | 中 | 好 | 传统 Web、同域前端 |
| OAuth2 / OIDC | 中 | 高 | 好 | 社交登录、第三方集成 |
| LDAP / AD | 中 | 高 | 一般 | 企业内网、统一身份 |
| SSO (CAS/SAML) | 高 | 高 | 一般 | 政企、多系统统一登录 |

**决策要点**：
- 面向 C 端用户、需微信/QQ/谷歌登录 → OAuth2
- 企业内网、已有 AD/LDAP → LDAP/SSO
- 独立 SaaS、快速启动 → JWT + Refresh Token 机制
- 高安全场景（金融、政务）→ 短时效 JWT + 服务端黑名单/Redis

## 8. 部署与运维

| 方案 | 启动成本 | 弹性扩展 | 运维负担 | 锁定风险 |
|------|----------|----------|----------|----------|
| Docker Compose | 极低 | 手动 | 低 | 无 |
| K8s (自建) | 高 | ★★★★★ | 极高 | 无 |
| K8s (托管) | 中 | ★★★★★ | 中 | 云厂商 |
| Serverless (FaaS) | 极低 | 自动 | 极低 | 高 |
| PaaS (Heroku/Render) | 低 | 中 | 低 | 高 |
| 传统服务器 | 低 | 手动 | 中 | 无 |

**决策要点**：
- 1-3 人团队、无专职运维 → Docker Compose / PaaS / Serverless
- 需要自动扩缩容、微服务 → 托管 K8s（阿里云 ACK、AWS EKS）
- 预算极低、无流量波动 → 传统服务器 + Nginx
- 快速验证 MVP → Serverless（注意冷启动与 Vendor Lock-in）

## 9. AI / 向量能力

| 维度 | 自部署开源模型 | 云 API (OpenAI/ Claude/ 国产) | 私有化部署 |
|------|---------------|------------------------------|------------|
| 成本 | 硬件高、调用免费 | 按 Token 计费 | 硬件一次性 + 维护 |
| 数据隐私 | 高 | 低（传输出境风险）| 极高 |
| 延迟 | 本地低、远程中 | 依赖网络 | 本地低 |
| 可控性 | 高 | 低 | 高 |
| 适用 | 实验、敏感数据 | 快速集成、通用场景 | 金融、政务、医疗 |

**向量数据库选型**：
- 轻量/本地开发 → pgvector（PostgreSQL 插件）
- 专业向量检索 → Milvus、Pinecone、Qdrant、Weaviate
- 已有 Elasticsearch → 使用其向量检索能力

**决策要点**：
- 验证阶段 → 云 API（最快）
- 数据敏感/合规要求 → 私有化 / 自部署
- 向量规模 < 100万 → pgvector 足够
- 大规模、需要混合搜索 → Milvus / Qdrant

## 10. 架构模式

| 模式 | 复杂度 | 团队规模要求 | 适用阶段 |
|------|--------|-------------|----------|
| 单体 (Monolith) | 低 | 1-5 人 | MVP、初创 |
| 模块化单体 | 中 | 3-8 人 | 成长期、需代码隔离 |
| 微服务 | 极高 | 15+ 人 + 平台团队 | 大规模、多团队并行 |
| Serverless / BaaS | 低 | 1-3 人 | 事件驱动、轻量业务 |

**决策要点**：
- 默认推荐模块化单体（单仓库 + 清晰模块边界），便于后续拆分
- 微服务不是目标，是团队规模与发布频率的副产品
- 如无 CI/CD、监控、服务发现基础设施，不要选微服务
