# 函数签名与契约提取参考

本文档指导 orchestrator 如何从落地规范和设计文档中提取接口契约，作为对抗性测试的唯一输入。

**提取完成后，必须使用 `scripts/validate_contract_expectations.py` 验证输出。**

---

## 提取目标

为每个公开函数/方法/接口，提取以下契约信息：

```json
{
  "function_name": "函数名",
  "module": "所属模块",
  "parameters": [
    {
      "name": "参数名",
      "type": "类型声明",
      "required": true/false,
      "constraints": ["约束1", "约束2"],
      "bounds": {"min": x, "max": y, "regex": "pattern", "allowed_values": ["a", "b"]}
    }
  ],
  "return_type": "返回值类型",
  "exceptions": [
    {
      "type": "异常类型",
      "trigger": "触发条件",
      "contract_reference": "契约条款编号"
    }
  ],
  "preconditions": ["前置条件1", "前置条件2"],
  "postconditions": ["后置条件1"],
  "side_effects": ["副作用1"]
}
```

---

## 提取来源与优先级

| 优先级 | 来源 | 提取内容 |
|:---|:---|:---|
| P0 | 落地规范「输入/输出类型定义」 | 参数类型、返回值类型、必填/可选 |
| P0 | 落地规范「异常处理」 | 异常类型、触发条件、重试策略 |
| P1 | 落地规范「状态机」 | 状态转换约束、前置条件 |
| P1 | 设计文档「接口契约」 | 业务层面的输入约束、边界定义 |
| P2 | 项目结构设计文档 | 命名规范、模块边界（影响公开接口范围） |

---

## 程序化提取步骤

orchestrator Phase 1 应按以下算法执行提取，避免遗漏和歧义。

### 步骤 1：解析落地规范的「输入/输出类型定义」章节

**输入**：落地规范 Markdown 文本  
**输出**：每个类型的字段列表

**解析算法**：

```
1. 定位二级标题 "## 输入/输出类型定义" 或 "## 类型定义"
2. 在其下搜索三级标题 "### {TypeName}"
3. 对每个三级标题下的列表项，按以下正则提取：
   模式: `^-?\s*\`?(\w+)\`?\s*[:：]\s*(.+)$`
   捕获组1: 字段名
   捕获组2: 类型声明及约束描述
4. 从捕获组2中进一步解析：
   - 类型: 取第一个单词（如 str, int, List[str]）
   - "必填"/"required" → required=true
   - "可选"/"optional" → required=false
   - "默认\s*[:=]\s*(.+?)" → default=捕获值
   - "长度\s*(\d+)(?:\s*-\s*(\d+))?" → bounds.min/max
   - "范围\s*\[(.+?),\s*(.+?)\]" → bounds.min/max
   - "格式\s*[:=]\s*(.+?)" → bounds.regex（如果值看起来像正则）
   - "枚举\s*[:=]\s*\[(.+?)\]" → bounds.allowed_values
```

**示例解析**：

```markdown
- `field_a`: string, 必填, 长度 1-100
- `field_b`: int, 可选, 默认 0, 范围 [0, 1000]
- `field_c`: List[str], 必填, 元素数量 1-50
- `status`: StatusEnum, 必填, 枚举 [CREATED, PENDING, APPROVED]
```

解析结果：

| 字段名 | 类型 | 必填 | 默认 | bounds |
|:---|:---|:---|:---|:---|
| field_a | str | true | — | {min:1, max:100} |
| field_b | int | false | 0 | {min:0, max:1000} |
| field_c | List[str] | true | — | {min:1, max:50} |
| status | StatusEnum | true | — | {allowed_values:[CREATED,PENDING,APPROVED]} |

---

### 步骤 2：将类型定义映射为函数参数契约

**输入**：步骤 1 的类型列表 + 函数签名清单中的参数列表  
**输出**：每个函数的参数契约 JSON

**映射规则**：

```
对于 function_signatures.json 中的每个函数:
    对于其每个参数:
        如果在类型定义中找到同名字段:
            直接复用该字段的约束信息
        否则如果参数名以 "input_" / "request_" 为前缀:
            尝试去除前缀后在类型定义中查找
        否则:
            标记为 "约束未声明，采用最宽松假设"
            必填参数至少标注 "non-empty" 约束
```

---

### 步骤 3：解析落地规范的「异常处理」章节

**输入**：落地规范 Markdown 文本  
**输出**：异常契约列表

**解析算法**：

```
1. 定位二级标题 "## 异常处理"
2. 搜索表格或列表形式的内容
3. 如果是表格，提取列：触发条件 | 异常类型 | 处理策略
4. 如果是列表，按以下模式提取：
   模式: `^-?\s*(.+?)\s*[:：]\s*抛出\s*(\w+Error)`
   捕获组1: 触发条件
   捕获组2: 异常类型
5. 为每条异常记录 contract_reference:
   如果原文中有 "§N.N" 引用，直接使用
   否则按章节顺序分配临时编号（如 §3.1, §3.2），并在备注中标注 "原文未编号"
```

---

### 步骤 4：解析落地规范的「状态机」章节

**输入**：落地规范 Markdown 文本  
**输出**：状态约束 + 前置条件

**解析算法**：

```
1. 定位二级标题 "## 状态机"
2. 提取表格列：当前状态 | 操作 | 下一状态 | 前置条件
3. 对每个状态转换行:
    如果前置条件列非空:
        生成前置条件: "在 {当前状态} 下执行 {操作} 前，必须满足: {前置条件}"
    如果操作列空缺或标记为 "-":
        标记该状态为终态
4. 对终态:
    生成约束: "{终态} 状态下调用任何转换操作应抛出异常"
5. 对非相邻状态:
    生成约束: "从 {状态A} 直接转换到 {状态B}（不经过中间状态）应被拒绝"
```

---

### 步骤 5：组装契约期望清单

**输入**：步骤 2-4 的解析结果  
**输出**：`contract-expectations.md`

**组装算法**：

```
对于每个公开函数的每个参数:
    如果参数有 bounds 或 constraints:
        生成条目:
            编号: 按 A01, A02, ... 顺序分配
            契约维度: "{函数名} 参数 {参数名} 的 {约束类型}"
            破坏性输入: 根据约束类型生成（见下方矩阵）
            期望行为: 根据约束违反后的预期结果
            来源章节: 异常/类型定义中的章节引用

对于每个状态约束:
    生成条目:
        编号: 按 B01, B02, ... 顺序分配
        契约维度: "{函数名} 状态前置条件"
        破坏性输入: "前置条件不满足的场景描述"
        期望行为: "抛出 {异常类型}"
        来源章节: 状态机章节引用
```

**破坏性输入生成矩阵**：

| 约束类型 | 生成的破坏性输入 | 期望行为（通用） |
|:---|:---|:---|
| required=true | `None`, `null`, `undefined` | 抛出 TypeError / ValueError |
| type=str, bounds.min>0 | `""`, `"   "` | 抛出 ValueError |
| bounds.min, bounds.max | `min-1`, `max+1` | 抛出 ValueError |
| bounds.regex | 不匹配正则的字符串 | 抛出 ValueError |
| bounds.allowed_values | 不在列表中的值 | 抛出 ValueError |
| type=int | `NaN`, `Infinity`, `-Infinity`（JS）, 浮点数 | 抛出 TypeError |
| type=List[T] | `[]`, 超长度列表 | 按契约要求（可能返回空或抛异常） |

## 输出格式：契约清单

提取完成后，生成契约清单文件：

```markdown
# {模块名} 接口契约清单

> 生成时间：{YYYY-MM-DD}
> 来源：{落地规范文件名}

## 函数：func_name

### 参数
| 名称 | 类型 | 必填 | 约束 |
|:---|:---|:---|:---|
| param_a | str | ✅ | 长度 1-100，非空 |
| param_b | int | ❌ | 范围 [0, 1000]，默认 0 |

### 返回值
- 类型：ResultType
- 结构：{...}

### 异常
| 异常类型 | 触发条件 | 契约条款 |
|:---|:---|:---|
| ValueError | param_a 为空或仅空白 | §3.2 |
| ValueError | param_b < 0 或 > 1000 | §3.2 |
| TimeoutError | 外部服务 5s 无响应 | §4.1 |

### 前置条件
- [ ] 模块已初始化
- [ ] 用户已认证（如适用）

---

## 函数：func_name_2
...
```

## 契约期望清单（冻结文件）格式

由契约清单转换而来的冻结文件，供对抗性测试生成器使用：

```markdown
# {模块名} 契约期望清单
> 来源：{落地规范文件名}
> 冻结时间：{YYYY-MM-DD HH:MM:SS}

| 编号 | 契约维度 | 破坏性输入 | 期望行为 | 来源章节 |
|:---|:---|:---|:---|:---|
| A01 | username 空值 | "" | 抛出 ValueError | §3.2 |
| A02 | username 类型 | None | 抛出 TypeError | §3.2 |
| B01 | approve_order 状态前置 | 订单状态为 CANCELLED | 抛出 RuntimeError | §5.2 |
```

**生成后验证**：

```bash
python scripts/validate_contract_expectations.py \
    contract-expectations.md \
    --function-signatures function-signatures.json
```

## 契约完整性检查

提取完成后，进行完整性检查：

- [ ] 每个公开函数都有参数列表
- [ ] 每个参数都有类型声明
- [ ] 每个必填参数都有约束定义（至少非空/非 None）
- [ ] 每个声明的异常都有触发条件
- [ ] 状态机中的每个状态转换都有前置条件（或明确标注"无条件"）

**不完整时的处理**：
- 缺少参数约束 → 标注"约束未声明，采用最宽松假设"
- 缺少异常触发条件 → 标注"触发条件未明确，需用户确认"
- 通过 `AskUserQuestion` 向用户报告不完整项

---

## 提取质量检查清单

提交契约期望清单前，逐项确认：

1. [ ] 编号全局唯一，格式 `[A-Z]\d{2,3}`
2. [ ] 每个公开函数至少有一条契约条目
3. [ ] 每个必填参数至少有一条破坏性输入
4. [ ] 每个异常条件至少有一条对应的期望行为
5. [ ] 状态机中的非法转换都有对应的期望行为
6. [ ] 已运行 `validate_contract_expectations.py` 并通过
7. [ ] 文件头部有来源和冻结时间标注
