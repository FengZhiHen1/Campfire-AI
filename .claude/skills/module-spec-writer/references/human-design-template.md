### 1.1 技术实现思路

> 用自然语言描述本模块的核心技术实现逻辑。聚焦于**为什么这样实现**，而非罗列实现步骤。
>
> **必须与意图文档对齐**：技术实现必须准确反映已冻结的意图文档中的业务定义。若实现时发现歧义，标注"（与意图文档一致性说明）"。
>
> 可包含：
> - 关键设计决策及其技术理由
> - 技术备选方案对比（如有）
> - 数据流向的宏观技术描述
> - 关键抽象和模型选择的技术理由
>
> 禁止在此节写原子级操作步骤或代码片段——这些属于落地规范的范畴。

**示例**：
> 世界观构建采用 CrewAI 的 `Process.sequential` 串行模式，而非并行模式。原因是世界观构建天然有先后依赖——先确定基础设定（科技水平、社会结构），再推导文化细节和地理环境。并行模式下 Agent 之间缺乏上下文传递，容易产生内部矛盾的设定。
>
> 数据流设计上，采用"输入校验→上下文加载→Agent 协作→输出校验→持久化"五个阶段。上下文加载阶段从 Neo4j 中读取已有的世界观片段，避免与已有设定冲突。输出校验阶段通过 Pydantic 强校验确保 Agent 返回的 JSON 结构完整且字段合法，失败时自动要求 Agent 修正。
>
> 降级策略：当 LLM 超时或 Token 耗尽时，不直接报错，而是将已生成的部分结果标记 `is_partial=True` 返回——这允许前端展示"草稿"状态，人工决定是否继续或重试。此策略与意图文档中"超时展示草稿让用户决定"的业务要求一致。

### 1.4 状态机设计（技术实现策略，如适用）

> 如有状态流转，描述状态机的技术设计意图和实现策略。禁止在此节使用状态转换表——精确的状态转换表属于落地规范。

**示例**：
> 世界观构建涉及 4 个状态。初始为 IDLE，收到构建指令后进入 BUILDING。BUILDING 状态有两个出口：所有 Agent 成功返回则进入 COMPLETED；重试耗尽则进入 FAILED。FAILED 状态允许人工介入后重新触发构建，回到 BUILDING。
>
> 技术实现上，状态持久化采用 Neo4j 节点属性存储，状态转换通过显式的事务包装保证原子性。幂等策略：同一 `build_id` 的重复请求，若当前状态为 BUILDING，直接返回进行中状态；若为 COMPLETED，返回缓存结果。
>
> ```
> IDLE ──start_build──▶ BUILDING ──build_success──▶ COMPLETED
>                         │
>                         └──build_fail──▶ FAILED ──retry──▶ BUILDING
> ```

> 不涉及状态机的模块，本节写 "本功能点不涉及状态流转，故无需状态机。"

### 1.6 架构权衡与备选方案

> 记录关键的技术选型和架构决策，以及为什么做出这样的选择。这是给未来维护者的备忘录。

| 决策点 | 最终方案 | 备选方案 | 选择理由 |
|--------|---------|---------|---------|
| 编排模式 | CrewAI 串行 Process | CrewAI 并行 Process | 世界观构建有先后依赖，串行避免内部矛盾 |

> 若某决策点直接受意图文档约束（如超时降级策略），标注"受意图文档约束"。
