## 一、项目基本信息

- **项目名称**：篝火智答 (Campfire-AI)
- **项目类型**：全栈 Hybrid Monorepo
- **核心功能**：智能应急咨询、个性化档案库、真实案例库管理、人工兜底通道、科普查阅
- **目标平台**：微信小程序（一期优先）+ H5 管理后台（远期预留）
- **预期规模**：垂直领域中小规模（初期 < 10 万案例切片，用户量级为领域内精准群体）
- **团队规模**：1-3 人，无专职运维
- **代码组织风格**：Hybrid Monorepo — 前端 Feature-Based + views/logics 分层；后端按应用边界拆分 app，领域能力集中在 packages

## 三、设计原则

- **厚 package、薄 app**：领域模型、DTO、基础设施适配器集中在 `packages/`；app 仅包含 HTTP 路由、依赖注入、业务编排与中间件。新应用（如未来的 admin-web、独立 worker）可零成本复用已有能力包。
- **单向依赖**：应用层 → 共享能力层 → 工程支撑层。禁止反向依赖，禁止 app 之间直接 import。
- **前后端契约先行**：Pydantic Schema（`py-schemas`）与 TypeScript 类型（`ts-shared`）作为前后端数据契约，通过 workspace 协议引用。
- **前端表现/逻辑硬隔离**：views/ 禁止直接调用 API Service 或读写 Store，必须通过 Hooks 桥接；logics/ 禁止 import views/。这是整个前端架构中最严格、最有价值的解耦约束。
- **最小化可工作**：不为未规划的功能预留代码实现，仅在 `packages/` 和 `apps/` 中为明确规划的能力预留目录位置。

## 五、分层架构

### 5.1 分层总览

本项目采用 **Hybrid Monorepo 三层架构**：顶层按构建与复用边界划分为三大横向层，应用内部再纵向分层。

| 层级 | 层名 | 职责 | 典型内容 | 技术组件 |
|:---:|:---|:---|:---|:---|
| L1 | 应用层 | 可独立构建运行的业务入口 | `apps/api-server/`、`apps/worker/`、`apps/mini-program/` | FastAPI, Taro + React, Celery（远期） |
| L2 | 共享能力层 | 跨应用复用的领域逻辑与基础设施适配 | `packages/py-*`、`packages/ts-shared/` | SQLAlchemy, Pydantic, LangChain, JWT, Redis, MinIO |
| L3 | 工程支撑层 | 部署、运维、配置、文档 | `infrastructure/`、`scripts/`、`docs/`、`docker-compose.yml` | Docker, Nginx, GitHub Actions |

### 5.2 应用内部分层（以 api-server 为例）

| 子层 | 子层名 | 职责 | 对应目录 |
|:---:|:---|:---|:---|
| L1-sub | 接口层 | 请求校验、路由分发、响应封装 | `apps/api-server/app/api/v1/`、`apps/api-server/app/dependencies/` |
| L2-sub | 业务层 | 用例编排、流程控制、权限校验 | `apps/api-server/app/services/` |
| — | 领域层 | 跨应用共享的实体、枚举、DTO、仓储 | `packages/py-db/models/`、`packages/py-db/repositories/`、`packages/py-schemas/` |
| — | 基础设施层 | 外部系统适配（RAG、LLM、Cache、Storage、Auth） | `packages/py-rag/`、`packages/py-llm/`、`packages/py-cache/`、`packages/py-storage/`、`packages/py-auth/` |

### 5.3 层间依赖规则

```
应用层 (L1)
  ├── mini-program ──── ts-shared (L2) ──► api-server (L1) ──► packages/py-* (L2)
  │                                                            ├── py-auth
  │                                                            ├── py-db
  │                                                            ├── py-schemas
  │                                                            ├── py-rag
  │                                                            ├── py-llm
  │                                                            ├── py-cache
  │                                                            ├── py-storage
  │                                                            ├── py-config
  │                                                            └── py-logger
  └── worker ───────── packages/py-* (L2) ──► Redis ◄── api-server
                           ├── py-rag
                           ├── py-llm
                           ├── py-db
                           ├── py-cache
                           └── py-config
```

**硬性约束**：

1. 依赖方向：应用层 → 共享能力层 → 工程支撑层（单向，禁止反向依赖）
2. 应用间禁止直接 import，必须通过 `packages/` 共享代码
3. `api-server` 和 `worker` 通过 Redis 队列实现异步解耦，禁止应用间直接 RPC 调用
4. 领域层（`py-db/models`、`py-schemas`）仅可被 L2 其他包和应用层引用，不可引用基础设施具体实现
5. 公共层（`py-config`、`py-logger`）可被任意层引用，不形成循环
6. 前端 `views/` → `logics/` → HTTP/SSE → 后端 API，前端禁止直连数据库或外部服务

## 九、包/模块组织约定

### 9.1 命名规范

| 元素 | Python（packages + apps） | TypeScript（mini-program + ts-shared） |
|:---|:---|:---|
| **包目录** | snake_case：`py_config/`、`py_db/` | kebab-case 或小写：`ts-shared/`、`consult/` |
| **模块文件** | snake_case：`auth_service.py`、`jwt_utils.py` | PascalCase（组件）：`ChatBubble.tsx`；camelCase（工具/Hook/Service）：`httpClient.ts`、`useConsult.ts` |
| **类 / 接口** | PascalCase：`AuthService`、`CaseRepository` | PascalCase：`IUserProfile`、`ConsultRequest` |
| **函数 / 变量** | snake_case：`get_user_by_id()`、`user_profile` | camelCase：`getCaseById()`、`userProfile` |
| **常量** | `UPPER_SNAKE_CASE`：`MAX_RETRY_COUNT` | `UPPER_SNAKE_CASE`：`MAX_RETRY_COUNT` |
| **私有成员** | `_leading_underscore`：`_build_query()` | 无语言级私有（`private` 关键字或 JSDoc `@private`） |
| **测试文件** | `test_<模块名>.py`：`test_consult_service.py` | `*.test.ts` 或 `*.spec.ts` |

### 9.2 文件组织规则

1. **每个文件只导出一个主要的类/函数/组件**，辅助类型和工具可共存于同一文件。
2. **Barrel 导出**：Python 包通过 `__init__.py` 统一对外暴露（显式 `__all__` 列表）；TypeScript 包通过 `index.ts` 统一对外暴露。
3. **测试镜像**：后端测试与源码分离（`apps/*/tests/`、`packages/*/tests/`）；根级 `tests/` 放跨服务集成与 E2E 测试。
4. **跨层 import 禁令**：`apps/api-server/app/api/` 不可直接 import `py_db/models/`（应通过 `services/` 和 `repositories/` 访问）；`apps/` 不可直接引用其他 `apps/` 的代码。
5. **单文件行数控制**：前端单文件 ≤ 200 行；Python 函数 50-80 行内。超出必须拆分。

### 9.3 模块边界

**Python workspace**（`pyproject.toml`）：

- 2 个 apps（`api-server`、`worker`）+ 9 个 packages（`py-config`、`py-db`、`py-schemas`、`py-rag`、`py-llm`、`py-cache`、`py-storage`、`py-auth`、`py-logger`）
- 每个 package 通过 `[tool.uv.sources]` 声明 workspace 依赖
- 依赖关系：`api-server` → `py-*`（全部 9 个）；`worker` → `py-db`、`py-rag`、`py-cache`、`py-config`；`py-rag` → `py-db`、`py-llm`、`py-config`；`py-db` → `py-config`

**Node workspace**（`pnpm-workspace.yaml`）：

- 1 个 app（`mini-program`）+ 1 个 package（`ts-shared`）
- `mini-program` 通过 `"@campfire/ts-shared": "workspace:*"` 引用共享类型

**禁止跨生态直接引用**：Python 代码不可 import TypeScript 包；反之亦然。共享契约通过 HTTP API 和 DTO 文档约定实现。

### 9.4 前端 views/logics 分层规则（DS-006 维持）

**`views/`（L1a）硬性规则**：

- Pages/Components 中**禁止**直接调用 `Taro.request()`、`fetch()` 或任何 API Service
- Pages/Components 中**禁止**直接 import 或读写 Zustand Store
- Pages/Components 中**禁止**包含数据转换、校验、格式化逻辑（应封装在 `logics/utils/` 或 Hooks 中）
- Pages 通过调用 `logics/` 的 Hooks 获取数据和方法，Components 通过 Props 接收数据和回调
- 允许 `import type` 从 `logics/` 或 `@campfire/ts-shared` 导入类型定义

**`logics/`（L1b）硬性规则**：

- 禁止 `import` 任何 `views/` 下的文件（Pages、Components、Layouts）
- API Service 层统一通过 `logics/shared/services/httpClient.ts` 发请求，禁止裸调 `Taro.request()`
- Store 的读写通过 Hooks 封装后暴露

**Hook 作为 L1a-L1b 的唯一桥梁**：

```
L1a (views/)  ──→ 调用 Hooks ──→  L1b (logics/)
                    ←── 返回数据/方法 ──
```

Hooks 是表现层获取逻辑层能力的**唯一入口**。任何绕过 Hook 直接 import Service/Store 的行为均违反分层约束。

## 十三、设计决策记录

### DSR-001：采用 Hybrid Monorepo（厚 package、薄 app）而非纯模块化单体

- **背景**：v1.0 将所有代码放在 `apps/api-server/app/` 内（modules + infrastructure + core），形成"厚 app"。worker 嵌入 api-server 内部，缺少 `packages/` 共享层。
- **决策**：v2.0 重构为 Hybrid Monorepo，参考 SAP 项目结构。
- **理由**：(1) ORM 模型、DTO、RAG 引擎、LLM 客户端被 api-server 和 worker 共同需要，集中在 `packages/` 避免代码副本；(2) 独立 worker app 为未来独立扩缩容预留空间；(3) uv + pnpm 双工作空间在各自生态内管理依赖，避免工具冲突；(4) 未来 `apps/admin-web/` 或其他服务可零成本复用已有 packages。
- **后果**：需维护两套 workspace 配置；packages 包修改可能同时影响多个 apps，需全量回归测试；跨生态契约无法通过类型系统强行同步。

### DSR-002：ORM 模型与 Pydantic Schema 集中管理

- **背景**：v1.0 将 models 和 schemas 分散在各 `modules/*/` 目录下，各模块独立维护自己的数据定义。
- **决策**：v2.0 将全部 ORM 模型迁入 `packages/py-db/models/`，全部 Pydantic DTO 迁入 `packages/py-schemas/`。
- **理由**：(1) Alembic 迁移需要全局视角，分散的模型定义导致迁移脚本容易遗漏；(2) 跨模块查询（如 consult 模块需要 profiles 数据）时，集中管理避免循环 import；(3) `py-schemas` 作为前后端数据契约的单一真相来源，便于与 `ts-shared` 对齐。
- **后果**：按业务域拆分的文件组织在 models 和 schemas 层仍然保留（`models/consult.py`、`schemas/consult.py`），不会变成单个巨型文件。

### DSR-003：基础设施能力拆分为独立 package

- **背景**：v1.0 将 RAG、LLM、Cache、Storage、Security 全放在 `apps/api-server/app/infrastructure/` 下，作为 api-server 的内部子目录。
- **决策**：v2.0 将每项基础设施能力拆分为独立 package（`py-rag`、`py-llm`、`py-cache`、`py-storage`、`py-auth`、`py-logger`、`py-config`）。
- **理由**：(1) worker 的案例索引任务需要 RAG（切片+向量化）和 DB 能力，拆分后直接引用 package 即可，无需依赖 api-server；(2) 每个 package 有独立的 `pyproject.toml`（可独立测试、独立版本管理）；(3) 边界清晰——每个 package 只做一件事，符合单一职责原则。
- **后果**：package 数量较多（9 个 Python 包），首次 `uv sync` 需解析全部依赖；对于 1-3 人团队，初期可在根 `pyproject.toml` 中统一管理版本号。

### DSR-004：Worker 独立为应用

- **背景**：v1.0 的 worker（`case_indexer.py`）放在 `apps/api-server/workers/` 下，与 API 服务共享同一进程空间和部署单元。
- **决策**：v2.0 将 worker 独立为 `apps/worker/`，与 api-server 同级。
- **理由**：(1) worker 和 api-server 有不同的扩缩容需求——API 需要多副本应对并发，worker 可能只需要单实例；(2) 独立 Dockerfile 允许 worker 使用不同的基础镜像或资源配置；(3) 清晰的进程边界——worker 崩溃不影响 API 服务。
- **后果**：MVP 阶段 worker 只有 `case_indexer` 一个任务，独立 app 结构略显"重"。但目录结构成本极低（`apps/worker/` 只有 3-4 个文件），未来收益远大于当前成本。

### DSR-005：前端保持 views/logics 分层，引入 ts-shared

- **背景**：v1.0 前端采用 views/logics 硬隔离（DS-006），这是整个架构中最严格的分层约束。SAP 前端使用 features/ 单层组织（组件与逻辑混合）。
- **决策**：维持 views/logics 分层，同时引入 `packages/ts-shared/` 存放跨项目共享的类型与枚举。
- **理由**：(1) views/logics 分离使组件成为"Dumb Component"——高度可测试、可复用、可替换；(2) Hooks 作为唯一桥接点形成了清晰的前端内部 API 契约；(3) `ts-shared` 解决跨项目类型复用（未来 H5 管理后台可直接引用同一套类型定义），不与 views/logics 分层冲突。
- **后果**：`ts-shared` 仅包含纯类型和枚举，不含任何运行时逻辑。`logics/shared/types/` 中的应用级类型（如 Store State 类型）保留在原位，不迁入 `ts-shared`。

### DSR-006：v1.0 中仍有效的结构决策保留

以下 v1.0 设计决策在 v2.0 中保持不变：

| 原决策 | 内容 | 保留原因 |
|:---|:---|:---|
| DS-003 | 前端 Feature-Based + 表现/逻辑分层 | 5 个功能模块边界清晰，views/logics 分离是核心优势 |
| DS-004 | api-server 和 mini-program 分离为独立 app | 技术栈异构、构建独立、部署独立 |
| DS-005 | 后端 flat-layout | 仅作为应用运行，Docker 容器内直接执行，不需要 pip install |
| ADR-001 | 后端 Python/FastAPI | AI 生态第一语言 + asyncio 异步 |
| ADR-002 | 前端 Taro + 微信小程序 | 最低触达门槛 |
| ADR-003 | 向量检索 pgvector | 统一数据层，减少运维组件 |
| ADR-004 | 架构模式模块化单体 | 1-3 人团队，微服务不合算 |
| ADR-005 | LLM DeepSeek API | 性价比最优 |

## 十四、最佳实践说明

1. **严格执行分层依赖**：`apps/` → `packages/` → `infrastructure/`（单向），禁止反向依赖。workspace 级循环依赖由 uv/pnpm 构建时自动检测。
2. **配置外置**：所有密钥、第三方地址、功能开关通过 `.env` 注入，`py-config` 使用 Pydantic Settings 统一校验；禁止硬编码。
3. **契约先行**：前后端共享 DTO 定义（`py-schemas`）与类型约定（`ts-shared`），减少联调歧义。接口变更时同步更新两端的类型定义。
4. **线上耗时任务异步化**：案例索引入库、报告生成等耗时操作统一走 Worker（Redis 队列）；常规查询接口 P95 < 300ms。
5. **可观测与审计**：关键动作（AI 调用、权限拒绝、工单创建）必须落审计日志（`py-logger` + trace_id）。
6. **零 Any 容忍**：前端禁止 `any`；Python 全面使用类型注解（`str | None`、`list[dict[str, object]]`）。
7. **AI 数据隐私**：DeepSeek API 仅传入脱敏后的案例上下文与标签，严禁传入用户真实姓名、手机号等 PII。Prompt 模板强制注入免责声明。
8. **Alembic 迁移为单一真相来源**：所有数据库 schema 变更必须通过 `packages/py-db/migrations/` 的 Alembic 脚本执行，禁止手动修改生产数据库。
9. **生产 HTTPS 必选**：`infrastructure/nginx/nginx.conf` 必须启用 TLS 1.3；JWT Token 禁止在 HTTP 明文下传输（开发环境除外）。
