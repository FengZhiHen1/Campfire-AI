## 二、选定的结构模式

| 维度 | 选定模式 | 选择理由 | patterns-by-stack.md 章节 |
|:---|:---|:---|:---|
| 前端 | React Feature-Based + 表现/逻辑分层（适配 Taro） | 5 个业务功能模块边界清晰，Feature-Based 便于长期维护；前端内部按 views/（表现层）和 logics/（逻辑层）严格分离，Pages/Components 不包含业务逻辑 | §1 模式 A + 自订扩展 |
| 后端 | FastAPI 三层架构（按模块组织） | 标准 Web 项目，领域复杂度适中；模块化单体要求先按模块划分、模块内按 api/services/repositories 三层组织 | §3 模式 A |
| Monorepo | 按应用边界分层（Hybrid Monorepo） | 前后端分离（Taro + FastAPI），中间通过 HTTP/SSE 通信；infrastructure/ 统一管理部署与运维资产 | §6 模式 A |
| Python 包 | flat-layout | 后端仅作为应用运行（Docker 容器内），不需要 pip install 或发布到 PyPI | §7 |

> 模式组合：**"Taro Feature-Based（表现/逻辑分层） + FastAPI 模块化三层 + 应用边界 Monorepo"**

## 三、分层架构

### 3.1 分层总览

| 层级 | 层名 | 职责 | 典型内容 | 技术组件 |
|:---:|:---|:---|:---|:---|
| L1a | 前端表现层 | 纯 UI 渲染与用户交互 | 页面、UI 组件、布局、样式、静态资源 | Taro + React + Taro UI |
| L1b | 前端逻辑层 | 前端业务逻辑、状态管理、数据获取 | Hooks、API Service、Zustand Store、类型定义、工具函数 | Zustand + TypeScript |
| L2 | 应用层 | 业务用例编排、请求处理、权限校验 | API Router、Service 编排、Middleware、DTO | FastAPI + Pydantic v2 |
| L3 | 领域层 | 核心业务逻辑与规则 | 业务实体、值对象、Repository 接口、领域服务 | SQLAlchemy ORM Models |
| L4 | 基础设施层 | 外部依赖适配与跨模块共享能力 | DB 连接、RAG 引擎、LLM 客户端、缓存、对象存储 | SQLAlchemy async、LangChain、Redis、MinIO |
| L5 | 公共层 | 跨层共享的基础类型与工具 | 配置、异常定义、日志、共享枚举/常量 | Python stdlib、Pydantic Settings |

### 3.2 层间依赖规则

```
L1a 前端表现层（views/）── 通过 Props/Hooks 获取数据 ──→ L1b 前端逻辑层（logics/）
                                                              │
                                                         HTTP/SSE
                                                              │
                                                              ↓
                                                         L2 应用层（FastAPI Router）
                                                              │
                                                              ↓
                                                         L2 应用层（Service 编排）
                                                              │
                                                              ↓
                                                         L3 领域层（Entity / Repository 接口）
                                                              ↑
                                                         L4 基础设施层（Repository 实现 / RAG / LLM / Cache）
                                                              ↑
                                                         L5 公共层（Config / Exceptions / Logging）
```

**前端内部依赖（L1a ↔ L1b）**：

- **L1a → L1b**：Pages/Components 通过调用 Hooks 获取数据和操作（如 `useConsult()`、`useAuth()`），Components 接收数据通过 Props 渲染，不直接调用 API Service 或读写 Store
- **L1b → L1a**：禁止——逻辑层不得 import 任何 views/ 下的文件（Pages、Components、Layouts）
- **L1b 共享区**：`logics/shared/` 下的 Hooks、Services、Store、Types 供各 feature 的 logics 引用，不依赖任何具体 feature

**跨端依赖**：

- **L1 → L2**：通过 HTTP REST API 或 SSE 流式接口通信，禁止前端直接访问数据库或基础设施
- **L2 → L3**：应用层 Service 调用领域层 Entity 和 Repository 接口，不直接操作 ORM Session
- **L2 → L4**：应用层可通过基础设施层调用 RAG 引擎、LLM 客户端等共享能力
- **L4 → L3**：基础设施层实现 L3 定义的 Repository 接口（依赖倒置）
- **L5**：所有层均可引用公共层（Config、Exceptions、Logging），公共层不依赖任何业务层
- **禁止反向依赖**：L3 不得 import L2，L4 不得 import L2 的业务模块

### 3.3 各层详解

#### L1a 前端表现层 — views/ 目录

- **职责边界**：负责 UI 渲染、用户交互、动画、样式；**不包含任何业务逻辑、API 调用、状态管理或数据转换**
- **包含内容**：
  - `views/<feature>/pages/`：页面组件（路由级别入口），通过调用 logics/ 的 Hooks 获取数据并传递给子组件
  - `views/<feature>/components/`：功能专有 UI 组件（纯渲染，通过 Props 接收数据，通过回调触发逻辑层操作）
  - `views/shared/components/`：跨功能复用的通用 UI 组件（Button、Modal、Loading 等）
  - `views/layouts/`：页面布局壳（TabBar、AuthLayout）
  - `views/assets/`：静态资源（图片、图标）
- **对外接口**：通过调用 `logics/` 中的 Hooks 获取数据和方法；Components 通过 Props 接口对外暴露
- **技术映射**：Taro 4.x + React 18 + TypeScript + Taro UI 3.x + SCSS
- **硬性规则**：
  - Pages/Components 中禁止直接调用 `Taro.request()` 或 `fetch()`
  - Pages/Components 中禁止直接读写 Zustand Store（通过 Hooks 封装）
  - Pages/Components 中禁止包含数据转换、校验、格式化逻辑（应封装在 logics/utils/ 或 Hooks 中）
  - Pages/Components 可以 import `logics/` 中的 types（仅类型导入）

#### L1b 前端逻辑层 — logics/ 目录

- **职责边界**：负责前端全部业务逻辑——API 调用、状态管理、数据转换与校验、业务流程编排；**不包含任何 UI 渲染代码（JSX/TSX）**
- **包含内容**：
  - `logics/<feature>/hooks/`：自定义 Hooks，封装该功能的数据获取、状态更新、业务编排
  - `logics/<feature>/services/`：API 调用封装（基于 `logics/shared/services/httpClient.ts`）
  - `logics/<feature>/store/`：Zustand Store Slice，管理该功能的状态
  - `logics/<feature>/types/`：该功能的 TypeScript 类型定义
  - `logics/shared/hooks/`：通用 Hooks（useRequest、useAuth、usePagination）
  - `logics/shared/services/`：HTTP 客户端、Token 管理器
  - `logics/shared/store/`：全局 Zustand Store（userStore、appStore）
  - `logics/shared/utils/`：纯函数工具（格式化、校验、存储）
  - `logics/shared/types/`：全局 TypeScript 类型
- **对外接口**：通过 Hooks 向 L1a 暴露数据和操作方法；通过 API Service 向 L2 发送 HTTP 请求
- **技术映射**：TypeScript + Zustand 5.x + Taro API（Taro.request、Taro.setStorage 等）
- **硬性规则**：
  - 禁止 import 任何 `views/` 下的文件
  - 禁止在 Hooks 中直接操作 DOM 或渲染 JSX
  - API Service 层统一通过 `httpClient.ts` 发请求，不直接使用裸 `Taro.request()`
  - Store 的读写必须通过 Hooks 暴露，组件不直接 import Store

#### L2 应用层 — FastAPI 业务编排

- **职责边界**：接收 HTTP 请求、参数校验（Pydantic）、路由分发、用例编排（调用 L3/L4 完成业务流程）、权限校验、响应组装；不包含核心领域规则（在 L3）和基础设施细节（在 L4）
- **包含内容**：API Router（`modules/*/api.py`）、Service 编排（`modules/*/services.py`）、FastAPI 依赖注入（`modules/*/deps.py`）、全局中间件
- **对外接口**：通过 REST API（OpenAPI 文档自动生成）向 L1b 暴露；通过 import 调用 L3 领域接口和 L4 基础设施
- **技术映射**：FastAPI + Pydantic v2 + python-jose（JWT）+ passlib（bcrypt）

#### L3 领域层 — 业务实体与规则

- **职责边界**：定义核心业务实体（SQLAlchemy ORM Models）、Repository 抽象接口、领域枚举与值对象；不包含 HTTP 细节、数据库连接管理、外部 API 调用
- **包含内容**：
  - `modules/*/repositories.py`：定义 Repository 抽象接口（Protocol 或 ABC）
  - ORM Models（SQLAlchemy declarative base 的子类）
  - 领域枚举（`BehaviorType`、`EmotionLevel`、`CaseStatus`、`TicketPriority` 等）
- **对外接口**：向上对 L2 暴露 Repository 接口和 Entity 类；向下由 L4 实现 Repository 接口
- **技术映射**：SQLAlchemy 2.0 ORM（declarative models）

#### L4 基础设施层 — 外部依赖适配

- **职责边界**：实现 L3 定义的 Repository 接口（提供具体的数据库查询实现）、封装外部服务调用（DeepSeek API、Redis、MinIO）、提供 RAG 引擎能力（向量检索、文本切片、Prompt 组装）；不包含业务规则
- **包含内容**：
  - `infrastructure/database/`：AsyncSession 工厂、SQLAlchemy Base、Alembic 迁移配置
  - `infrastructure/rag/`：pgvector 检索、阿里 text-embedding-v4、LangChain Chains、Prompt 模板、自定义 TextSplitter
  - `infrastructure/llm/`：DeepSeek API 客户端（含重试/熔断）
  - `infrastructure/cache/`：Redis 客户端（缓存/限流/队列）
  - `infrastructure/storage/`：MinIO 客户端
  - `infrastructure/security/`：JWT 签发/校验、bcrypt 哈希、RBAC 权限检查器
- **对外接口**：实现 L3 定义的 Repository 接口；向 L2 暴露 RAG/LLM/Cache 等能力供 Service 调用
- **技术映射**：SQLAlchemy async、asyncpg、LangChain、redis-py、minio-py、httpx

#### L5 公共层 — 跨层共享

- **职责边界**：提供所有层均可引用的基础能力；不包含任何业务逻辑
- **包含内容**：`core/config.py`（Pydantic Settings 环境变量管理）、`core/exceptions.py`（自定义异常类层级）、`core/logging_config.py`（结构化 JSON 日志配置）、`core/events.py`（FastAPI lifespan 启动/关闭事件）
- **对外接口**：被 L2-L4 所有层 import，不依赖任何业务层
- **技术映射**：Pydantic Settings、Python logging、FastAPI lifespan

| `apps/api-server/app/infrastructure/database/` | L4 | 数据库连接池、会话管理、Base 类 | → 实现 L3 Repository 接口; → L5 core/config.py |

| `apps/api-server/app/infrastructure/llm/` | L4 | DeepSeek API 客户端封装 | → L5 core/config.py; 被 L4 rag/ 和 L2 调用 |
| `apps/api-server/app/infrastructure/cache/` | L4 | Redis 客户端（缓存/限流/队列） | → L5 core/config.py; 被 L2/middleware 调用 |

### 5.1 场景一：应急咨询 SSE 流式响应

```

  │
  ▼

  │
  ▼

  │
  ▼
```

### 6.1 命名规范

| 元素 | Python 后端（api-server） | TypeScript 前端（mini-program） |
|:---|:---|:---|
| **目录** | `snake_case`：`api_server/`、`case_indexer.py` | `kebab-case` 或功能名：`consult/`、`useAuth.ts` |
| **Python 文件** | `snake_case`：`case_api.py`、`rag_retrieval.py` | — |
| **TypeScript 文件** | — | PascalCase（组件）：`ChatBubble.tsx`；camelCase（工具/Hook/Service）：`httpClient.ts`、`useConsult.ts`；`index.ts`（barrel） |
| **类 / 接口** | `PascalCase`：`ConsultService`、`CaseRepository` | `PascalCase`：`ConsultService`、`IUserProfile` |
| **函数 / 变量** | `snake_case`：`get_case_by_id()`、`user_profile` | `camelCase`：`getCaseById()`、`userProfile` |
| **常量** | `UPPER_SNAKE_CASE`：`MAX_RETRY_COUNT` | `UPPER_SNAKE_CASE`：`MAX_RETRY_COUNT` |
| **私有成员** | `_leading_underscore`：`_build_query()` | 无语言级私有（`private` 关键字或 JSDoc `@private`） |
| **测试文件** | `test_<模块名>.py`：`test_consult_api.py` | `*.test.ts` 或 `*.spec.ts` |

### 6.2 文件组织规则

#### Python 后端

- **每个模块独立为一个 Python package**（含 `__init__.py`），位于 `app/modules/<module_name>/`
- **模块内文件职责单一**：`api.py`（路由）、`services.py`（业务逻辑）、`repositories.py`（数据访问接口与实现）、`schemas.py`（DTO）、`models.py`（ORM Entity）、`deps.py`（FastAPI 依赖）
- **模块间引用规则**：
  - 允许：模块 A 的 `services.py` → import 模块 B 的 `services.py`（通过公开接口调用）
  - 禁止：模块 A 的 `services.py` → import 模块 B 的 `models.py`（应通过 B 的 Repository 接口访问数据）
  - 禁止循环依赖：模块 A ↔ 模块 B 的 `services.py` 互相 import
- **基础设施层**（`infrastructure/`）是共享能力池，所有模块的 `services.py` 均可引用
- **公共层**（`core/`）是全局基础配置，所有层均可 import
- **Barrel 导出**：每个 Python package 的 `__init__.py` 应显式导出公开接口（`__all__` 列表）
- **测试镜像**：`tests/unit/` 目录结构镜像 `app/` 目录结构

#### TypeScript 前端

**表现层与逻辑层分离规则**（核心约束）：

- **`views/`（L1a）硬性规则**：
  - Pages/Components 中**禁止**直接调用 `Taro.request()`、`fetch()` 或任何 API Service
  - Pages/Components 中**禁止**直接 import 或读写 Zustand Store（`consultStore`、`userStore` 等）
  - Pages/Components 中**禁止**包含数据转换、校验、格式化逻辑（应封装在 `logics/utils/` 或 Hooks 中）
  - Pages 通过调用 `logics/` 的 Hooks 获取数据和方法，Components 通过 Props 接收数据和回调
  - 允许 `import type` 从 `logics/` 导入类型定义（纯类型导入，编译后不产生 runtime 依赖）
- **`logics/`（L1b）硬性规则**：
  - 禁止 `import` 任何 `views/` 下的文件（Pages、Components、Layouts）
  - API Service 层统一通过 `logics/shared/services/httpClient.ts` 发请求，禁止裸调 `Taro.request()`
  - Store 的读写通过 Hooks 封装后暴露，不鼓励 Pages 直接 import Store（但允许同 feature 的 Hook 直调 Store）
- **Feature 间隔离**：
  - `views/<feature>/` 可 import `views/shared/components/`（共享 UI 组件）和 `logics/<feature>/` 的 Hooks
  - `logics/<feature>/` 可 import `logics/shared/`（共享逻辑）和其他 feature 的 `logics/<other>/services/`（跨模块 API 调用）
  - 禁止 `views/<featureA>/` import `views/<featureB>/` 的 components
  - 若 feature 间需共享 UI 组件，提升至 `views/shared/components/`
- **页面注册**：每个 page 组件在 `app.config.ts` 的 `pages` 数组中注册，路径指向 `views/<feature>/pages/`
- **一个文件一个主导出**：每个 `.tsx` 文件只导出一个主要的组件/类/函数（辅助类型可用 `export type`）
- **Hook 作为 L1a-L1b 的唯一桥梁**：
  ```
  L1a (views/)  ──→ 调用 Hooks ──→  L1b (logics/)
                      ←── 返回数据/方法 ──
  ```
  Hooks 是表现层获取逻辑层能力的**唯一入口**。任何绕过 Hook 直接 import Service/Store 的行为均违反分层约束。

### 6.3 模块边界

本项目为模块化单体架构，模块边界通过 **Python package 隔离 + 接口契约** 维护：

#### 后端模块边界

- **边界定义**：`app/modules/<name>/` 即为模块边界，模块内包含完整的 api/services/repositories/schemas/models
- **共享机制**：模块间通过 `services.py` 的公开函数进行调用（如 `consult/services.py` 调用 `tickets/services.py: create_ticket()`）
- **共享包**：`infrastructure/` 是跨模块的共享能力池（数据库、RAG、LLM、缓存），所有模块均可引用
- **禁止事项**：
  - 禁止模块 A 直接 import 模块 B 的 `models.py`（绕过 Repository）
  - 禁止模块间循环 import（A → B → A）
  - 禁止模块的 `api.py` 之间互相 import（路由应独立）

#### 前端模块边界

- **表现/逻辑边界**：`views/` 与 `logics/` 之间仅通过 Hooks 桥接
  - `views/<feature>/` page 调用 `logics/<feature>/hooks/` 中的 Hooks 获取数据
  - Hooks 返回的数据通过 Props 向下传递给 Components
  - 禁止 views/ 直接 import logics/ 的 services、store（必须通过 Hooks 封装）
- **前后端边界**：前端通过 `logics/shared/services/httpClient.ts` 统一与后端通信，前端不直接操作数据库或调用 LLM
- **前后端数据契约**：Pydantic Schema（后端 L2）与 TypeScript 类型（前端 L1b）应保持同步，作为前后端的数据契约

## 附录 B：设计决策记录

### DS-001：为什么选择"模块优先而非层优先"的包组织方式？

传统的三层架构通常按层组织（`api/`、`services/`、`repositories/` 顶层目录），但本项目选择了先按模块划分（`modules/consult/`、`modules/profiles/`），模块内再分层的组织方式。理由：

1. **模块化单体架构要求**（ADR-004）：模块化单体的核心纪律是按业务模块划分边界，为将来独立拆出为微服务做准备。若按层组织，模块边界被层边界切割，难以独立拆出。
2. **模块内聚 > 层内聚**：1-3 人团队按业务功能分工（如 A 负责应急咨询、B 负责案例库），模块优先的组织方式让开发者在一个目录内完成 api/services/repositories/models 全部工作，无需跨目录跳转。
3. **基础设施层已承担跨模块共享职责**：数据库连接、RAG 引擎、LLM 客户端、缓存等已在 `infrastructure/` 中统一管理，不会因模块优先而产生代码重复。

### DS-002：为什么 RAG 引擎放在 infrastructure/ 而非 consult 模块内部？

RAG 引擎（向量检索、文本切片、Embedding）可能被多个业务模块复用：
- **应急咨询（consult）**：核心使用场景，查询时调用 pgvector 检索 + LLM 生成
- **案例管理（cases）**：案例入库时调用切片 + 向量化
- **科普查阅（knowledge）**：未来可能对科普内容做语义搜索

将其放在 `infrastructure/rag/` 作为共享基础设施，避免了代码重复和模块间的不必要耦合。若 RAG 引擎仅服务于 consult 模块（且未来确定不会扩展），可降级至 `modules/consult/rag/` 内部。

### DS-003：为什么前端选择 Feature-Based + 表现/逻辑分层？

Taro 官方模板默认使用 Layer-Based 结构（`pages/`、`components/`、`services/` 顶层目录）。本项目选择 Feature-Based + 表现/逻辑分层的理由：

1. 5 个业务功能模块边界清晰（咨询、档案、案例、工单、科普），Feature-Based 避免 pages/ 目录膨胀和跨目录引用混乱
2. 每个 feature 可独立开发、测试、删除（一键清理整个 feature 目录）
3. 团队 1-3 人，每个 feature 的内部文件数量不多，不会产生过深的目录嵌套
4. **表现/逻辑分离**是 React 社区的最佳实践——Pages/Components 为纯渲染组件，Hooks/Service/Store 承载全部业务逻辑，使组件易于测试和复用

### DS-004：为什么 api-server 和 mini-program 分离为独立的 app 而非合并？

- **技术栈异构**：api-server 使用 Python/FastAPI，mini-program 使用 TypeScript/Taro，无共享运行时
- **构建独立**：两者有独立的 `pyproject.toml` 和 `package.json`，独立的构建工具链（poetry/pip vs pnpm/npm）
- **部署独立**：api-server 部署为 Docker 容器，mini-program 编译后上传至微信小程序平台
- **Monorepo 意义**：在同一仓库中管理保持版本一致性，通过 `infrastructure/` 共享部署配置，而非通过代码共享

### DS-005：为什么用 flat-layout 而非 src-layout？

根据 patterns-by-stack.md 第 7 节的决策要点：仅作为应用运行、不需要 `pip install -e .` 或发布到 PyPI 的项目，flat-layout 更简洁。API Server 在 Docker 容器中直接运行（`uvicorn app.main:app`），无需可安装包结构。若未来需将 `infrastructure/` 下的组件拆出为独立可复用包，可再为该包采用 src-layout。

### DS-006：为什么前端采用 views/logics 分离而非单层 Feature-Based？

在最初的 Feature-Based 设计中，每个 feature 目录内混合了 pages、components、hooks、services、store 等文件。调整为 `views/` 和 `logics/` 两层分离的理由：

1. **职责边界清晰**：Pages/Components 为纯渲染组件，不含业务逻辑（API 调用、状态管理、数据转换），使其成为"Dumb Component"——高度可测试、可复用、可替换
2. **逻辑层可独立测试**：Hooks、Services、Store 不依赖 UI 渲染，可完全脱离 Taro/React 渲染环境进行单元测试
3. **依赖方向可验证**：禁止 `logics/` import `views/` 的规则可被 ESLint 静态检查，从根本上杜绝 UI 层向逻辑层泄漏渲染依赖
4. **人员分工自然映射**：UI 开发者集中在 `views/` 目录，逻辑开发者集中在 `logics/` 目录，减少 Git 冲突
5. **Hook 作为唯一桥接点**：所有 `views/` 到 `logics/` 的交互必须通过 Hooks，形成了清晰的契约边界——Hook 签名即为"前端内部 API"
