# 应急咨询 — 主咨询页 页面骨架

> 生成日期：2026-06-28
> 源文档：`docs/pages/consult-main-skeleton.md`（旧版骨架 v1）、`docs/功能设计/功能模块全拆解.md#02-智能应急咨询`
> 目标平台：miniprogram (750rpx 基准)
> 关联 ViewModel：`useConsult` Hook — 8 状态联合类型，25 个字段/方法
> 设计方向：应急方案仪表盘 — 方案卡片是绝对主角

---

## 元信息

| 属性 | 值 |
|------|-----|
| 目标平台 | miniprogram（750rpx 基准） |
| 视图名称 | 主咨询页（consult-index） |
| 所属页面 | 应急咨询（consult） |
| 页面职责 | 入口引导(idle) → 加载过渡(submitting) → 方案生成(streaming) → 结果展示(completed) → 异常恢复(error) |
| 关联页面 | `select.tsx`（选择页，独立页面）、`history.tsx`（历史列表） |
| 视觉方向 | 抛弃聊天隐喻，采用**应急方案仪表盘** — 方案卡片是绝对主角 |

## 平台布局

### 状态路由总览

```
Tab "应急咨询"
    │
    ▼
idle (本页)
    │ 点击 "开始咨询" → startConsult()
    ▼
selecting_behavior (select.tsx)
    │ 提交 → submitting (本页)
    │ 取消 → idle (本页)
    ▼
submitting → streaming → completed (本页)
    │
    ├── startNewConsult() → selecting_behavior (select.tsx)
    ├── goToTicket() → tickets/detail (工单详情)
    └── goBackToSelecting() → selecting_behavior (select.tsx)

异常路径：
    submitting ──[网络异常]──→ submit_failed
    streaming  ──[SSE 中断]──→ stream_failed
    streaming  ──[置信度<0.7]──→ ticket_guide
```

## 功能块详述

### 区块A — 导航栏

- **Logic-ID**: `sessionState`, `crisisLevel`
- **职责**: 页面标识 + 危机等级可视化
- **idle/submitting/error/ticket_guide**: 仅标题 "应急咨询"，居中
- **streaming/completed**: 标题 + 右侧危机徽章（仅 `crisisLevel` 有值时显示）
- **危机徽章三态**:

  | 等级 | crisisLevel | 背景色 token | 文字 |
  |------|-------------|-------------|------|
  | 轻度 | `mild` | `$cf-tertiary` | 等级：轻度 |
  | 中度 | `moderate` | `$cf-primary` | 等级：中度 |
  | 重度 | `severe` | `$cf-error` | 等级：重度（脉动动画） |

- **徽章形状**: pill, 8rpx radius, 8rpx 12rpx padding, 高度 56rpx
- **三态**:
  - Loading: 不适用（导航栏无异步加载）
  - Empty: 无 crisisLevel 时不显示徽章
  - Error: 不适用
- **data-testid**: `consult-navbar`, `consult-navbar-crisis-badge`

### 区块B — idle 入口区

- **Logic-ID**: `sessionState`, `startConsult`
- **职责**: 建立场景认知 + 引导行动
- **抽象图形**: 160rpx 圆形，`primary-container` 背景，内含抽象几何线条（篝火/暖光意象）。纯 CSS/SVG 实现，不依赖 emoji
- **标题 & 副标题**: 垂直居中排列, 上下间距 32rpx
- **CTA 按钮**: `primary` 背景, `on-primary` 文字, 96rpx 高, 32rpx 圆角
- **三态**:
  - Loading: 不适用（idle 是稳态）
  - Empty: 不适用（idle 本身即展示态）
  - Error: 不适用
- **data-testid**: `consult-idle-entry`, `consult-idle-graphic`, `consult-idle-cta`

### 区块C — submitting 加载区

- **Logic-ID**: `sessionState`, `isConsultActive`
- **职责**: 告知系统正在工作，缓解等待焦虑
- **暖光呼吸动画**: 160rpx 圆形区, `primary` 色光晕 `opacity 0.4↔0.8`, 2s ease-in-out infinite
- **进度点**: 4 个横条 (48rpx × 8rpx), 从左到右依次填充 `primary` 色, 循环
- **文案**: "正在分析案例库…" (主) / "匹配最相似的历史案例" (副)
- **过渡**: SSE 首段结构数据到达 → 自动切换到 streaming 卡片
- **三态**:
  - Loading: 暖光呼吸 + 进度点序列播放（本区块的唯一态）
  - Empty: 不适用
  - Error: 不适用（提交失败走 submit_failed 状态）
- **data-testid**: `consult-submitting-glow`, `consult-submitting-dots`, `consult-submitting-text`

### 区块D — 用户查询摘要

- **Logic-ID**: `behaviorDescription`, `behaviorTypeSelection`, `emotionLevel`
- **职责**: 紧凑展示用户提交内容，作为方案的上下文锚点
- **形态**: `surface` 背景卡片, 16rpx 圆角, 32rpx 内边距, 左右 32rpx margin
- **内容**:
  - 第一行: 行为类型 `primary-container` pill + "·" + 情绪等级文字
  - 第二行: 用户描述文本, 最多 3 行, 超出省略
- **三态**:
  - Loading: 不适用（streaming 首帧即显示完整数据）
  - Empty: 不适用（必有用户输入才进入 streaming）
  - Error: 不适用
- **data-testid**: `consult-user-summary`, `consult-user-summary-type`, `consult-user-summary-desc`

### 区块E — 干预方案卡片 (HERO)

- **Logic-ID**: `planSections`, `accumulatedText`, `sessionState`
- **职责**: 承载 AI 生成的四段式应急方案 — 用户来此页面的唯一目的
- **卡片外壳**: `surface` 背景, 16rpx 圆角, `diffuse shadow`, 32rpx 内边距, 左右 32rpx margin

#### streaming 态

- 卡片完整出现（标题 + 四段标题 + accent bars 始终可见）
- 未开始的段显示浅色占位文字 "（待生成）"
- 当前活跃段: 打字机光标 + 逐字浮现
- 已完成的段: 右上角显示 tertiary 色 ✓

#### completed 态

- 全部内容展示，各段独立滚动区域

#### 四段式规范

| 段名 | 左侧 accent bar | 标题色 token | 正文规格 |
|------|----------------|-------------|---------|
| 即时安全干预 | `tertiary` 4rpx | `$cf-tertiary` | body-md, 1.8 line-height |
| 情绪安抚话术 | `primary` 4rpx | `$cf-primary` | italic + 引用块样式 |
| 后续观察指标 | `secondary` 4rpx | `$cf-secondary` | body-md, bullet list |
| 就医判断标准 | `error`/`secondary` 4rpx | `$cf-error`/`$cf-secondary` | body-md（仅需就医时红色） |

#### 安抚话术特殊处理

- 第一句作为引用块: `surface-variant` 背景 + 4rpx `primary` 左竖条, italic
- 后续句子普通列表

#### 段间距

- completed 态: 32rpx
- streaming 态: 24rpx

#### 底部信息栏

- ghost border 分割线 + 案例引用 pill（如 "基于 4 个相似案例"）+ 置信度徽章

#### 三态

| 态 | 行为 |
|----|------|
| Loading (streaming) | 标题 + 结构可见，内容逐字填充 |
| Empty | 不适用 — streaming 必有内容 |
| Error | 不适用 — streaming 失败走 error 状态页 |

- **data-testid**: `consult-plan-card`, `consult-plan-card-header`, `consult-plan-section-{n}`, `consult-plan-footer`

## 文件拆分方案

```
consult/pages/
├── index.tsx                   ← 主页面（接线容器，≤150 行）
│   → 导入 useConsult Hook
│   → 按 sessionState 路由到对应子组件
│   → 通过 Props 注入数据和方法
├── select.tsx                  ← 选择页面（selecting_behavior，独立页面）
└── components/
    ├── IdleEntry.tsx           ← idle 入口区（区块B, ≤80 行）
    ├── SubmittingGlow.tsx      ← submitting 暖光加载（区块C, ≤80 行）
    ├── UserQuerySummary.tsx    ← 用户查询摘要（区块D, ≤60 行）
    ├── PlanCard.tsx            ← 干预方案卡片（区块E, ≤200 行）
    │   → streaming/completed 两态由 isStreaming prop 驱动
    ├── ConfidenceBadge.tsx     ← 置信度徽章（区块F, ≤50 行）
    ├── ReferenceCases.tsx      ← 参考案例（区块G, ≤100 行）
    ├── EscalationBar.tsx       ← 人工兜底条（区块I, ≤50 行）
    └── DisclaimerBar.tsx       ← 免责声明（区块J, ≤40 行）
```

**原则**:
- index.tsx 是唯一接线点 — import useConsult，将 Hook 返回值拆解后通过 Props 注入子组件
- 所有子组件为纯展示型 — 不 import 任何 Hook，所有数据和回调通过 Props 接收
- 每个组件 ≤ 200 行
- Props 类型内联定义于组件文件（简洁优先；如需跨文件共享，抽取到 `contracts/` 目录）

## 设计决策记录

### 为什么抛弃聊天隐喻？

1. **危机场景 ≠ 社交聊天** — 聊天 UI 传递"轻松交谈"的信号，与紧张处境冲突
2. **方案是交付物，不是消息** — 把四段式应急方案塞在气泡里，严重降级了它的权威感
3. **Chat 暗示多轮对话** — 当前 MVP 是单轮 consult，不需要聊天历史感
4. **布局稳定 > 动态插入** — 聊天列表的消息插入导致滚动位置跳动，streaming 期间尤其糟糕

### 为什么 streaming 直接显示卡片骨架？

当前实现先显示裸文本气泡，检测到结构化数据后切换为卡片 — 这一跳破坏了阅读连续性。方案卡片骨架从第一帧就告诉用户："你的结构化方案正在组装"，视觉预期一致。

### 为什么用 accent bar 而非 emoji 图标？

1. emoji 在不同平台渲染不一致（iOS/Android/Web 各有差异）
2. accent bar 是更抽象、更专业的视觉语言
3. 4rpx 彩色竖条比 emoji 更节省水平空间
