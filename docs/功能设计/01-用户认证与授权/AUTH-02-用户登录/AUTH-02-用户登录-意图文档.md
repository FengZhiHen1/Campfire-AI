### 1.1 项目定位

用户登录是「篝火智答」平台的认证入口，负责验证用户凭据（用户名和密码）的合法性，验证通过后签发访问凭证。它在用户注册（AUTH-01）之后执行——用户先注册账号，再用注册时设置的凭证登录。登录成功后签发的访问凭证是用户后续所有业务操作的身份依据，被 Token 续期（AUTH-03）、五级 RBAC 鉴权（AUTH-04）和认证会话管理（AUTH-06）等模块使用。在整体流程中，它是一个**认证签核节点**——输入用户身份凭据，输出经过签核的访问凭证。

### 1.4 业务目标

1. 已注册用户可凭正确的用户名和密码成功登录，登录后获得有效的访问凭证
2. 登录时使用的密码必须通过不可逆的加密比对方式进行验证，系统不存储或比对密码明文
3. 登录成功后签发的访问凭证具有明确的有效期限制，过期后不可继续使用
4. 登录成功后签发的凭证必须包含用户的角色身份信息，供下游鉴权模块使用
5. 使用不存在的用户名或错误的密码登录时，系统拒绝并给出统一的提示信息

### 1.5 用户旅程 / 业务流程

1. **进入登录页**：用户已注册但未登录，打开应用后进入登录页面
2. **输入登录凭据**：用户输入已注册的用户名和对应的密码
3. **提交登录请求**：用户点击登录按钮，系统将凭据提交至认证服务
4. **验证凭据**：系统根据用户名查找账号，通过密码比对验证凭据合法性
5. **签发访问凭证**：验证通过后，系统签发短期有效的访问令牌和长期有效的续期令牌，并将用户角色信息嵌入令牌中
6. **前端持久化**：前端收到令牌后，持久化存储到本地，更新应用登录状态，跳转至应用主界面
7. **携带令牌访问**：用户后续的 API 请求自动携带访问令牌，由鉴权模块校验身份和权限

### 1.7 状态的业务定义（如适用）

本功能点不涉及状态流转，故无需状态机。登录流程为一次性的请求-响应操作——用户提交凭据后，系统要么验证通过（签发凭证），要么验证失败（返回错误提示）。不涉及中间状态或异步流程。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：用户名或密码错误
- **业务触发条件**：用户提交的用户名不存在，或密码与注册时设置的密码不匹配
- **业务处理策略**：
  1. 使用统一提示语"用户名或密码错误，请重新输入"，不得区分具体是用户名不存在还是密码错误
  2. 不清空用户已输入的用户名，允许用户修正后重新提交
  3. 连续多次登录失败时触犯安全限制（见业务约束清单第 3 条）
- **业务影响**：本次登录被拒绝，用户需修正凭据后重新登录

#### 1.8.2 异常 2：必填字段缺失
- **业务触发条件**：用户未填写用户名或密码直接提交
- **业务处理策略**：
  1. 明确告知用户哪些字段缺失
  2. 不允许进入凭据验证流程
  3. 保留用户已填写的合法信息，仅要求填写缺失字段后重新提交
- **业务影响**：本次登录被拒绝，不产生任何访问凭证

#### 1.8.3 异常 3：账号状态异常
- **业务触发条件**：用户提交的用户名存在但账号处于非正常状态（如被管理员禁用等）
- **业务处理策略**：
  1. 统一提示"当前账号无法登录，请联系管理员"
  2. 不透露具体是何种账号状态异常
  3. 不引导用户重试，引导用户联系客服或管理员
- **业务影响**：本次登录被拒绝，用户需通过人工渠道处理账号问题

#### 1.8.4 异常 4：系统内部错误
- **业务触发条件**：凭据验证过程中系统内部出现异常（如数据库连接失败等）
- **业务处理策略**：
  1. 向用户展示"系统繁忙，请稍后重试"的通用提示
  2. 保持当前登录页面状态，不清空已输入的信息
  3. 允许用户稍后重新提交
- **业务影响**：用户体验短暂受损，重试后应可正常完成登录

### 1.11 业务约束清单

1. **[访问令牌有效期硬约束]** 登录成功后签发的访问令牌有效期最长不超过 15 分钟，超期后必须通过续期令牌刷新或重新登录获取新的令牌。禁止签发永久有效的访问令牌。
2. **[续期令牌有效期硬约束]** 续期令牌有效期最长不超过 7 天，超期后必须重新登录。禁止以任何方式延长续期令牌的有效期。
3. **[登录失败信息模糊保护]** 登录失败时禁止区分"用户名不存在"和"密码错误"，必须统一返回"用户名或密码错误"的提示。连续多次登录失败应触发限制机制，防止暴力破解。
4. **[密码不可逆验证]** 系统在任何场景下均不得存储或比对密码明文。密码验证必须通过不可逆的加密哈希比对完成，哈希算法和参数与安全设计规范一致。
5. **[令牌角色不可篡改]** 登录成功后嵌入令牌的角色信息在令牌有效期内不可修改。如用户的角色发生变更（由管理员操作），用户必须重新登录才能获得包含新角色信息的令牌。
6. **[令牌传输安全]** 访问令牌在客户端和服务端之间传输时，必须放置在 HTTP 请求头的标准认证字段中（Bearer 模式），禁止通过 URL 查询参数或请求体传输令牌。
7. **[数据持久化要求]** 登录验证所使用的用户账号数据必须持久化存储，系统重启或故障后不丢失用户凭据和账号状态，保障已注册用户可正常登录。
8. **[设计边界]** 本模块不负责 Token 过期后的续期（该职责归属 AUTH-03 Token续期）。本模块不负责请求级的权限校验（该职责归属 AUTH-04 五级RBAC鉴权）。本模块不负责前端 Token 的持久化存储和请求拦截注入（该职责归属 AUTH-06 认证会话管理）。本模块不负责登录页面的渲染（该职责归属 AUTH-05 登录注册界面）。本模块不负责账号注册（该职责归属 AUTH-01 用户注册）。

### 1.12 留给规范阶段的技术决策

| 序号 | 技术决策项 | 意图文档中的约束 | 留给规范阶段的理由 |
|------|-----------|-----------------|-------------------|
| 1 | 密码哈希比对算法与参数 | 密码必须通过不可逆的加密哈希比对验证，不可存储或比对明文 | 具体哈希算法（如 bcrypt）、salt rounds 等参数由 spec-writer 根据安全策略和性能基准确定 |
| 2 | 令牌签名算法与密钥管理 | 访问令牌必须具有防篡改能力 | 具体签名算法（如 HS256/RS256）、密钥长度和轮换策略由 spec-writer 根据安全规范确定 |
| 3 | 访问令牌和续期令牌的精确有效期 | 访问令牌不超过 15 分钟，续期令牌不超过 7 天 | 精确的秒级有效期和过期宽容策略由 spec-writer 根据系统架构和用户体验平衡确定 |
| 4 | 续期令牌的轮换机制 | 续期令牌使用一次后作废，防止重放攻击 | 具体的轮换策略（是否同步作废旧续期令牌、轮换的并发安全性）由 spec-writer 根据分布式一致性需求确定 |
| 5 | 登录接口协议 | 接受用户名和密码，返回访问令牌和续期令牌 | 具体 HTTP 方法、路径、请求体格式、响应体结构由 spec-writer 根据 API 设计规范确定 |
| 6 | 数据模型技术类型 | 用户登录凭证包含用户名和密码，输出包含访问令牌、续期令牌和令牌类型 | 具体的 Pydantic/数据库字段类型、Token payload 结构（claims）、编码格式由 spec-writer 确定 |
| 7 | 角色信息在令牌中的存储方式 | 令牌必须包含用户角色信息供后续鉴权 | 具体角色信息的数据结构（单角色 vs 多角色数组）、序列化方式由 spec-writer 根据鉴权模块需求确定 |
| 8 | 暴力破解防护策略 | 连续多次登录失败应触发限制 | 具体的失败次数阈值、限制时长、计数重置策略由 spec-writer 根据安全与用户体验平衡确定 |
| 9 | 异常场景的技术处理 | 登录失败时返回统一模糊提示 | 具体的异常类设计、HTTP 状态码（401 vs 403 vs 400）、错误响应体格式由 spec-writer 根据系统错误处理规范确定 |
