### 1.1 项目定位

认证会话管理模块是篝火智答前端的认证状态中枢。它在用户登录成功后接管认证凭证在客户端的全生命周期管理——包括 Token 的持久化存储、请求自动注入、过期自动续期、以及续期失败后的会话清理与登录引导。该模块运行在微信小程序端，作为用户认证链路的前端锚点，确保用户在一次登录后能够无感维持会话直至主动退出或 Refresh Token 过期，避免反复登录对家属用户造成额外使用障碍。

### 1.4 业务目标

1. 用户登录成功后，系统将访问令牌和刷新令牌持久化到客户端本地存储，小程序冷启动或重启后能从存储中自动恢复会话状态
2. 前端发起的每一次 API 请求自动携带有效的访问令牌，无需用户或开发者手动处理令牌注入
3. 当 API 请求因访问令牌过期返回 401 状态时，系统自动触发令牌续期流程，对用户保持无感
4. 令牌续期连续失败达到 3 次后，系统自动清除所有令牌数据，将会话状态切换为未登录并将用户引导至登录页面
5. 刷新令牌在每次成功续期后被轮换，防止续期请求被重放攻击利用

### 1.5 用户旅程 / 业务流程

1. **用户登录**：用户在登录页输入凭证并提交，登录成功后系统将服务端返回的访问令牌和刷新令牌写入本地存储，同时将会话状态更新为"已登录"。用户在此过程中无感知令牌存储操作。
2. **用户使用功能**：用户正常浏览和操作小程序页面，每次向后端发起的 API 请求由 HTTP 请求拦截器自动从本地存储读取访问令牌并注入请求头。用户在此过程中无感知令牌注入。
3. **访问令牌过期自动续期**：用户持续使用过程中访问令牌过期，API 请求返回 401 状态码。HTTP 拦截器拦截到 401 响应后暂停当前请求，自动使用刷新令牌调用续期接口。续期成功后，新访问令牌替换旧令牌并写入本地存储，拦截器使用新令牌重新发送原始请求。用户在此过程中无感知续期发生。
4. **续期失败后登出**：若刷新令牌也已过期或服务端续请求失败，拦截器在连续 3 次续期尝试均失败后，清除本地存储中的所有令牌数据，将会话状态置为"未登录"，自动跳转至登录页面。用户在此过程中看到登录页面后需要重新登录。
5. **小程序重启后会话恢复**：用户关闭小程序后重新打开，系统从本地存储读取令牌数据。若刷新令牌仍在有效期内，直接从存储恢复会话为"已登录"状态，用户可直接使用无需重新登录。若刷新令牌已过期，会话标记为"未登录"，引导用户重新登录。

### 1.7 状态的业务定义（如适用）

认证会话管理涉及 3 个业务状态：

- **已登录**：本地存储中存在有效的访问令牌或刷新令牌，用户可直接使用小程序功能。进入"已登录"的触发条件：用户成功登录或续期成功。退出触发条件：续期连续失败 3 次（进入"未登录"）。
- **续期中**：访问令牌过期且自动续期流程已启动，正在等待服务端返回续期结果。进入"续期中"的触发条件：拦截器从 401 响应中判断访问令牌过期，开始调用续期接口。退出触发条件：续期成功（回到"已登录"）或续期失败（进入"未登录"）。在"续期中"状态下，后续请求等待续期完成后再决定是否发送。
- **未登录**：本地存储中没有任何有效令牌，用户必须重新登录才能使用小程序功能。进入"未登录"的触发条件：用户手动登出、续期连续失败 3 次、或冷启动后检测到刷新令牌已过期。退出触发条件：用户重新登录成功（进入"已登录"）。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：访问令牌过期但续期成功
- **业务触发条件**：用户当前的访问令牌已过 15 分钟有效期，API 请求返回过期提示；自动续期流程调用服务端续期接口后获得新令牌
- **业务处理策略**：
  1. 续期对用户完全无感——页面不出现任何加载或刷新提示
  2. 新令牌自动写入本地存储，替换旧令牌
  3. 被中断的原始请求使用新令牌自动重发
  4. 用户继续看到业务功能，无任何中断感知
- **业务影响**：无业务影响——用户正常使用不受干扰

#### 1.8.2 异常 2：续期连续失败 3 次
- **业务触发条件**：刷新令牌已过期或服务端无法处理续期请求，续期流程连续失败达到 3 次上限
- **业务处理策略**：
  1. 立即清除本地存储中的所有令牌数据（访问令牌和刷新令牌）
  2. 将会话状态切换为"未登录"
  3. 主动跳转至登录页面，避免用户继续停留在已失去会话的功能页面上
  4. 不清除其他业务数据（如草稿、本地设置），仅清除认证相关数据
- **业务影响**：用户需要重新登录才能继续使用小程序功能；登录前的操作行为和本地数据不受影响

#### 1.8.3 异常 3：小程序重启后令牌全部或部分丢失
- **业务触发条件**：用户关闭小程序后重新打开，本地存储中的令牌数据因存储异常或用户清理缓存而不完整或不存在
- **业务处理策略**：
  1. 冷启动时检查本地存储中是否有完整的访问令牌和刷新令牌对
  2. 令牌不完整或不存在时，将会话状态置为"未登录"
  3. 直接展示登录页面，不尝试使用部分存在的令牌发送请求
- **业务影响**：用户需要重新登录，但业务数据不受影响

#### 1.8.4 异常 4：多个请求同时触发续期
- **业务触发条件**：在续期流程启动后、新令牌返回前，多个其他 API 请求也因 401 响应触发续期逻辑
- **业务处理策略**：
  1. 首个 401 响应触发续期后，设定"续期中"状态锁
  2. 后续其他请求检测到"续期中"状态锁时，暂停发送续期请求，等待第一个续期请求完成
  3. 续期完成后，所有等待中的请求统一使用新令牌重发
  4. 续期失败后，所有等待中的请求不重发，统一触发会话清理和跳转登录页
- **业务影响**：用户感知不到并发续期的存在，体验与单次续期完全一致

### 1.11 业务约束清单

1. **[无感续期约束]** 令牌续期必须对用户完全无感——续期过程中不允许出现加载提示、弹窗或页面跳转。用户在访问令牌过期前后看到的页面状态应无差别。
2. **[续期次数硬上限]** 单次会话续期失败重试次数上限为 3 次。达到上限后必须清除会话并跳转登录页，不可无限重试或静默忽略失败。
3. **[并发续期互斥]** 续期流程执行期间，其他 API 请求即使收到 401 响应也不得发起第二个续期请求。续期状态锁在整个续期完成或失败之前不得释放。
4. **[Token 安全存储约束]** 令牌数据必须使用小程序提供的安全存储机制持久化，不得以明文形式写入非安全存储区域或通过页面参数传递。冷启动恢复令牌时需验证令牌数据结构完整。
5. **[业务数据保护约束]** 会话清理时仅清除认证相关数据（访问令牌、刷新令牌），不得影响用户的其他本地业务数据（如草稿、本地设置、浏览历史）。
6. **[设计边界]** 本模块不负责服务端的 Token 签发、校验和续期逻辑——这些职责归属 AUTH-02 用户登录模块和 AUTH-03 Token续期模块。本模块仅处理客户端侧的令牌管理、自动注入和续期触发。本模块也不涉及用户角色的判断和权限校验，该职责归属 AUTH-04 五级RBAC鉴权模块。
