### 1.1 技术实现思路

AUTH-04 的核心职责是**在业务请求处理管道的入口处，基于 JWT payload 中的角色信息完成权限校验，决定放行或拒绝**。其技术实现围绕三个层次展开：路由级权限校验、字段级脱敏控制、以及角色变更后的 Token 实时失效。

**层次一：路由级权限校验 — FastAPI Depends 注入**

采用 FastAPI 的 `Depends()` 依赖注入系统作为权限校验的执行机制。权限校验不通过独立中间件实现，而是通过 `require_role()` 函数作为路由的 Depends，在路由处理函数执行之前被 FastAPI 框架自动调用。这一选择基于以下考量：

- KNOW-01 已建立 `require_role(["admin", "maintainer"])` 的使用惯例，沿用此模式可保持项目内一致
- Depends 的粒度精确到单个路由端点，允许不同端点声明不同的角色要求，而中间件方案按路径前缀匹配，粒度不够细
- Depends 与 FastAPI 的依赖注入生命周期天然集成，无需额外管道传递上下文

`require_role()` 的实现逻辑：从 `request.state.user`（由上游 AUTH-02 登录模块的 `get_current_user` Depends 注入）读取当前用户的角色列表，与目标资源声明的允许角色列表进行比较。比较规则支持两种模式：

1. **层级累加模式**（默认）：检查用户最高角色层级是否 >= 目标资源要求的最低层级。例如，资源要求 `["teacher"]`，则 `level(teacher)=2 <= level(expert)=3`，专家自动拥有权限。
2. **精确模式**（运维操作专用）：用户角色必须在允许角色集合内的精确命中。例如，`["admin", "maintainer"]` 仅允许管理员和维护人员访问，专家（层级 3 但不在集合内）被拒绝。

> **业务矛盾推断**：意图文档 §1.4 目标 3 明确了累加原则。对于运维操作（意图文档 §1.11 约束 3），精确模式自然满足"仅管理员和维护人员"的限制。`require_role` 在实现上支持两种参数：`require_role(min_level=UserRole.EXPERT)` 触发累加模式，`require_role(exact_roles=["admin", "maintainer"])` 触发精确模式。这种双模式设计兼容了 KNOW-01 已有的精确列表调用方式。参见 §1.6 矛盾处理记录。
>
> **多角色用户推断**：若用户持有多个角色（如同时为"老师"和"专家"），权限判定取最高层级，而非要求同时满足所有角色。此方案与累加原则的精神一致，且 `UserRole.level` 属性的 `max()` 计算即可实现。参见 §1.6 矛盾处理记录。

**层次二：字段级脱敏控制 — 纯函数 + SEC-01 响应中间件**

手机号字段的可见性控制通过分离判责与执行实现。AUTH-04 提供 `get_masked_phone()` 纯函数（判定角色并返回脱敏结果），SEC-01 的响应脱敏中间件在序列化阶段自动调用该函数对手机号字段进行处理。这种设计的原因：

- 意图文档 §1.11 约束 2 明确规定字段级控制**仅**适用于手机号字段，无需设计通用的脱敏框架
- SEC-01 已实现响应脱敏中间件（`masking.py`），且其设计预留了按角色脱敏的扩展点
- 职责分离清晰：AUTH-04 判定"谁看到什么"，SEC-01 执行"如何展示"

**层次三：角色变更后的 Token 实时失效 — Redis 黑名单 + JWT jti 校验**

用户角色变更（如管理员将某家属升级为老师）后，要求旧 Token 立即失效、用户强制重新登录。实现方案是 Redis 黑名单：角色变更时，将被变更用户的所有当前有效 Token 的 `jti` 加入 Redis 黑名单，TTL 设置为 Access Token 有效期（15 分钟）。`get_current_user` Depends 在校验 JWT 签名和过期时间后，额外查询 Redis 黑名单：

- 若 `jti` 命中黑名单 → 拒绝请求，返回 401，要求重新登录
- 若 `jti` 未命中黑名单 → 正常放行

> **矛盾推断**：对于角色变更后已通过鉴权但尚未完成的请求，允许其执行完成（不中断）。理由是：（1）意图文档 §1.8.3 表述"已登录期间的操作不会丢失"，说明业务侧允许已开始的请求继续；（2）技术上，`require_role` 在请求开始时就已完成校验，Redis 黑名单查询也仅发生在请求入口处，不会在请求执行过程中再次检查。这个极短窗口（毫秒级）是可接受的。
>
> **Redis 不可用降级推断**：采用 fail-open 策略——Redis 不可用时，跳过黑名单查询，旧角色 Token 继续有效直到自然过期（最多 15 分钟）。理由：（1）项目已引入 Redis 用于缓存和限流，Redis 故障属于低概率事件；（2）fail-closed 将导致所有请求被拒绝，可用性降为零，影响所有业务模块；（3）15 分钟安全窗口与 Access Token 原生有效期一致，风险可控。

**数据流概览**：

```
客户端请求
  → Nginx (HTTPS)
    → FastAPI 路由匹配
      → [依赖注入链] SEC-05 输入校验 Depends
        → [依赖注入链] AUTH-02 get_current_user Depends (JWT 验证 + Redis 黑名单查询)
          → [依赖注入链] AUTH-04 require_role Depends (角色层级/精确校验)
            → 路由处理函数（业务逻辑）
              → 响应序列化（SEC-01 脱敏中间件调用 get_masked_phone）
```

| `packages/py-logger` | 输出审计日志 | 权限拒绝事件记录结构化日志，含 `user_id`、`target_route`、`required_roles`、`actual_roles`、`timestamp`、`trace_id` |
| SEC-01 传输存储安全 | 下游消费方 | 响应序列化阶段调用 AUTH-04 的 `get_masked_phone()` 进行手机号脱敏 |
| 业务模块（CSLT / CASE / TICK / PROF / KNOW） | 下游消费方 | 各模块路由或 Service 方法上标注 `require_role()` 注解，在请求处理前完成权限校验 |

### 1.4 状态机设计（技术实现策略，如适用）

本功能点不涉及状态流转。权限校验为无状态同步查询，每次请求独立校验，不维护校验状态。意图文档 §1.7 已确认此结论。

唯一的"状态"概念存在于 Redis 黑名单中——被撤销的 Token jti 有明确的 TTL（15 分钟），过期后自动清除。这是键过期行为，非业务状态机。

### 1.6 架构权衡与备选方案

| 决策点 | 最终方案 | 备选方案 | 选择理由 |
|--------|---------|---------|----------|
| 权限执行机制 | FastAPI `Depends()` 依赖注入 | ASGI 中间件 | Depends 粒度精确到单个路由，中间件按前缀匹配粒度不足。KNOW-01 已建立 Depends 惯例，保持一致 |
| 角色命名 | 双命名：英文枚举值 + 中文 display_name | 纯中文/纯英文 | 兼容 KNOW-01 已冻结的 `["admin", "maintainer"]` 调用方式，避免 BREAKING CHANGE |
| 权限配置 | 声明式注解 `require_role(...)` | 配置文件/数据库动态配置 | 项目中角色固定（5 个），无需动态配置。注解方式将权限声明绑定在代码中，阅读即理解 |
| 角色变更失效 | Redis 黑名单 + jti 校验 | 仅依赖 JWT 15min 到期 | 意图文档明确要求角色变更后立即失效。依赖到期存在 15 分钟安全窗口 |
| 字段级脱敏 | 纯函数 `get_masked_phone()` 供 SEC-01 调用 | 在 AUTH-04 中自行实现响应拦截 | 避免重复实现响应拦截逻辑。SEC-01 已预留脱敏中间件，AUTH-04 提供判定函数，职责分离 |
| 拒绝响应格式 | HTTP 403 + `{"detail": "无权限提示"}` | 返回更详细的错误结构 | 意图文档 §1.11 约束 5（信息最小化）禁止透露权限规则细节。格式与已有模块（KNOW-01）一致 |

### 1.7 注意事项与禁止行为（设计层面）

1. **[角色命名一致性]** 双命名体系中，JWT payload、数据库、Python 枚举值统一使用英文小写（`family`/`teacher`/`expert`/`admin`/`maintainer`）。中文名仅作为业务显示名通过 `display_name` 属性在序列化层映射。禁止在 JWT payload 或数据库中使用中文角色值，否则将导致 KNOW-01 等已有模块的 BREAKING CHANGE。

2. **[Depends 链顺序]** `require_role` 必须在 `get_current_user` 之后执行，因为 `require_role` 需要消费 `request.state.user.roles`。路由 Depends 声明顺序必须为：`Depends(get_current_user)` → `Depends(require_role(...))`。禁止在没有 `get_current_user` 的情况下单独使用 `require_role`。

3. **[信息最小化约束]** 拒绝响应仅返回 `{"detail": "当前角色无权执行此操作，如需权限请联系管理员"}`。禁止在拒绝响应中返回 `expected_role`、`required_level` 等权限规则细节，防止权限信息泄露。此约束直接源于意图文档 §1.11 约束 5。

4. **[设计边界]** 本模块不负责用户身份的认证（JWT 创建/验证归 AUTH-02）、Token 续期（归 AUTH-03）、响应脱敏执行（归 SEC-01）。AUTH-04 仅负责：角色枚举定义、角色权限校验、字段级脱敏判定函数、Redis 黑名单维护（与 AUTH-02 的 `get_current_user` 联动）。

5. **[禁止绕过权限校验]** 禁止在路由或 Service 层通过 `@router.get(...)` 直接调用业务逻辑而不经过 `require_role` Depends。所有需要权限保护的端点必须在路由装饰器或全局 Depends 中显式声明角色要求。

6. **[禁止将脱敏逻辑嵌入业务代码]** 手机号脱敏必须在 SEC-01 的响应中间件层统一处理，禁止在各个业务 Service 方法中重复实现 `get_masked_phone()` 调用。业务模块只需正常返回完整手机号数据，脱敏由中间件自动完成。

7. **[黑名单 Key 命名规范]** Redis 黑名单 Key 必须使用统一前缀 `token_blacklist:`，禁止不同模块或不同场景使用不同的前缀格式，否则将导致运维混乱。
