### 1.1 项目定位

登录注册界面是「篝火智答」平台的用户身份入口页面，也是用户首次接触平台的第一站。它为未登录用户提供登录已有账号和注册新账号两个核心功能入口，接收用户输入的身份凭证（用户名和密码）或注册信息（含角色选择），通过认证会话管理模块（AUTH-06）的 Hook 桥接层将数据提交给后端认证服务，并根据返回结果切换界面状态。在整体流程中，它是一个**用户交互节点**——接收用户输入，交由逻辑层处理后端通信，将后端结果以用户可理解的形式呈现给用户。本页面不直接调用任何 API 接口，不存储或传递任何认证令牌，所有数据收发均通过逻辑层的 Hook 进行。

- **业务依赖的其他功能模块**：

### 1.4 业务目标

1. 用户在登录页面输入用户名和密码后，系统在合理时间内完成身份验证并将结果反馈给用户，登录成功后自动跳转至应用主页面
2. 用户在注册页面选择角色并填写注册信息提交后，系统完成账号创建，注册成功后引导用户至登录页面
3. 输入校验不通过时，界面在 500 毫秒内呈现明确的错误反馈，告知用户具体哪个字段不符合要求
4. 未登录状态下，非认证页面的访问请求被自动重定向到登录页面，确保未认证用户无法访问受限内容
5. 整个页面仅负责用户输入呈现、事件收集和结果展示，不含任何 API 调用或 Token 存储逻辑

### 1.5 用户旅程 / 业务流程

**场景一：用户登录**
1. **进入登录页**：用户打开应用，因未登录被自动展示登录页面（或用户主动从注册页切换至登录页）
2. **输入凭证**：用户在登录表单中输入用户名和密码，可选择勾选"记住我"
3. **提交登录**：用户点击"登录"按钮，表单数据通过 useAuth Hook 提交至后端进行身份验证
4. **结果处理**：
   - 登录成功：页面展示成功状态，自动跳转至应用主页面
   - 登录失败：页面在 500 毫秒内展示"用户名或密码错误"的错误提示，保留已填写内容，允许用户修正后重新提交
5. **登录后体验**：登录成功后，用户进入应用主页，可使用应急咨询、档案管理、科普查阅等全部功能

**场景二：用户注册**
1. **进入注册页**：用户在登录页点击"注册"按钮，切换到注册页面
2. **选择角色**：用户从"家属"、"老师"、"专家"三种角色中选择一项，角色选择后不可在注册流程中变更
3. **填写信息**：用户填写用户名、密码、手机号码等必要注册信息，界面实时校验输入合法性
4. **提交注册**：用户确认信息后点击"注册"按钮，表单数据通过 useAuth Hook 提交至后端
5. **结果处理**：
   - 注册成功：页面展示成功提示，引导用户前往登录页面进行登录
   - 注册失败：页面在 500 毫秒内展示具体失败原因（如用户名已被注册、密码强度不足等），保留已填写的合法信息，要求用户修正后重新提交
6. **注册后体验**：注册成功仅是账号创建完成，用户需主动前往登录页使用刚创建的账号登录后，才能使用平台全部功能

### 1.7 状态的业务定义（如适用）

本功能点涉及 5 个业务状态：

- **空闲**：页面已加载完成，等待用户操作（填写表单、切换登录/注册、点击按钮等）
- **输入中**：用户正在填写表单内容，界面实时校验输入合法性，展示校验反馈
- **提交中**：用户已点击"登录"或"注册"按钮，数据已提交但尚未收到后端响应，界面展示加载状态，禁止用户重复提交
- **成功**：登录或注册成功，界面展示成功提示并执行跳转或引导操作
- **失败**：登录或注册失败，界面展示错误提示，保留已填写的合法字段内容，允许用户修正后重新提交

进入"提交中"的条件：用户点击"登录"或"注册"按钮且前端表单校验通过。
退出"提交中"的条件：收到后端响应（进入"成功"或"失败"），或网络异常（进入"失败"）。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：输入信息不合法

- **业务触发条件**：用户提交的登录或注册信息不符合业务规则（如用户名格式错误、密码强度不足、必填字段缺失、手机号格式不正确）
- **业务处理策略**：
  1. 在用户输入过程中实时校验，不合规的字段在输入完成后立即在字段下方展示提示信息
  2. 提交时再次校验全部字段，任一字段不合规则不允许提交
  3. 保留用户已填写的合法字段内容，仅要求修正不合规的字段
- **业务影响**：本次提交被阻止，用户需修正不合规字段后重新提交

#### 1.8.2 异常 2：登录失败

- **业务触发条件**：用户提交的登录信息格式正确，但后端验证发现用户名不存在或密码不匹配
- **业务处理策略**：
  1. 统一展示"用户名或密码错误"的提示，不区分告知用户具体是"用户名不存在"还是"密码错误"
  2. 不清空用户已填写的用户名和密码，允许用户直接修改后重新提交
  3. 密码输入框可提供"清空重新输入"的便捷操作
- **业务影响**：用户无法登录，保持在登录页面，可继续尝试

#### 1.8.3 异常 3：提交过程中网络异常或服务不可用

- **业务触发条件**：用户提交登录或注册请求后，因网络故障或服务端异常无法收到有效响应
- **业务处理策略**：
  1. 提交等待超时后，展示"网络异常，请检查网络后重试"的通用提示
  2. 保持表单状态和已填写内容不变，用户可重新提交
  3. 防止重复提交——用户点击提交后按钮立即置为不可用状态，收到响应或超时后恢复可用
- **业务影响**：用户体验短暂受损，表单内容不丢失，用户重试后可正常提交

### 1.11 业务约束清单

1. **[错误反馈时效]** 所有登录或注册失败场景下，错误提示必须在 500 毫秒内呈现在用户界面上，避免用户等待不确定感。
2. **[登录失败信息保护]** 登录失败时不区分告知用户是"用户名不存在"还是"密码错误"，统一提示"用户名或密码错误"，防止攻击者探测有效账号。
3. **[禁止直连后端]** 页面代码中不得出现任何直接调用 HTTP 客户端或后端 API 的语句，所有数据收发必须通过 useAuth Hook 桥接层进行。
4. **[禁止 Token 操作]** 页面代码中不得出现任何 Token 的读取、存储、传递或清除逻辑，认证令牌的管理完全由 AUTH-06（认证会话管理）模块负责。
5. **[角色选择不可逆]** 用户在注册流程中选择角色后，提交前不可变更角色类型，如需变更必须重新开始注册流程。
6. **[提交防重复]** 用户点击"登录"或"注册"按钮后，按钮应立即置为不可用状态，防止用户因网络延迟而重复提交，直至收到响应或超时后方可恢复。
7. **[输入保留]** 登录或注册失败时，用户已填写的合法字段内容应保留，不得清空整个表单，仅要求用户修正被提示的错误字段。
8. **[设计边界]** 本模块不负责用户身份验证、账号创建、Token 签发与管理等任何后端服务逻辑，这些职责归属 AUTH-01（用户注册）、AUTH-02（用户登录）和 AUTH-06（认证会话管理）模块。本模块不负责密码的加密强度校验和哈希存储，密码安全性由后端服务保障。
9. **[页面切换清空]** 用户在登录页和注册页之间切换时，已填写的表单内容应清空，避免混淆。
