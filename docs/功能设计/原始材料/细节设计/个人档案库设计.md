# 「篝火智答」个性化档案库设计方案

## 一、概述

### 1.1 核心定位
个性化档案库是**以单个孤独症儿童为单位的干预事件时间线**。它不做知识检索，不做语义匹配，而是作为应急咨询时的**「个体上下文」**，在 RAG 生成建议前直接注入 Prompt，让 AI 知道「这个孩子以前发生过什么、什么方法有效」。

### 1.2 与标准案例库的关系

| 维度 | 个性化档案库 | 标准案例库 |
|---|---|---|
| **本质** | 个人病史/干预史 | 通用医学知识 |
| **存储** | MySQL 结构化数据 | ChromaDB 向量 + MySQL 元数据 |
| **检索方式** | 精确匹配 `child_id` | 语义相似度检索 |
| **消费方式** | 直接注入 Prompt（第一轮） | RAG 检索后注入 Prompt（第二轮） |
| **可见范围** | 仅关联家长 + 关联老师/专家 | 全员开放 |
| **内容来源** | 家长日常记录 + 老师补充 | 专家提炼 + 机构脱敏 + 工单沉淀 |

### 1.3 设计原则
- **家长主权**：家长拥有档案的绝对控制权，可随时解除老师关联
- **老师补充**：老师仅可补充专业评估，不可修改家长记录
- **轻量启动**：冷启动仅需 3 个必填标签，其余在使用中逐步完善
- **物理隔离**：按 `child_id` 严格隔离，无关联者绝对不可访问

## 二、数据模型

### 2.1 儿童基础档案（Child Profile）

**说明**：一个家长账号下可创建多个儿童档案，应急咨询时自动关联默认档案。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `child_id` | UUID | 是 | 儿童唯一标识 |
| `parent_id` | FK | 是 | 关联家长用户 |
| `nickname` | String | 否 | 家长自定义称呼（如「小明」），仅家庭内可见 |
| `birth_date` | Date | 是 | 出生日期，系统自动计算年龄区间 |
| `diagnosis_type` | Enum | 是 | **冷启动必填**：诊断类型（ASD / 疑似 ASD / 其他发育障碍） |
| `primary_behavior` | Enum | 是 | **冷启动必填**：主要行为类型（刻板 / 情绪崩溃 / 自伤 / 攻击 / 社交退缩 / 多动） |
| `language_level` | Enum | 否 | 语言水平（无语言 / 单字词 / 短句 / 可对话），后续完善 |
| `sensory_profile` | JSON | 否 | 感官特征（听觉敏感 / 触觉敏感 / 味觉敏感 / 前庭寻求等），多选 |
| `triggers` | JSON | 否 | 已知情绪触发因素（噪音 / 环境变化 / 陌生人 / 任务中断等），后续完善 |
| `medication_notes` | Text | 否 | 用药记录（仅家长和老师可见，应急时作为安全参考） |
| `is_default` | Boolean | 是 | 是否为该账号下默认关联档案 |
| `created_at` | DateTime | 是 | 创建时间 |
| `updated_at` | DateTime | 是 | 最后更新时间 |

### 2.2 干预事件记录（Event Log）—— 即「个性化案例」

**说明**：家长每次记录 = 新增一条事件。这是档案库的核心数据，应急咨询时直接读取注入 Prompt。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `event_id` | UUID | 是 | 事件唯一标识 |
| `child_id` | FK | 是 | 关联儿童 |
| `recorded_by` | FK | 是 | 记录人（家长或老师） |
| `recorded_by_role` | Enum | 是 | `parent` / `teacher` / `expert` |
| `event_time` | DateTime | 是 | 事件发生时间（家长可补录历史事件） |
| `behavior_type` | Enum | 是 | 本次事件行为类型（同基础档案枚举） |
| `severity_level` | Enum | 是 | 家长自评严重程度：轻 / 中 / 重 |
| `setting` | Enum | 否 | 发生场景：家庭 / 学校 / 公共场合 / 机构 |
| `trigger_description` | Text | 是 | **触发因素描述**（如「商场吹风机声音」「换床单」） |
| `manifestation` | Text | 是 | **具体表现**（如「捂耳朵蹲下，拒绝移动，持续尖叫」） |
| `intervention_tried` | Text | 是 | **家长尝试了哪些干预**（如「拉他起身无效，给了水不喝」） |
| `intervention_result` | Text | 是 | **结果如何**（如「3 分钟后自己停止，靠在我肩上」） |
| `follow_up_by_teacher` | Text | 否 | **老师补充的专业评估**（如「建议下次提前带降噪耳机，属感官过载反应」） |
| `is_professional` | Boolean | 是 | 是否包含老师专业补充（`true` 表示有老师补充内容） |
| `tags` | JSON | 否 | 本次事件的自定义标签（如「吹风机」「商场」） |
| `created_at` | DateTime | 是 | 记录时间 |

**记录形式**：采用**结构化表单 + 自由文本补充**。家长填写上述字段，系统引导式提问降低认知负担，同时允许在 `manifestation` 等字段自由描述细节。

### 2.3 老师专业评估记录（Professional Note）

**说明**：老师/专家不直接修改家长的 Event Log，而是独立提交专业评估，系统将其**挂载**到相关 Event 下。

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `note_id` | UUID | 是 | 评估唯一标识 |
| `event_id` | FK | 是 | 关联事件 |
| `teacher_id` | FK | 是 | 提交老师/专家 |
| `assessment` | Text | 是 | 专业评估（如功能行为分析 FBA 简要结论） |
| `recommended_action` | Text | 是 | 建议的干预策略 |
| `evidence_basis` | Enum | 否 | 依据：NCAEP EBP / 机构经验 / 个案观察 |
| `created_at` | DateTime | 是 | 提交时间 |

### 2.4 标签体系（Tag System）

**说明**：预设标签 + 有限自定义。

| 类型 | 内容 | 使用场景 |
|---|---|---|
| **预设标签** | 年龄区间、诊断类型、语言水平、行为类型（刻板/自伤/攻击/逃跑/情绪崩溃）、感官特征（听觉/触觉/前庭等）、触发因素（噪音/变化/社交压力等） | 冷启动必填 + 事件记录勾选 |
| **自定义标签** | 家长在限定范围内添加（如「害怕吹风机」「只肯穿蓝色衣服」「旋转物迷恋」） | 事件记录时添加，最多 5 个/事件 |

**标签自定义规则**：
- 家长输入自定义标签时，系统提供**推荐标签池**（基于历史高频标签）
- 新标签需经简单归一（去除特殊符号，限制 10 字以内）
- 自定义标签仅在该儿童档案内生效，不进入全局标签库

## 三、记录流程设计

### 3.1 家长记录路径

```
家长打开「记录」页面
    ↓
选择儿童（多孩家庭）→ 默认自动关联
    ↓
系统引导式提问：
  1. 「今天发生了什么？」→ 选择行为类型（图标化）
  2. 「在哪里发生的？」→ 选择场景
  3. 「是什么引发的？」→ 输入触发因素（提供历史标签推荐）
  4. 「孩子具体表现是什么？」→ 自由文本描述
  5. 「你做了什么？结果怎么样？」→ 干预动作 + 结果
    ↓
可选：添加自定义标签
    ↓
保存 → 生成一条 Event Log
```

### 3.2 老师补充路径

```
老师在「档案共建」页面查看关联儿童的 Event Log
    ↓
选择某条事件 → 点击「补充专业评估」
    ↓
填写：
  - 专业评估（功能行为分析视角）
  - 建议干预策略
  - 依据（可选）
    ↓
提交 → 该事件标记 `is_professional = true`
    ↓
家长收到通知：「XX 老师为您的记录补充了专业建议」
```

### 3.3 冷启动引导

**首次使用**：
- 必填：出生日期、诊断类型、主要行为类型（3 项，30 秒完成）
- 提示：「完善更多信息，能让 AI 给出更精准的建议」→ 引导补充语言水平、感官特征

**渐进式完善**：
- 每次应急咨询结束后，弹出 1-2 题微问卷（如「本次触发因素是什么？」「干预是否有效？」）
- 答案自动沉淀为档案标签或 Event Log

## 四、与应急咨询的联动机制（第一轮注入）

### 4.1 注入时机
家长发起应急咨询时，**在 RAG 检索标准案例库之前**，系统先读取该儿童的个性化档案。

### 4.2 注入内容筛选

| 步骤 | 操作 | 数量限制 |
|---|---|---|
| **Step 1** | 读取该儿童最近 **30 天内**的 Event Log | 最多 10 条 |
| **Step 2** | 筛选与**当前查询行为类型相同**的事件 | 最多 5 条 |
| **Step 3** | 若同类事件不足 2 条，补充**最近 3 条**其他类型事件（作为背景） | 总计不超过 5 条 |
| **Step 4** | 优先选取 `is_professional = true` 的事件（有老师评估的） | 排序加权 |

### 4.3 注入文本格式

系统读取 Event Log 后，格式化为以下文本注入 System Prompt：

```
【该儿童个性化干预历史】
儿童基础档案：7岁，ASD，主要行为类型为刻板+情绪崩溃，听觉敏感。

近期相关干预记录：
1. [2026-05-20 商场] 因吹风机声音崩溃，捂耳朵蹲下拒绝移动。家长尝试拉起身无效。后使用降噪耳机+挤压玩具，3分钟平复。（老师评估：感官过载，建议提前携带降噪工具）
2. [2026-05-15 家中] 睡前换床单引发情绪崩溃，尖叫拒睡。改用旧床单+轻拍背部，10分钟入睡。（家长记录）
3. [2026-04-28 学校] 任务被中断后攻击同学。老师使用反应中断/重定向策略，带离至安静角，5分钟平复。（老师评估：任务转换困难，建议使用视觉时间表预告）
```

### 4.4 注入位置

在 Prompt 中的顺序：

```
System Prompt（角色定位 + 安全约束）
  ↓
【个性化干预历史】（第一轮注入）
  ↓
【参考通用干预方案】（第二轮：标准案例库检索结果）
  ↓
User Query（家长当前输入）
  ↓
输出指令（四段式结构）
```

**设计意图**：让 LLM 先「认识这个孩子」，再「参考通用方案」，生成的建议自然带有个体适配性。

## 五、隐私与权限控制

### 5.1 访问矩阵

| 操作 | 家长本人 | 关联老师/专家 | 其他用户 | 管理员 |
|---|---|---|---|---|
| 查看基础档案 | ✅ | ✅ | ❌ | ⚠️ 仅运维，不看内容 |
| 查看 Event Log | ✅ | ✅ | ❌ | ⚠️ 仅运维 |
| 新增 Event | ✅ | ❌ | ❌ | ❌ |
| 补充专业评估 | ❌ | ✅ | ❌ | ❌ |
| 修改/删除 Event | ✅（仅自己创建的） | ❌ | ❌ | ❌ |
| 解除老师关联 | ✅（随时，无需审核） | 被解除后失去访问权 | — | — |

### 5.2 数据隔离

- **物理隔离**：所有个性化档案数据按 `child_id` 分表/分区存储
- **查询隔离**：所有 API 强制携带 `child_id` 参数，后端校验该用户是否有权访问
- **脱敏要求**：Event Log 中不得出现真实姓名、学校全称、家庭住址（家长填写时系统实时检测并提示脱敏）

---

## 六、边界与约束

- **不入向量库**：个性化档案数据绝对不以任何形式进入 ChromaDB 或任何向量索引，避免语义检索导致的交叉污染
- **不做全文检索**：家长不可在档案库内搜索历史记录，仅支持按时间/行为类型筛选
- **不对外展示**：个性化档案内容不向其他家长、未关联专家、公众展示
- **无激励机制**：家长记录行为不设积分/奖励，避免记录失真
- **容量限制**：单个儿童档案最多保留 500 条 Event Log，超出后自动归档最早记录（仅保留基础字段，详细描述压缩存储）

## 七、待明确项

1. **老师补充的触发方式**：是家长主动邀请老师补充某条记录，还是老师定期主动查看并补充？
2. **Event Log 的编辑窗口**：家长提交后是否允许修改？若允许，多长时间内可修改？
3. **多孩切换的默认逻辑**：应急咨询时自动关联哪个孩子？是否基于最近活跃记录自动推断？
