### 1.1 项目定位

置信度后校验是篝火智答应急咨询安全保障流水线的最后一道关卡。它位于应急方案生成完成之后、人工工单触发之前，对已生成的应急方案内容进行质量可信度评估——检查方案是否可靠、是否需要追加额外安全提示、是否需要启动人工兜底。在整体业务流程中，它是区分「AI 可独立交付」与「必须人工介入确认」的关键决策节点，也是平台对用户安全承诺的核心执行机制。

### 1.4 业务目标

1. 每次应急方案生成完成后，必须在可接受时间内完成置信度后校验，并输出明确的判定结果给下游模块消费
2. 高危关键词（自伤/自杀/药物）命中时，必须百分之百阻断深度回答，仅输出预设的安全提示文本，不得有任何遗漏
3. 置信度分数低于 0.7 时，必须百分之百追加强制免责提示，并异步触发人工工单创建，不得遗漏
4. 置信度复合评分必须同时包含 LLM 自评估（内容可信度）和规则校验（来源引用完整性、语义连贯性）两个维度，单一维度不可独立作为最终判定依据
5. LLM 自评估不可用时，必须降级为纯规则评分并记录降级原因，不得因单点故障而中断整个校验流程
6. 校验完成后输出的置信度分数和判定结论必须持久化到咨询历史中，供家属后续查阅和平台质量分析

### 1.5 用户旅程 / 业务流程

本模块为后台校验服务，无面向终端用户的直接交互界面。以下从系统内部业务流程描述其工作场景：

1. **接收校验请求**：CSLT-03 应急方案生成完毕后，将完整的应急方案全文（四段式结构化建议、来源引用清单、免责声明）、上游 CSLT-01 判定的危机等级和阻断深度回答标记、患者行为描述原文一并传入本模块，启动后校验流程

2. **高危关键词检测（第一优先级）**：本模块首先对方案全文进行高危关键词扫描。若命中自伤、自杀、药物等高危关键词，立即判定为强制阻断——直接替换方案为预设安全提示文本（如「请立即移除周围危险物品，确保患者安全；本情况建议立即就医或联系专业医疗机构」），同时异步调用 TICK-01 创建特急工单。关键词未命中则进入置信度校验流程

3. **置信度计算（关键词未命中时）**：本模块对方案内容进行双重评估——LLM 自评估（让大模型评估自身生成内容的可信度，如来源引用是否充分、逻辑是否连贯、是否有无来源声明的内容）和规则校验（来源引用覆盖率、免责声明是否完整、四段式结构是否齐全），两者加权得到 0 到 1 之间的置信度分数

4. **判定与动作**：若置信度大于等于 0.7，标记为「通过」，方案全文不做修改，直接交付下游。若置信度低于 0.7，在方案全文末尾追加强制免责提示（如「以上建议由 AI 生成，置信度偏低，强烈建议联系专业专家进行确认」），同时异步调用 TICK-01 创建人工工单（工单携带当前咨询上下文）

5. **结果交付**：将最终置信度分数、后校验判定结果（通过/追加强制提示/强制阻断）、修改后的方案全文（如有追加或替换）、工单触发标记一并传递给 CSLT-08 编排层，由其决定前端的 UI 展示策略

6. **持久化记录**：校验完成后，置信度分数和判定结论写入咨询历史中，供家属回溯和平台质量分析使用

### 1.7 状态的业务定义

本模块为无状态校验服务——每次应急咨询独立执行后校验，不维护跨咨询会话的持久状态。单次校验流程的内部阶段如下：

- **等待输入**：尚未接收到上游的完整输入数据，空闲等待中
- **关键词检测中**：正在对方案全文执行高危关键词扫描
- **置信度计算中**：关键词未命中，正在执行 LLM 自评估和规则校验的复合评分计算
- **降级模式**：LLM 自评估不可用，正在以纯规则评分方式计算置信度
- **已完成**：校验流程结束，判定结果已产出，等待下游模块消费

进入「关键词检测中」的条件：接收到完整输入数据，且上游未标记阻断深度回答。
退出「关键词检测中」的条件：关键词扫描完成，命中时直接进入「已完成」；未命中时进入「置信度计算中」。
进入「置信度计算中」的条件：关键词扫描未命中任何高危词。
退出「置信度计算中」的条件：复合评分计算完成，进入「已完成」。
进入「降级模式」的条件：LLM 自评估调用失败或超时。
退出「降级模式」的条件：纯规则评分计算完成，进入「已完成」。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：LLM 自评估不可用

- **业务触发条件**：LLM 自评估接口调用失败、超时或返回的评估结果格式无法解析
- **业务处理策略**：
  1. 自动降级为纯规则评分模式，仅基于来源引用覆盖率、免责声明完整性、四段式结构完整性等可量化指标计算置信度分数
  2. 在判定结果中记录降级原因，供平台质量分析追溯
  3. 降级后的置信度分数标注为「纯规则评分」，与正常复合评分做区分
- **业务影响**：置信度分数的准确性可能降低，但校验流程不会中断，用户体验不受影响

#### 1.8.2 异常 2：置信度校验流程超时

- **业务触发条件**：置信度校验整体流程耗时超过上游约定的可接受时间
- **业务处理策略**：
  1. 默认按照「置信度不足」处理——追加强制免责提示，并触发工单创建
  2. 标注超时原因，供后续质量分析追溯
  3. 不得因超时而阻塞整体应急咨询流程
- **业务影响**：可能产生不必要的工单（实际方案质量可接受但校验超时），但确保安全底线不突破——宁可多触发一个工单，不可因校验超时而放行低质量方案

#### 1.8.3 异常 3：工单创建接口调用失败

- **业务触发条件**：本模块判定需要触发工单，但调用 TICK-01 的工单创建接口时返回失败
- **业务处理策略**：
  1. 将工单创建失败作为独立异常事件记录，供平台运维监控
  2. 本模块自身流程不受影响——置信度分数和判定结果仍正常产出和交付
  3. 在输出中标注「工单创建失败，请手动联系专家」，由 CSLT-08 编排层在前端向家属展示此信息
- **业务影响**：用户无法自动获得人工兜底，需要手动发起求助或等待系统恢复后补创建

### 1.11 业务约束清单

1. **[安全底线]** 高危关键词（自伤、自杀、药物）命中时，必须百分之百阻断 AI 深度回答，仅输出预设安全提示文本。此项为不可妥协的安全红线，不允许任何场景下的绕过
2. **[安全兜底]** 置信度分数低于 0.7 时，必须追加强制免责提示文本，且必须触发人工工单创建。置信度校验不可用时，默认按「置信度不足」处理
3. **[内容完整性]** 本模块只可在方案全文末尾追加警示文本或整体替换为安全提示，不得修改 CSLT-03 生成的方案正文的任何原始内容。用户需要能够区分哪些是 AI 原始建议、哪些是系统追加的安全提示
4. **[时效性]** 校验流程必须在方案生成完成后尽快完成，不得因校验延迟而显著影响用户体验。超时需按安全策略兜底，不得阻塞流程
5. **[可追溯性]** 每次校验的置信度分数、判定结果和降级原因必须完整记录，支持按咨询维度查询和按时间维度统计，供质量分析和词库迭代使用
6. **[设计边界]** 本模块不负责工单的紧急分级（该职责归属 TICK-02）、不负责工单的专家分配与接单（该职责归属 TICK-03）、不负责案例的审核工作流（该职责归属 CASE-03）。本模块仅负责判断「是否需要人工介入」，具体的人工介入执行由工单模块负责

### 1.12 留给规范阶段的技术决策

| 序号 | 技术决策项 | 意图文档中的约束 | 留给规范阶段的理由 |
|------|-----------|-----------------|-------------------|
| 1 | LLM 自评估与规则校验的加权比例 | 必须包含两个维度，不可单一维度独立判定 | 具体加权比例（如 LLM 自评估 60% + 规则校验 40%）需由 spec-writer 根据技术验证确定 |
| 2 | 来源引用完整性的量化方式 | 必须可量化，不得基于主观判断 | 具体计分算法（如被引用案例数 / 总返回案例数、来源标注格式校验）由 spec-writer 确定 |
| 3 | 语义连贯性的判断标准 | 必须可量化 | 具体判定方式（如基于规则的关键段落检测 vs 基于嵌入向量的连贯性评估）由 spec-writer 根据技术可行性确定 |
| 4 | LLM 自评估的提示设计 | LLM 需评估内容可信度，如来源引用是否充分、有无无来源声明的内容 | 具体的系统指令文本、Few-shot 示例和输出格式要求由 spec-writer 设计 |
| 5 | 高危关键词完整列表及维护机制 | 必须覆盖自伤、自杀、药物及相关语义变体 | 完整的关键词列表、否定词过滤规则和动态更新机制（如从 Redis 热更新）由 spec-writer 根据 CSLT-01 和 SEC-02 的共享词库定义 |
| 6 | 追加强制免责提示的精确文案 | 必须明确提示用户方案可信度不足，引导联系专家 | 低置信度场景和高危关键词阻断场景的预设安全提示精确文案由 spec-writer 与产品方确定 |
| 7 | 异步调用 TICK-01 的容错和重试策略 | 失败时不影响本模块自身流程，需记录和告知用户 | 具体的重试次数、重试间隔和失败后的用户提示文案由 spec-writer 确定 |
| 8 | 置信度分数的存储格式和精度 | 必须持久化，支持查询和统计 | 存储的数据类型（浮点数精度）、数据库字段设计和索引策略由 spec-writer 确定 |
