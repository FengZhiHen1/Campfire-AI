### 1.1 项目定位

案例录入管理是真实案例库的**入口节点**，位于整体业务流程的第一阶段。它负责接收老师或专家提交的真实干预案例元数据，并支持对已有案例进行编辑修订。本模块的产出（案例数据）是整个案例库生命周期的起点——录入后的案例进入审核流程，审核通过后经切片和向量化成为 RAG 检索引擎的知识燃料，为智能应急咨询提供可检索、可溯源、可执行的案例支撑。

在整体流程中，本模块是一个**数据采集与编辑节点**：输入专业人员撰写的干预经验文本，输出结构化的案例草稿，交给下游的案例审核模块处理。

### 1.4 业务目标

1. 老师或专家能够通过系统提交包含完整六项必填字段的真实干预案例元数据，提交成功率不低于 98%
2. 已提交的案例允许作者在审核前进行编辑修订，编辑后案例状态自动重置为草稿状态以确保重新审核
3. 案例录入支持至少两条并行来源通道：专家自行撰写、合作机构脱敏提供，无需对录入流程做分支差异化处理
4. 录入的案例数据在提交后即刻对 CASE-03 审核模块可见，审核流程在提交后的 1-2 个工作日内由系统保障启动
5. 录入过程中涉及的个人身份信息（PII）应在提交前经过检测提示，不得将未脱敏的案例直接通过录入入口入库

### 1.5 用户旅程 / 业务流程

1. **专家撰写案例**：认证老师或专家登录系统，进入案例管理界面，点击"录入新案例"。系统展示案例录入表单，包含原始叙事撰写区和结构化字段填写区。
2. **填写案例信息**：专家以自然语言撰写案例的完整场景描述（L1 原始叙事），包括：场景背景、患者基本特征、行为表现细节、干预过程描述、结果与反思。同时填写结构化字段（L2 卡片层），包括行为类型、适用年龄区间、严重程度、发生场景、循证实践标签、循证等级、四段式干预输出等。
3. **关联附件（可选）**：专家可为案例附加参考文件（如评估报告、干预记录照片），系统通过附件模块处理上传和存储，案例表仅记录引用路径。
4. **提交审核**：专家确认信息完整后点击"提交审核"。系统对必填字段进行完整性校验，并对叙事文本进行 PII 脱敏检测。如检测到疑似 PII（真实姓名、地址、学校名等），向用户提示并要求修正或确认已脱敏。校验通过后案例进入"待审核"状态。
5. **编辑已提交案例**：在案例进入专家终审前，作者可对案例内容进行编辑修改。编辑后案例状态自动从"待审核"重置为"草稿"，需重新提交后方可进入审核流程。

### 1.7 状态的业务定义

案例录入管理涉及以下业务状态：

- **草稿**：案例正在进行内容填写，尚未提交审核。此时案例仅对撰写者本人可见，不进入任何下游流程。
- **待审核**：案例已完成内容填写并通过前端完整性校验，已提交进入 CASE-03 审核工作流。此时案例对审核人员可见，但尚未有审核结论。
- **已驳回**：案例被审核流程驳回，返回至撰写者进行修改。驳回时附有审核意见，撰写者可据此修改后重新提交。

进入"待审核"的条件：撰写者确认六项必填字段均已填写完成，PII 脱敏检测未发现未处理的高风险项，且点击"提交审核"操作。

退出"待审核"的条件：CASE-03 审核工作流完成审核并给出通过或驳回结论（不属于本模块状态机范围）。

进入"已驳回"的条件：CASE-03 审核模块返回驳回结论时，本模块将案例状态同步为已驳回。

退出"已驳回"的条件：撰写者完成修改后重新提交，案例状态恢复为"待审核"。

> 注：审核通过后的状态（"已通过"）、案例发布、向量化入库等状态由 CASE-03（审核工作流）和 CASE-04（向量化入库）模块管理，不属于本模块的状态机范围。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：必填字段缺失

- **业务触发条件**：撰写者未填写全部六项必填字段（案例标题、原始叙事文本、案例来源类型、行为类型、适用年龄区间、适用严重程度、循证实践标签、家属端展示大类、四段式输出四字段、循证等级、禁忌与注意事项、是否模板）而尝试提交
- **业务处理策略**：
  1. 明确告知用户哪些字段尚未填写或不符合要求
  2. 不允许案例进入提交流程
  3. 标记缺失字段在表单中的位置，引导用户补全
- **业务影响**：本次提交请求被拒绝，案例保留在草稿状态，已填写的字段内容不丢失

#### 1.8.2 异常 2：PII 脱敏检测告警

- **业务触发条件**：系统在原始叙事文本中检测到疑似个人身份信息（真实姓名、手机号码、家庭住址、学校名称、身份证号）
- **业务处理策略**：
  1. 在提交前向撰写者展示检测到的疑似 PII 字段列表及在文本中的位置
  2. 提示撰写者确认这些内容是否已经过脱敏处理（如使用化名或通用化改写）
  3. 如撰写者确认已脱敏，允许继续提交并记录确认日志
  4. 如撰写者未做处理，引导其进行脱敏修改后方可提交
- **业务影响**：案例提交流程临时中断，不影响草稿数据完整性。本检测为辅助提示功能，不替代审核阶段的专业 PII 审查

#### 1.8.3 异常 3：循证等级与循证标签不一致

- **业务触发条件**：撰写者将循证等级标注为"NCAEP 循证实践（EBP）"，但所选循证实践标签中包含不属于 NCAEP 2020 确认的 28 种 EBP 之外的标签
- **业务处理策略**：
  1. 提交时检测 EBP 标签与循证等级的一致性
  2. 如发现不一致，提示撰写者核对：将循证等级降级为"机构经验总结"或修正标签为 NCAEP EBP 范围内的标签
  3. 允许撰写者在确认后仍以当前填写内容提交（不一致不强制阻断），但标记该案例在审核阶段需重点关注此项
- **业务影响**：案例可正常提交，但审核阶段可能因此项被驳回

|--------|---------|-------------|
| 用户认证与授权模块（AUTH-04 五级RBAC鉴权） | 上游前置条件 | 案例录入功能仅对"老师"和"专家"角色开放；家属、管理员、维护人员无权录入案例 |
| CASE-02 案例附件管理 | 横向协作 | 案例录入时可关联上传附件，附件上传和存储由 CASE-02 负责，本模块仅维护附件引用列表 |
| CASE-03 案例审核工作流 | 下游数据消费 | 案例提交后进入 CASE-03 管理的审核流程；审核结论（通过/驳回）由 CASE-03 反馈，本模块根据反馈同步更新案例状态 |
| CASE-07 模板提炼生成 | 下游数据消费 | 同一行为类型下已通过审核案例积累到触发条件时，CASE-07 从已通过案例中提炼共性生成标准化模板 |

### 1.11 业务约束清单

1. **角色准入约束**：仅认证"老师"和"专家"角色可执行案例录入操作。"家属"角色可查阅通用案例但无录入权限。"管理员"和"维护人员"角色不参与案例录入。
2. **PII 脱敏硬约束**：所有案例录入时必须完成 PII 脱敏。原始叙事文本中不得包含患者真实姓名、家庭住址、学校名称、具体日期和手机号码。机构合作脱敏案例需签署脱敏使用协议后方可录入。
3. **编辑即重置约束**：任何对案例内容的修改操作（包括 L1 叙事文本和 L2 结构化字段）均触发案例状态重置为"草稿"，已通过的案例编辑后必须重新进入审核流程，不可跳过。
4. **四段式强绑定约束**：四段式输出字段（即时安全干预动作、情绪安抚话术、后续观察指标、就医判断标准）必须全部填写，不可部分留空。这是案例被 RAG 检索消费的质量底线。
5. **免责声明继承约束**：录入的案例与模板在本模块层面不过滤此类标签（由 CASE-03 审核阶段统一附加免责声明），但录入界面应提示撰写者"案例发布后将附带仅供参考免责声明"。
6. **设计边界**：本模块不负责审核流程的状态流转判定（归属 CASE-03），不负责案例文本的切片与向量化入库（归属 CASE-04），不负责案例版本的迭代历史管理（归属 CASE-05），不负责案例的失效标记与强制下架（归属 CASE-06），不负责从多案例提炼标准化模板（归属 CASE-07）。

### 1.12 留给规范阶段的技术决策

| 序号 | 技术决策项 | 意图文档中的约束 | 留给规范阶段的理由 |
|------|-----------|-----------------|-------------------|
| 1 | 具体技术类型定义 | 字段业务含义与约束已定义 | 具体 Pydantic Schema 类型、TypeScript 接口类型、Enum 值定义由 spec-writer 确定 |
| 2 | L1-L2 双层存储的物理模型 | L1 原始叙事层与 L2 结构化卡片层的字段体系和关联关系已定义 | 是否采用单表 JSONB 还是双表外键关联、L2 卡片与 L1 叙事的衍生数量上限由 spec-writer 根据数据库性能确定 |
| 3 | PII 检测策略 | 业务上要求提交前做 PII 检测提示 | 具体检测算法（正则匹配 / NLP 模型）、PII 类型识别范围（5 类基础 PII 是否扩展）、检测为提示还是强制阻断由 spec-writer 确定 |
| 4 | 编辑冲突处理 | 编辑后状态重置为草稿 | 并发编辑冲突的检测机制（乐观锁 / 悲观锁）、编辑锁定时长由 spec-writer 确定 |
| 5 | 附件关联的引用方式 | 案例仅存储附件引用列表，不存储文件本身 | 引用路径格式、附件排序规则、附件删除后的引用清理策略由 spec-writer 确定 |
| 6 | 表单自动保存策略 | 无特定约束 | 是否支持表单草稿自动保存、保存间隔、离线场景处理由 spec-writer 确定 |
| 7 | LLM 自动提取（L1到L2）的触发时机 | L1 原始叙事录入后可由系统辅助提取 L2 结构化字段 | 提取是在提交时同步触发还是异步触发、使用的轻量模型选型、提取结果展示方式由 spec-writer 确定 |
| 8 | 前端表单状态管理 | 表单字段分为必填和选填两类 | 前端状态管理策略（单一 Store / 多 Slice）、表单校验规则的客户端与服务端分工由 spec-writer 确定 |
