### 1.1 项目定位

Token续期模块是「篝火智答」平台用户会话维持的核心保障机制。它在用户登录后 Access Token 过期时介入，通过 Refresh Token 重新签发 Access Token，使用户无需重复登录即可持续使用平台功能。在整体流程中，它是一个**会话维持节点**——输入用户持有的 Refresh Token，输出新签发的 Access Token 和新的 Refresh Token，同时执行轮换策略防止旧 Token 被重放攻击。该模块直接决定了用户会话的连续性和认证安全水平，是连接用户登录（AUTH-02）与权限鉴权（AUTH-04）之间的桥梁。

### 1.2 所属模块与溯源

- **对应总设计章节**：`docs/篝火智答-技术栈设计.md` §5（安全设计 — 认证与鉴权）
- **业务依赖的其他功能模块**：
  - AUTH-02（用户登录）：本模块消费登录时签发的 Refresh Token 进行续期，续期后的新 Token 继承原用户的身份和角色
  - AUTH-04（五级RBAC鉴权）：续期后的新 Access Token 携带用户最新角色信息，供鉴权模块进行权限校验
  - AUTH-06（认证会话管理）：前端的 Token 存储、自动续期触发与本模块的后端续期接口协同工作
  - SEC-04（防刷限流）：Token 续期接口属于敏感操作，需受防刷限流保护

### 1.4 业务目标

1. 用户在 Access Token 过期后，使用有效期内的 Refresh Token 成功获取新的 Access Token 和 Refresh Token，无需重新登录
2. Refresh Token 每次被用于续期后必须即时失效（轮换），防止同一 Refresh Token 被重复使用导致的重放攻击
3. 已加入黑名单或超过有效期的 Token 不可用于续期，用户需重新登录
4. 续期后新签发的 Access Token 必须携带用户最新的角色信息，确保权限判定准确

### 1.5 用户旅程 / 业务流程

1. **用户登录获发双 Token**：用户通过登录模块（AUTH-02）验证身份后，系统同时签发 Access Token（短有效期）和 Refresh Token（长有效期）
2. **用户正常使用**：用户在 Access Token 有效期内正常访问平台各项功能，请求携带 Access Token 通过认证
3. **Access Token 过期触发续期**：用户请求时携带的 Access Token 已过期，前端或客户端自动使用 Refresh Token 发起续期请求
4. **系统校验 Refresh Token**：系统验证 Refresh Token 的签名有效性、是否在有效期内、是否已被轮换失效、是否已被加入黑名单
5. **轮换签发新 Token**：校验全部通过后，系统使当前 Refresh Token 立即失效（标记为已轮换），签发新的 Access Token 和新的 Refresh Token
6. **用户继续使用**：用户使用新的 Access Token 继续访问平台功能，前端自动更新本地存储的 Token 数据
7. **Refresh Token 过期**：若用户连续 7 天未使用平台，Refresh Token 自然过期，续期失败，用户需重新登录

### 1.7 状态的业务定义（如适用）

Token续期涉及 2 个业务状态（针对 Refresh Token）：

- **有效**：Refresh Token 处于有效期内且未被轮换使用或加入黑名单，持有者可凭此凭证发起续期请求
- **失效**：Refresh Token 已被轮换使用、超过 7 天有效期、或被管理员操作加入黑名单，不可用于续期

进入"有效"状态的条件：用户登录成功时首次签发 Refresh Token，或续期成功时系统重新签发新 Refresh Token。
退出"有效"状态的条件：用户发起续期且系统成功轮换（原 Token 进入"失效"）、Token 自然超过 7 天有效期、用户注销账号或管理员因安全原因将 Token 加入黑名单。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：Refresh Token 已过期
- **业务触发条件**：用户持有一个距签发时间已超过 7 天的 Refresh Token 尝试续期
- **业务处理策略**：
  1. 拒绝本次续期请求
  2. 告知用户"登录凭证已过期，请重新登录"
  3. 引导用户进入登录页面重新完成身份验证
- **业务影响**：用户需重新登录后方可继续使用平台功能；用户的会话状态和安全数据不受影响

#### 1.8.2 异常 2：Refresh Token 已被轮换（重放攻击检测）
- **业务触发条件**：用户使用一个已被上一次续期操作轮换失效的 Refresh Token 再次尝试续期
- **业务处理策略**：
  1. 拒绝本次续期请求
  2. 告知用户"登录凭证已失效，请重新登录"
  3. 将此行为记录为可疑安全事件，供后续安全审计
- **业务影响**：用户需重新登录；此机制有效防止 Refresh Token 在网络传输中被截获后的重放攻击，是保障认证安全的核心防线

#### 1.8.3 异常 3：Refresh Token 已被加入黑名单
- **业务触发条件**：用户持有已被加入黑名单的 Refresh Token 尝试续期（如用户已在其他设备注销账号、管理员因安全事件吊销了该 Token）
- **业务处理策略**：
  1. 拒绝本次续期请求
  2. 告知用户"登录凭证已失效，请重新登录"
  3. 不透露 Token 被吊销的具体原因，防止泄露用户安全状态
- **业务影响**：用户需重新登录；已注销或被吊销的用户无法通过续期绕过注销状态

| 依赖方 | 关系类型 | 业务交互说明 |
|--------|---------|-------------|
| 用户登录模块（AUTH-02） | 上游数据来源 | 登录时签发 Refresh Token，本模块基于该 Token 进行续期验证和重新签发 |
| 五级RBAC鉴权模块（AUTH-04） | 下游数据消费 | 续期后新 Access Token 中的用户角色信息供 AUTH-04 在请求处理前执行权限校验 |

| 防刷限流模块（SEC-04） | 受保护服务方 | Token 续期接口可能成为暴力攻击的目标，需受 SEC-04 的用户级限流保护 |

### 1.11 业务约束清单

1. **[轮换机制约束]** 每次使用 Refresh Token 续期成功后，该 Refresh Token 必须即时失效，不可重复使用。这是防止 Token 重放攻击的核心保障，不可关闭或绕过。
2. **[Token 有效期约束]** Access Token 有效期固定为 15 分钟，Refresh Token 有效期固定为 7 天。续期间隔不可超过 Refresh Token 的有效期。
3. **[黑名单机制]** 用户注销、安全事件触发或管理员主动吊销时，相关的 Token 必须即时加入黑名单。已加入黑名单的 Token 不可用于续期，也不可通过其他途径恢复有效性。
4. **[角色信息同步]** 续期后新签发的 Access Token 必须在 payload 中包含用户最新的角色信息。若用户角色在两次续期间发生变更（如家属被升级为老师），新 Token 应反映最新角色。
5. **[最小信息泄露]** 续期失败的响应仅返回用户可理解的业务提示，不透露 Token 结构、签名密钥、黑名单逻辑或安全规则细节。
6. **[设计边界]** 本模块不负责用户身份的首次认证（登录验证、密码校验），该职责归属 AUTH-02 用户登录模块。本模块不负责权限校验，该职责归属 AUTH-04 五级RBAC鉴权模块。本模块不负责前端的 Token 存储和自动续期触发逻辑，该职责归属 AUTH-06 认证会话管理模块。
