### 1.1 项目定位

> 用 2-3 句话简述本模块在整个项目中的业务角色与价值：它解决什么业务问题、处于整体业务流程的哪个环节、对上下游业务的意义。
> 使用业务语言，不涉及具体技术实现。

五级RBAC鉴权模块是篝火智答全平台的权限管控枢纽。它在用户认证成功之后、业务请求处理之前介入，根据用户所属角色（家属、老师、专家、管理员、维护人员）判断其是否有权执行特定操作或访问特定数据。该模块决定了"谁能做什么、谁能看到什么"，是保障平台数据隔离、操作安全和合规运营的核心基础设施——所有业务功能模块（应急咨询、档案管理、案例库、工单系统）均依赖本模块完成权限校验。

- **业务依赖的其他功能模块**：

  - PROF-05 档案隐私控制：依赖本模块的字段级权限控制能力实现档案数据的角色可见性

### 1.4 业务目标

> 用可量化的方式描述本模块需要达成的业务目标。回答"做到什么程度算完成"。

1. 用户发起业务请求时，系统根据用户角色判断其是否有权执行该操作，未经授权的请求被明确拒绝，拒绝率达 100%
2. 当响应数据中包含手机号字段时，仅拥有管理员角色的用户可见，其他角色在响应中自动隐藏该字段
3. 五级角色之间的权限关系符合累加原则：高级角色自动拥有低级角色的全部授权，无需为高级角色重复配置低级权限
4. 新增业务模块接入权限校验时，只需声明所需角色和权限级别，无需修改本模块的核心逻辑
5. 敏感运维操作（案例强制下架、用户角色变更等）仅限于管理员或维护人员角色执行，低层级角色不可越权

### 1.5 用户旅程 / 业务流程

> 描述本模块涉及的主要使用场景和业务流程。让读者理解"谁在什么场景下使用本模块，期望得到什么结果"。

1. **家属发起应急咨询**：家属用户登录后进入应急咨询页面，提交行为描述。系统校验家属角色有权调用咨询接口，请求正常放行。若家属尝试访问案例管理或工单接单等非授权功能，系统拒绝并提示无权限。
2. **老师查看个人档案**：老师用户登录后访问关联家属的档案页面。系统校验老师角色有权查看关联档案，正常返回档案数据；但档案中的手机号字段因老师角色无权限查看而被隐藏。
3. **专家审核案例**：专家用户登录后访问待审核案例列表。系统校验专家角色有权调用审核接口，正常返回案例数据和审核操作界面。家属角色的用户尝试访问同一接口时被拒绝。
4. **管理员管理用户**：管理员用户登录后访问用户管理页面。系统校验管理员角色有权查看全平台用户列表和基本信息，同时可查看用户的手机号等敏感字段。
5. **维护人员系统运维**：维护人员登录后执行系统配置修改、日志查阅等运维操作。系统校验维护人员角色有权执行全部运维操作，但不应涉及用户业务数据的内容查看（遵从最小权限原则）。

### 1.8 异常的业务策略

> 从业务角度描述什么情况下算业务异常，以及业务层面的处理策略。**不得出现技术处理细节**（如重试次数、指数退避、HTTP 状态码等）。

至少 2 种异常场景：

#### 1.8.1 异常 1：用户角色无权限访问目标资源
- **业务触发条件**：用户的角色等级低于目标资源所要求的最低等级，或用户角色不属于目标资源允许的角色列表
- **业务处理策略**：
  1. 拒绝本次请求，不执行目标业务操作
  2. 向用户呈现清晰的拒绝提示，说明无权限的事实，但不透露具体权限规则细节
  3. 建议用户联系管理员获取相应权限
- **业务影响**：该用户无法完成本次业务操作，但系统数据完整性和安全性得到保障

#### 1.8.2 异常 2：用户身份或角色信息缺失
- **业务触发条件**：请求未携带有效的用户身份信息，或已认证的用户信息中不包含角色字段
- **业务处理策略**：
  1. 拒绝本次请求，不执行目标业务操作
  2. 引导用户重新登录以获取完整的身份和角色信息
  3. 如问题持续出现，提示联系技术支持
- **业务影响**：该用户无法使用任何受保护的业务功能，但可访问无需认证的公开页面（如有）

#### 1.8.3 异常 3：角色信息与已有业务规则冲突
- **业务触发条件**：用户的角色已在注册后发生变更（如老师的账号被重新归类为家属），但请求中的角色信息仍为旧角色
- **业务处理策略**：
  1. 拒绝本次请求
  2. 强制用户重新登录以刷新角色信息
  3. 用户重新登录后，使用最新角色执行权限校验
- **业务影响**：用户需重新登录后方可继续操作，已登录期间的操作不会丢失

### 1.11 业务约束清单

> 列出业务方对系统的硬性约束。这些是业务层面的红线，技术实现必须遵守。

1. **[角色累加性约束]** 五级角色必须严格执行累加模型：家属权限 <= 老师权限 <= 专家权限 <= 管理员权限 <= 维护人员权限。不允许出现"高级角色反而无权访问低级角色的某功能"的反直觉情况。
2. **[字段级权限范围约束]** 字段级可见性控制仅应用于手机号字段，不得扩展到其他业务字段。其他字段的可见性通过路由级权限控制（即整接口访问控制）统一管理。
3. **[运维操作隔离约束]** 案例强制下架、用户角色变更、全平台配置修改等敏感运维操作，必须由管理员或维护人员角色执行。家属、老师、专家角色在任何情况下不可执行运维操作。
4. **[权限校验不可绕过约束]** 权限校验必须在后端执行，前端不可单独依赖前端路由或 UI 元素隐藏来保障权限安全。后端是权限校验的最终执行点。
5. **[信息最小化约束]** 拒绝权限请求时，仅告知用户"无权限"的事实，不透露目标资源的角色要求、存在性或详细权限规则，防止权限信息泄露被利用。
6. **[设计边界]** 本模块不负责用户身份的认证（登录验证、密码校验等），该职责归属 AUTH-02 用户登录模块。本模块假定调用方已提供经过认证的合法用户身份和角色信息。
