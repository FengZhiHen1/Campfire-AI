### 1.1 项目定位

健康检查是系统可观测性域的组成部分，位于工程支撑层（L3）。它通过统一的健康检查端点回答"系统当前是否能正常对外提供服务"这一核心运维问题。在整体流程中，它是一个**状态感知节点**——接收外部探针或编排系统的检查请求，返回各基础服务（数据库、缓存、对象存储）的连通性状态，供容器编排模块决定容器健康状态和重启策略，以及告警通知模块在异常时触发通知。在整体架构中，它是连接容器编排层与系统可观测性层的桥梁，确保运维人员可以随时了解系统底层基础设施的运行状况。

### 1.4 业务目标

1. 运维人员通过 HTTP GET 请求 `/health` 端点，在 1 秒内获得所有基础服务（PostgreSQL、Redis、MinIO）的连通性状态
2. 当任一基础服务不可达时，系统能够明确标识出故障组件及其失败原因，而非笼统返回"不健康"
3. 健康状态从健康变为不健康时，系统自动记录一条结构化的状态变更日志，并通过告警模块通知运维人员
4. Docker HEALTHCHECK 探针每 30 秒执行一次健康检查，连续 3 次失败后触发容器自动重启

### 1.5 用户旅程 / 业务流程

1. **运维日常巡检**：运维人员执行 `curl http://api-server:8000/health` 获取系统整体健康状态，返回结果清晰显示 PostgreSQL、Redis、MinIO 三个组件的各自状态（连通/不连通），可在 1 秒内判断系统是否可以对外提供服务
2. **容器编排自动探测**：Docker 引擎根据 HEALTHCHECK 指令，每 30 秒向 `/health` 端点发起一次 HTTP GET 请求，根据响应状态码（200 为健康，非 200 为不健康）和连续失败次数决定容器健康状态
3. **状态变更告警**：当健康检查发现整体状态从"健康"变为"不健康"时，系统自动记录一条结构化日志，并通过告警模块的 Webhook 通道通知运维人员，告知故障组件和失败原因
4. **容器自动恢复**：当健康检查连续 3 次返回不健康时，Docker 引擎根据 `unless-stopped` 重启策略自动停止并重新创建容器，新容器启动后重新执行健康检查验证恢复状态
5. **分布式部署健康聚合**（远期）：在扩展为多节点部署后，统一健康检查端点作为负载均衡器的后端健康探针，自动将不健康节点从负载池中移除

### 1.7 状态的业务定义

健康检查端点自身不维护持久状态（每次请求实时检测全部组件的连通性），但健康检查结果反映系统基础设施的 3 种运行状态：

- **健康**：全部三个基础服务（PostgreSQL、Redis、MinIO）均连通且可正常响应，系统可以对外提供服务。进入条件：所有组件连通性检查通过。退出条件：任一组件连通性检查失败。
- **降级**：至少一个基础服务不连通但非全部不连通，系统核心功能部分受损但未完全瘫痪。进入条件：至少一个组件检查失败且至少一个组件检查通过。退出条件：全部组件恢复连通（进入"健康"）或全部组件均不连通（进入"不健康"）。
- **不健康**：全部三个基础服务均不连通，系统无法对外提供任何服务。进入条件：全部组件连通性检查均失败。退出条件：至少一个组件恢复连通（进入"降级"或"健康"）。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：单组件不可达

- **业务触发条件**：PostgreSQL、Redis 或 MinIO 中的某一个服务因网络闪断、服务重启或配置变更等原因暂时无法连接
- **业务处理策略**：
  1. 其他组件正常返回各自的连通性状态
  2. 该组件的状态标记为"不连通"并记录具体的失败原因（如连接超时、认证失败、服务未启动）
  3. 整体健康状态标记为"不健康"
  4. 状态变更事件记录到结构化日志，并通过告警模块发送通知
- **业务影响**：依赖该组件的业务功能暂时不可用，不依赖该组件的功能正常运行。例如 Redis 故障时，限流和会话管理降级为无状态模式，但数据库查询正常

#### 1.8.2 异常 2：全部组件不可达

- **业务触发条件**：PostgreSQL、Redis 和 MinIO 三个服务全部无法连接（如宿主机网络故障、Docker 守护进程异常或数据层容器集体崩溃）
- **业务处理策略**：
  1. 整体健康状态标记为"不健康"
  2. 每个组件分别返回各自的失败原因，帮助运维人员快速定位问题范围
  3. 立即触发告警通知运维人员
  4. 容器自身的 HEALTHCHECK 连续失败 3 次后，Docker 引擎根据重启策略自动重启容器
- **业务影响**：系统完全不可用，所有依赖数据层的业务请求都将失败。容器可能被 Docker 引擎自动重启，重启后重新执行健康检查验证恢复状态

#### 1.8.3 异常 3：健康检查端点自身不可用

- **业务触发条件**：API 服务进程崩溃、端口被占用或路由配置错误导致 `/health` 端点无法响应请求
- **业务处理策略**：
  1. 健康检查端点无法响应时，Docker HEALTHCHECK 返回不健康状态
  2. Docker 引擎根据连续失败次数和重启策略自动重启 API 服务容器
  3. 外部监控系统（如云厂商健康检查或第三方监控）发现端点不可用时触发运维告警
- **业务影响**：健康检查本身不可用，但其他业务功能可能仍然正常。容器重启期间 API 服务短暂中断

### 1.11 业务约束清单

1. **检查范围约束**：健康检查端点仅验证与基础服务（PostgreSQL、Redis、MinIO）的网络连通性，不验证业务数据完整性、业务逻辑正确性或索引健康度。业务数据的状态检查由各业务模块自行负责。
2. **响应时间红线**：健康检查端点的响应时间不得超过 5 秒（含全部三个组件的连通性验证），避免因健康检查本身超时导致误判。如果某一组件在规定超时时间内未响应，该组件标记为不连通，不应延长整体响应时间。
3. **幂等性要求**：健康检查是只读操作，不得修改任何数据或状态（包括数据库记录、缓存键值、对象存储对象），多次请求返回结果一致。
4. **无鉴权要求**：健康检查端点不对请求来源做身份验证，以便 Docker HEALTHCHECK 和外部监控系统无需携带 Token 即可调用。这是整个项目中唯一不经过认证鉴权的公开端点。
5. **轻量检查原则**：连通性验证使用最简单的操作（数据库执行 SELECT 1、Redis 执行 PING、MinIO 执行 ListBuckets），不执行重量级查询或操作，避免对正常业务造成影响。
6. **设计边界**：本模块不负责业务层面的健康度判断（如某个案例的索引是否完整、某个用户的会话是否正常）；不负责组件故障后的自动恢复（容器重启由 DEPLOY-01 的容器编排策略处理）；不负责告警规则的配置和通知通道的管理（该职责归属告警通知模块 OBS-03）；不负责监控指标的数据采集和可视化（该职责归属指标监控模块 OBS-02）；不负责日志的收集和存储（该职责归属结构化日志模块 OBS-01）。
