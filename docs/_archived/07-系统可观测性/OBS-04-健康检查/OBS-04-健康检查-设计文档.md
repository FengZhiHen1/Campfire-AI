### 1.1 技术实现思路

健康检查模块的核心目标是回答"系统当前能否对外提供服务"——以最轻量的方式验证三个基础服务的连通性，并在每次请求中实时返回结果。设计上遵循两个硬性原则：**检查必须轻量且无副作用**（意图文档 §1.11 约束 5），**异常必须明确到组件级别**（意图文档 §1.4 目标 2）。

**并发检查策略**：三个组件的连通性检查通过 `asyncio.gather()` 并发执行，每个检查包裹在独立的 async 函数中并由 `asyncio.wait_for()` 控制超时。串行执行的总耗时等于三者超时之和（最坏 3+3+5=11 秒），远超意图文档的 5 秒响应红线。并发执行使总耗时约等于最慢组件的检查时间（≤5 秒），恰好落在红线内。这是 FastAPI 原生 async 能力的零成本利用——无需引入任务队列或线程池。

**连接池隔离**：健康检查使用独立的轻量数据库引擎，而非共享业务连接池。主业务连接池的角色是为业务请求提供稳定的连接供应——若每 30 秒一次的探测占用业务连接，在并发高峰时可能加剧连接争抢。独立引擎配置 `pool_size=1`、`max_overflow=0`，连接字符串与主引擎相同但连接池完全隔离。Redis 和 MinIO 同样使用短连接（用完即关），不订阅业务侧的长连接池。这一设计直接响应意图文档 §1.12 第 8 项技术决策项。

**状态变更防抖**：健康检查每次请求实时执行全部检测，但状态变更日志不做周期性重复记录。模块内部维护一个 `_last_overall_status` 变量，仅在实际状态发生转变时（如 "healthy" -> "unhealthy"）才写入结构化日志。这避免了每 30 秒重复写入同状态日志的噪音问题，使运维人员聚焦于真正的变更事件。

**HTTP 状态码策略**：当全部组件连通时返回 200，任一组件不连通时返回 503。这与意图文档 §1.6.2 的二元输出（healthy/unhealthy）完全一致——Docker HEALTHCHECK 将非 2xx 视为不健康并计入失败计数。对于"降级"场景（§1.7 中定义的中间状态），系统通过组件级详情（部分 disconnected、部分 connected）让消费者自行判断影响范围，而非映射到独立的 HTTP 状态码。503 的语义明确传递"当前不应接收流量"的信号，对上游 Nginx 负载均衡的节点摘除同样适用。

**路由设计**：主路径为 `GET /health`（与应用级路由注册一致），同时作为 `/api/v1/health` 的别名同步注册。意图文档明确要求路径为 `/health`，但为此添加版本化的别名路径可确保项目结构设计 §6.1 中预留的 `health.py` 文件位置的约定被遵守。Docker HEALTHCHECK 使用短路径 `/health`（无需配置复杂命令行），外部监控系统可按需使用带版本前缀的路径。

**启动就绪探针**：额外提供 `GET /ready` 端点，仅检查 PostgreSQL 组件（不检查 Redis/MinIO）。在数据库迁移进行期间（容器启动阶段），Redis 和 MinIO 可能尚未就绪——此时 Docker HEALTHCHECK 若因它们不连通而判定不健康，会在 startup 期间错误触发容器重启。`/ready` 将判定条件收窄为仅数据库，确保容器在等待数据库迁移完成期间不被重启。Docker Compose `depends_on` 的 `condition: service_healthy` 使用 `/health`，而容器自身的 HEALTHCHECK 指令在 `start_period` 宽限期内使用更宽松的判定逻辑。

### 1.4 状态机设计（技术实现策略，如适用）

健康检查端点自身不维护持久状态（每次请求实时检测），但模块内部需要追踪两类状态以支持防抖日志和连续失败计数。

**基础设施健康状态（观察态，非持久化）**：

```
健康 ──(任一组件故障)──▶ 降级 ──(全部组件故障)──▶ 不健康
  ▲                        │                        │
  └──(全部组件恢复)────────┘                        │
  └──────────────────(全部组件恢复)──────────────────┘
```

- **健康**：PostgreSQL、Redis、MinIO 全部连通
- **降级**：部分组件不连通但非全部（至少一个连通、至少一个不连通）
- **不健康**：全部三个组件均不连通

技术实现上，这三个状态均为每次请求实时计算出的结果——根据 `asyncio.gather()` 返回的各组件检查结果即时判定，不作为持久状态写入数据库或缓存。`_last_overall_status` 变量仅用内存存储最近一次检查的整体健康状态（healthy/unhealthy/degraded），用于状态变更防抖：若当前状态与上次相同，不写入日志；状态变化时，写入一条包含变更前后状态和组件详情的日志。

**连续失败计数器（内存变量）**：
- `_consecutive_failures: int`，初始值 0
- 每次检查中发现任一组件失败 → +1；全部成功 → 归零
- 计数器仅影响 API 响应的 HTTP 状态码（≥3 时返回 503），不持久化
- Docker 层的 HEALTHCHECK 失败计数（`retries=3`）与本模块的计数器并行运作，形成双重安全网

### 1.6 架构权衡与备选方案

| 决策点 | 最终方案 | 备选方案 | 选择理由 |
|--------|---------|---------|---------|
| 并发执行策略 | `asyncio.gather()` 并发三个组件的检查，每个包装 `asyncio.wait_for()` 超时控制 | (A) 串行检查（依次执行各组件探测）；(B) 短时缓存（5 秒内复用上次结果） | **受意图文档约束**：响应时间红线 ≤5 秒（§1.11 约束 2）。串行最坏 11 秒（3+3+5），超标。`asyncio.gather()` 使总耗时 ≈ 最慢组件（5 秒），恰在红线内。短时缓存虽能加速到接近 0ms，但缓存 5 秒窗口内的故障延迟检测（最坏 5 秒+30 秒探测间隔=35 秒才发现故障），违反意图文档"实时检测"要求（§1.12 第 3 项）。FastAPI 原生 async 下 `asyncio.gather()` 零额外依赖。 |
| 连接池策略 | 健康检查使用独立轻量引擎（`pool_size=1, max_overflow=0`），Redis/MinIO 使用独立短连接 | (A) 共享业务连接池；(B) 仅检查 TCP 端口可达性（不验证认证和查询执行） | **受意图文档约束**：§1.12 第 8 项——"健康检查应避免从业务连接池中借用连接"。方案 A 在并发高峰时健康检查会占用业务连接（pool 中的连接被探测持有），加剧连接争抢。方案 B 太弱——TCP 端口可达但认证失败、数据库不可读时仍会错误报告"健康"。独立引擎使健康检查零影响业务连接可用性，额外一个 PG 连接的代价忽略不计。 |
| HTTP 状态码策略 | 全部健康→200；任一不连通→503 | (A) healthy/degraded→200，仅 unhealthy→503；(B) 始终返回 200，通过 JSON 字段区分 | **受意图文档约束**：§1.5 旅程 2——"Docker 根据响应状态码（200 为健康，非 200 为不健康）"判定容器健康。方案 A 将"降级"返回 200 会导致 Docker 认为容器健康（即使部分功能受损），负载均衡器继续向该节点分发流量——而依赖已故障组件的请求将失败。方案 B 使 Docker HEALTHCHECK 完全失效——容器永远返回 200，引擎无法感知故障。最终方案中，任一组件故障即返回 503——Docker 识别为不健康并积累失败计数，Nginx 从 upstream 摘除节点。运维人员通过响应 JSON 的组件级详情区分"部分故障"与"全部故障"。 |
| 组件检查方法 | PostgreSQL: `SELECT 1`；Redis: `PING`；MinIO: `bucket_exists("campfire")` | (A) PostgreSQL: `SELECT count(*) FROM users`；(B) Redis: `GET health-check-key`；(C) MinIO: 写入+删除临时对象 | **受意图文档约束**：§1.11 约束 5——"连通性验证使用最简单的操作"。`SELECT 1` 无需访问任何业务表、零权限依赖，能验证 TCP 连接+认证+查询三层。`PING` 是 Redis 协议级保活，O(1)。`bucket_exists()` 验证认证+基本 I/O，不产生垃圾数据（方案 C 的临时文件可能因删除失败而残留）。方案 A 的 `count(*)` 在大表上执行全表扫描，严重违反轻量原则。 |
| 状态变更日志防抖 | 内存维护 `_last_overall_status`，仅状态实际变化时写入日志 | (A) 每次检查都写日志；(B) 使用 Redis 分布式锁去重 | **受意图文档约束**：§1.9 AC-07——"健康状态变更记录"。方案 A 会导致每 30 秒一条日志，运维人员被噪音淹没——无法从日志流中区分"一直不健康"和"刚变为不健康"。方案 B 引入 Redis 依赖（若 Redis 恰是故障组件则锁失效）和分布式复杂性——对于单 worker 部署的健康检查去重而言过度复杂。内存变量去重：零外部依赖、极简、满足需求。 |
| 健康检查路径 | 主路径 `GET /health`，别名 `GET /api/v1/health` | (A) 仅 `/api/v1/health`（符合 API 版本化）；(B) 仅 `/health`（简洁但破坏版本化约定） | **受意图文档约束**：路径固定为 `/health`（§1.6.1）。但同时需遵守项目结构设计中预留的 `api/v1/health.py` 路径。双注册方案：Docker HEALTHCHECK 和运维脚本使用简洁的 `/health`，外部监控系统按需使用带版本前缀的 `/api/v1/health`——两者指向同一处理函数，零维护费用差异。 |
| `/ready` 启动就绪探针 | 提供 `GET /ready`，仅检查 PostgreSQL | (A) 依赖 Docker HEALTHCHECK 的 `start_period` 参数抑制启动期探测；(B) 不加 `/ready`，让 Docker 在启动期间也探测 `/health` | Docker Compose `start_period=60s`（DEPLOY-01 HealthCheckProbe 契约）可覆盖 api-server 的启动时间（含 uv sync 和连接池初始化）。但数据库迁移（DEPLOY-04）的执行时长不可预知——若迁移耗时超过 60s，容器在迁移期间被标记 unhealthy 并被重启（重启后又触发迁移，形成无限循环）。`/ready` 提供精确的就绪信号：仅数据库可连通时才返回 200，确保容器在迁移完成前不被重启。这在 DEPLOY-01 落地规范中已通过 `depends_on migration: service_completed_successfully` 保障——但 Docker HEALTHCHECK 仍可能在此之前超时。`/ready` 为这层时序依赖提供冗余安全网。 |

### 1.7 注意事项与禁止行为（设计层面）

1. **[约束：无鉴权公开端点]** 健康检查端点是整个项目中唯一豁免 JWT 认证的端点（意图文档 §1.11 约束 4）。路由注册时必须显式配置为无需认证的路径，确保 Docker HEALTHCHECK 和外部监控系统能不带 Token 直接调用。实现需在 FastAPI 中间件或路由级别将 `/health` 和 `/ready` 加入认证白名单。

2. **[约束：严禁副作用]** 健康检查是纯只读操作，禁止在检查过程中执行任何写操作——不得写入数据库（含临时表）、修改缓存键值、创建或删除对象存储文件。连接资源使用后必须立即释放/归还（短连接用完即关，引擎连接归还池）。

3. **[约束：响应时间红线 5 秒]** 若 `asyncio.gather()` 的整体等待超过 5 秒，必须截断未完成的检查——将超时组件标记为 `disconnected` 并附带 `"timeout"` 错误信息，不等待其完成。不因个别组件的超时延长整体响应时间。

4. **[易错点：连接池隔离的独立性]** 健康检查的独立引擎必须是全新的 `AsyncEngine` 实例，不得通过参数复制主引擎的连接配置。同时确保独立引擎使用后正确释放连接——`max_overflow=0` 且 `pool_size=1` 时，连接数量极有限，泄漏一条即导致后续检查失败。

5. **[易错点：asyncio.gather 的异常不互相干扰]** `asyncio.gather(return_exceptions=True)` 是关键——若任一组件检查抛出异常（如连接超时、DNS 解析失败），该异常不应终止其他组件的并发检查。`return_exceptions=True` 将异常作为结果返回而非向上传播，确保三个组件结果总是完整收集。

6. **[设计边界]** 本模块只负责将基础服务的连通性状态转化为 HTTP 响应和日志记录。以下能力明确不在本模块范围内：(a) 容器重启决策（Docker 引擎负责）；(b) 告警规则的配置和通知通道管理（OBS-03 负责）；(c) Prometheus 指标暴露和可视化（OBS-02 负责）；(d) 业务级健康度检查（如某个案例索引的完整性、某条数据库记录的数值合法性——各业务模块自行负责）；(e) 自动修复故障组件（运维人工或容器编排层负责）。

7. **[禁止行为]** 禁止将业务级健康度嵌入健康检查端点的响应（如"案例库索引健康度 85%"）——健康检查仅验证连通性。禁止在健康检查请求处理中执行重量级查询或 I/O（如全表扫描、大文件的 MD5 校验）。禁止在健康检查中持久化检查结果——每次请求必须实时检测，不依赖缓存或上次结果。
