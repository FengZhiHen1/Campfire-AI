### 1.1 技术实现思路

结构化日志模块的核心设计理念是**日志可用性优先、零侵入调用**。作为 L2 共享能力层组件（项目结构 §6.1），它被平台所有后端模块共同依赖，因此实现上必须满足三个硬性要求：调用方接口极简（一行调用即输出 JSON）、内部异常永不外溢（不阻塞主业务流程）、trace_id 上下文自动传播（调用方无需手动传递）。

**日志输出链路**：调用方调用日志方法 → `contextvars` 自动注入 trace_id → 日志条目组装为 dict → `json.dumps()` 序列化 → stdout 输出 → Docker 日志驱动采集 → 可选汇入 Loki。

选择 `contextvars` 而非线程局部变量的原因：Python asyncio 下同一线程可能同时处理多个协程，`threading.local` 会导致 trace_id 跨协程串扰。`contextvars` 是 Python 3.7+ 标准库特性，asyncio 原生支持，每个协程自动隔离，无需额外依赖。

**审计日志的接口强制**：对 AI 调用、权限拒绝、工单创建三个关键动作，提供独立的 `critical()` 方法，将 `op_type` 设为必填参数。调用方无法通过省略参数绕过审计——编译器/类型检查器即可在代码提交前发现遗漏。这比运行时检查更早暴露问题，也无需在日志输出端做兜底过滤（意图文档 §1.9 AC-03）。

**异常屏障设计**：所有日志写入方法在最外层用 `try/except Exception` 包裹。任何内部异常（JSON 序列化失败、stdout 阻塞、上下文丢失）都不会向上传播到调用方。日志模块的最后逃生通道是 `print()` 到 stderr——即使 stdout 完全不可用，也能以原始文本形式保留一条错误记录。这在 Docker 环境下尤为关键：stderr 和 stdout 由 Docker 日志驱动分别采集，不会互相阻塞。

**输出通道降级**：当 stdout 写入失败或阻塞时，采用环形缓冲区在内存中暂存。缓冲区按等级优先淘汰（保留 ERROR > WARNING > INFO > DEBUG），保证高优先级日志不丢失。输出通道恢复后按时间序刷出，不因网络抖动丢失关键审计记录。

### 1.3 依赖关系概述（技术层面）

| 依赖方 | 关系类型 | 技术交互说明 |
|--------|----------|-------------|
| 平台全部后端模块（`api-server`、`worker`、各 `py-*` 包） | 下游调用方 | 在业务逻辑中通过 `py_logger` 包的方法写入日志；调用方仅需传 level、message、service，trace_id 由 context 自动注入 |
| Docker 日志驱动 | 输出消费方 | 日志以 JSON Line 格式写入 stdout，由 Docker 的 `json-file` 或 `loki` 日志驱动采集 |
| `py-config`（可选依赖） | 配置读取 | 启动时读取各环境的默认日志级别（`LOG_LEVEL` 环境变量）；py-config 不可用时回退至 `INFO` 作为硬编码默认值 |
| OBS-02 指标监控展示 | 数据消费方（间接） | 日志中的 `extra` 结构化字段（如 `duration_ms`）供 Prometheus 指标采集，最终在 Grafana 面板展示；本模块不直接向 Prometheus 暴露指标 |
| OBS-03 告警通知 | 数据消费方（间接） | 日志中 `severity=ERROR` 的记录供 Grafana Alert 规则引擎消费；本模块不直接触发告警 |
| Python 标准库 `json` | 内部依赖 | `json.dumps()` 序列化日志条目为 JSON 格式 |

> **说明**：本模块位于 L2 共享能力层，被 L1 应用层所有后端组件引用，符合项目结构 §5.3 "公共层可被任意层引用"的约束。本模块不依赖任何 L1 组件，保证 L2→L1 单向依赖不被破坏。

### 1.4 状态机设计（技术实现策略，如适用）

本功能点不涉及状态流转，故无需状态机。结构化日志是典型的无状态管道式处理——日志事件从调用到输出的全流程是瞬时的单向管道：调用 → 上下文注入 → 序列化 → stdout。环形缓冲区虽涉及"暂存→恢复→刷出"的状态切换，但这是内部实现细节而非业务状态，不构成需要文档化追踪的状态机。

### 1.5 设计原则兑现清单（技术视角）

| 原则编号 | 原则名称 | 技术响应 |
|----------|----------|----------|
| 1 | 厚 package、薄 app | 本模块以 L2 共享能力层 package 形式实现，封装所有日志逻辑。api-server 和 worker 仅需 `from py_logger import logger` 一行导入即可使用，无需在 app 层重复配置日志格式 |
| 2 | 单向依赖 | 本模块位于 L2，不依赖任何 L1 应用层组件。唯一可选依赖 `py-config` 同为 L2 层 package，不违反层间依赖规则 |
| 3 | 最小化可工作 | 不引入重型日志框架（如 ELK Stack 的客户端 SDK），仅覆盖意图文档明确要求的 stdout JSON 输出 + trace_id 传播 + 审计日志强制，不预留给未来可能需求的功能（如日志加密、自动脱敏） |
| 4 | 可观测与审计 | 技术栈设计 §14 设计原则第 5 条——"关键动作必须落审计日志"。本模块通过 `critical()` 接口强制 `op_type` 必填，在接口层面保障审计不可绕过，而非依赖调用方自觉传参 |

### 1.6 架构权衡与备选方案

| 决策点 | 最终方案 | 备选方案 | 选择理由 |
|--------|---------|---------|---------|
| 日志库选型 | 标准库 `logging` + 自定义 `JSONFormatter` | (A) `structlog`（生态集成佳，但引入外部依赖）；(B) `python-json-logger`（仅 JSON Formatter，定位过窄） | **受意图文档约束**：必须输出合法 JSON 到 stdout。标准库方案零外部依赖，自定义 `JSONFormatter` 完全可控输出格式。`structlog` 的上下文绑定（`bind()`）与代理处理器对 MVP 阶段是过度抽象——本项目的日志上下文通过 `contextvars` 管理，无需 structlog 的 `BoundLogger` 模式。`python-json-logger` 仅提供 Formatter 替换，价值不足以引入额外依赖。<br>**（待确认）** 若后续阶段发现需 OpenTelemetry 集成，structlog 的 `structlog.processors` 管道式处理器是更优选择——届时可重构切换。 |
| trace_id 生成算法 | UUID4（`uuid4().hex` → 32 字符十六进制） | (A) ULID（26 字符 base32，时间有序，不符合 32 字符约束）；(B) Snowflake（需维护 worker_id） | **受意图文档约束**：trace_id 格式为 32 位十六进制字符串。UUID4 的 `.hex` 恰好 32 字符，标准库即用，无分布式协调成本。ULID 时间有序性的优势在 MVP 阶段（单 worker、无分布式查询需求）不构成实际收益。Snowflake 需引入 worker_id 分配机制，与小团队部署模型冲突。<br>**（待确认）** 若未来需求扩展到跨服务日志关联且需时间排序，可迁移至 ULID（放宽长度约束至 26 字符）或 UUID v7（兼容 32 字符 hex 格式，自带时间排序）。 |
| 环形缓冲区大小 | 最大 5000 条，高水位（80%）触发淘汰，低水位（50%）停止淘汰 | 固定条数 / 按内存容量计算 | 按单条日志平均 1KB 估算，5000 条约 5MB 内存占用——对小团队的 2-4GB 云服务器内存完全安全。高水位触发淘汰（保留 ERROR/WARNING），降低到 50% 时停止淘汰，避免频繁触发。<br>**（待确认）** 具体条数阈值需运维确认服务器内存余量后调整。 |
| FastAPI 中间件字段 Schema | 核心字段：method、path、status_code、duration_ms。扩展字段：client_ip、user_id（如有认证）、error_type（如有异常） | (A) 仅核心四字段；(B) 含 request_id、response_size、请求 body 摘要 | 核心四字段满足意图文档 §1.5 的"故障排查"和"性能分析"场景。client_ip 和 user_id 用于审计追溯。request_id 可通过已存在的 trace_id 满足，无需新增字段。response_size 和请求 body 摘要会增加日志体积且引入隐私风险——不符合意图文档 §1.11 约束 4（隐私保护）。<br>**（待确认）** 扩展字段集合需业务确认。 |
| 环境日志级别映射 | 开发=DEBUG，测试=INFO，生产=INFO | (A) 生产=WARNING（保守，可能丢失 INFO 级关键日志）；(B) 生产=DEBUG（过于宽松） | 生产环境设为 INFO 而不是 WARNING 的理由：审计日志（AI 调用、权限拒绝、工单创建）通常使用 INFO 级别——若生产级别为 WARNING，审计日志将全部丢失，违反意图文档 §1.11 约束 2（审计不可绕过）。生产环境支持运行时通过环境变量动态调整级别。<br>**（待确认）** 需与运维团队确认生产环境的默认级别和动态调整机制。 |
| 跨服务 trace_id 传播 | W3C Trace Context（`traceparent` HTTP header） | (A) 自定义 header `X-Trace-ID`（简洁，未来迁移成本高）；(B) Redis 消息体嵌入（侵入性高） | W3C Trace Context 是 OpenTelemetry 标准，格式为 `00-<trace_id>-<span_id>-<flags>`。选择标准方案的理由：(1) 未来若引入 Jaeger/Zipkin 等分布式追踪系统，零改造成本接入；(2) 标准的 `traceparent` 格式同时承载 trace_id 和 span_id，支持跨越多跳传播；(3) 即使当前只使用 trace_id 部分，标准 header 也不增加解析复杂度。api-server 在接收请求时从 `traceparent` header 提取 trace_id（或生成新 ID），在向 worker 投递 Redis 消息时将 trace_id 注入消息头。worker 消费时从消息头恢复 trace_id 到 `contextvars`。<br>**（待确认）** 需确认团队是否考虑未来引入 OpenTelemetry。若不需要，可降级为自定义 header 降低实现复杂度。 |

### 1.7 注意事项与禁止行为（设计层面）

1. **[约束：日志不可阻塞主业务]** 所有日志写入方法的调用方代码不得因日志异常而收到异常。日志模块内部捕获所有异常，最坏情况下回退至 stderr 输出原始文本。实现时需确保 `json.dumps()` 的 `default` handler 和 stdout 写入操作均在外层 try/except 内。
2. **[约束：审计日志不可绕过]** 调用方不得通过传入空字符串或通用值（如 `"other"`）逃避审计日志的 `op_type` 要求。`critical()` 方法在参数校验阶段拒绝空值或非法值，并在日志中强制记录 `op_type` 字段。
3. **[约束：隐私保护]** 日志中禁止包含用户个人身份信息（真实姓名、手机号码、身份证号、家庭住址）。业务模块若需在 `extra` 中传递用户上下文，须使用 `user_id` 等脱敏标识。本模块不负责内容审查——此为调用方责任——但应在设计文档中明确此约束以指导调用方。
4. **[易错点：contextvars 的作用域]** `contextvars` 仅在同一个 asyncio Task 作用域内传播。跨 Task 边界（如 `asyncio.create_task()`）需要手动复制上下文。设计上应明确：trace_id 由 FastAPI 中间件在请求入口设置，中间件之后的整个请求处理链（包括 `await` 调用链）自动继承。但显式创建新 Task 的代码（如 `asyncio.gather()` 内部分支）需手动通过 `contextvars.copy_context()` 传递。
5. **[设计边界]** 本模块只负责日志的生成、格式化与 stdout 输出。以下能力明确不在本模块范围内：(a) 日志采集与聚合存储（Docker 日志驱动 / Loki）；(b) 日志可视化与检索（Grafana / Loki）；(c) 告警规则触发（Grafana Alert）；(d) 日志自动脱敏或加密；(e) 日志的持久化磁盘写入（stdout 即唯一输出通道，不重复落地文件）。
6. **[禁止行为]** 禁止日志模块直接写入文件或数据库——这违反"日志可用性优先"原则（文件 I/O 可能阻塞且增加故障面）。禁止在日志中硬编码服务名称——必须由调用方传入，以保持跨模块通用性。禁止在日志输出阶段使用异步 I/O——日志写入必须是同步操作，避免 asyncio 事件循环中被日志 I/O 抢占。
