## 1.1 项目定位

档案隐私控制是「篝火智答」个性化档案体系的安全守卫模块。它定义并执行个人档案数据在五类平台角色（家属本人、关联老师、关联专家、其他用户、管理员）之间的可见范围与操作权限规则，确保每一条档案数据仅对被授权的用户可见。在整体业务流程中，它是一个**横切安全节点**——所有涉及档案数据访问的功能模块（个人档案管理、事件记录管理、专业评估补充等）都必须经过其权限校验才能执行读写操作。它对业务的价值在于：为孤独症家庭对个人数据的绝对控制权提供制度性保障，这是用户信任产品的基石。

## 1.4 业务目标

1. 个人档案的所有读写操作必须经过权限校验，未授权的访问请求在 API 层即被拒绝，不可出现绕过校验直接访问数据的路径
2. 五类角色（家属本人、关联老师、关联专家、其他用户、管理员）的访问权限必须严格符合访问矩阵定义，无越权操作发生
3. 家属可随时单方面解除老师与其档案的关联，被解除者立即失去所有访问权，无任何延迟窗口或审核环节
4. 管理员仅可查看档案的元数据信息（数据量、存储大小、创建时间等），不可查看档案的具体业务内容
5. 同一家庭的多位家属账号权限一致，均可执行全部家属职能操作
6. 老师被解除关联后，其历史提交的专业评估数据一并隐藏，不再对任何人可见

## 1.5 用户旅程 / 业务流程

1. **家属查看档案**：家属登录系统后，系统自动识别其家属角色及其关联的所有个人档案。家属可查看全部关联档案的完整内容（包含基础档案、事件记录和专业评估），无需额外授权。
2. **老师/专家查看档案**：与家属建立了关联关系的老师或专家可查看对应个人档案的完整内容（与家属可见范围一致），但不可新增事件记录或修改任何数据。老师仅可补充专业评估。
3. **老师解除关联**：家属在档案设置中可单方面解除某位老师的关联关系，无需对方同意。解除操作完成后，该老师立即失去对该档案的全部访问权限。
4. **老师解除关联后的数据处理**：家属解除老师关联后，该老师此前提交的所有专业评估数据一并从当前视图中隐藏，不再对任何用户可见。
5. **管理员运维操作**：管理员在系统后台可查看档案的元数据（存储量、访问频率等统计信息），但无法查看档案内的具体业务内容（个人基础信息、事件记录、专业评估等）。
6. **非关联用户访问禁止**：任何与目标档案无关联关系的用户（含其他家属、未关联的老师/专家）尝试访问档案时，系统拒绝请求且不泄露该档案是否存在的信息。

## 1.8 异常的业务策略

### 1.8.1 异常 1：未授权用户尝试访问档案
- **业务触发条件**：用户尝试访问其无权查看的个人档案（如未关联的老师尝试查看非关联学生的档案、普通用户尝试查看任意档案）
- **业务处理策略**：
  1. 系统静默拒绝访问请求，向用户返回通用性提示（如"数据不存在"），不泄露档案的存在性或权限结构信息
  2. 不允许请求进入后续数据处理流程
  3. （待确认）建议记录本次越权尝试的时间、发起人和目标档案标识，供事后审计
- **业务影响**：本次访问请求被拒绝，不产生任何数据泄漏风险

### 1.8.2 异常 2：关联关系断裂后的访问延续
- **业务触发条件**：老师与家属的关联关系已被解除，但该老师在同一会话中尝试继续访问此前关联的个人档案
- **业务处理策略**：
  1. 系统在每次访问请求到达时实时校验关联关系的有效性，不依赖客户端缓存的关联状态
  2. 关联关系断裂后，该老师的所有后续访问请求均被拒绝
  3. 系统不需要通知已解除关联的老师（关联断裂由家属主动执行，无需对方确认）
- **业务影响**：档案数据的安全隔离得到保障，解除关联立即生效

### 1.8.3 异常 3：多位家属操作冲突
- **业务触发条件**：同一家庭中的两位家属同时对同一档案执行互斥操作（如一位家属解除老师关联的同时，另一位家属正在让该老师补充专业评估）
- **业务处理策略**：
  1. 以系统实际接收请求的时间顺序为准，后到达的请求基于最新的关联关系状态进行裁决
  2. 不会因为存在并发请求而对已经断裂的关联关系临时开放访问权限
  3. 家属之间不需要协调操作时序，系统保证操作的原子性
- **业务影响**：在极端情况下后发起的操作可能因关联关系已变更而失败，但数据一致性得到保证

## 1.11 业务约束清单

1. **[零延迟切断]** 家属解除老师关联后，被解除者必须立即（在下一请求前）失去全部访问权限，不得存在任何延迟窗口或缓冲期
2. **[静默拒绝]** 越权访问时必须向用户返回通用性提示（如"数据不存在"），不得泄露档案是否存在或其权限结构信息
3. **[物理隔离]** 所有个人档案数据必须按个人档案标识进行物理隔离存储，不允许跨档案标识的数据检索
4. **[不入向量库]** 个人档案数据绝对不以任何形式进入向量索引或语义检索库，避免因检索行为导致的交叉污染
5. **[不对外展示]** 个人档案内容（含基础档案、事件记录、专业评估）不向任何非关联用户或公众展示
6. **[管理员边界]** 管理员在任何场景下均不可查看个人档案的具体业务内容，包括但不限于：个人基础信息、事件记录描述、专业评估文本
7. **[角色参照]** 本模块依赖 AUTH-04 定义的五级角色层级（家属 < 老师 < 专家 < 管理员 < 维护人员）进行权限裁决，角色层级本身由 AUTH-04 维护
8. **[多位家属一致性]** 同一家庭下的多位家属账号权限完全一致，不区分主次或操作优先级
9. **[设计边界]** 本模块不负责角色层级的定义和维护（归属 AUTH-04），不负责档案数据的 CRUD 业务逻辑（归属 PROF-01/PROF-03/PROF-04），不负责用户认证与会话管理（归属 AUTH-02/AUTH-06）
