### 1.5 用户旅程 / 业务流程

> 以下流程基于代码中可观察的 API 接口签名和文件注释推断。

1. **用户创建案例**：用户在案例录入界面填写案例信息（标题、叙事文本、来源类型等基本字段；行为类型、年龄段、严重程度、场景、循证标签等详细字段）。表单支持逐个字段填写或批量更新。
2. **自动保存草稿**：用户填写过程中，每次字段变更后启动 30 秒防抖计时器，30 秒无操作自动将当前字段保存到本地存储。

4. **用户查看案例列表**：用户可查看已创建的案例列表，支持按状态筛选和分页浏览（默认每页 15 条）。
5. **用户查看案例详情**：用户可查看单个案例的完整详情。
6. **用户编辑案例**：用户可对已创建的案例进行部分字段更新（乐观锁控制，需传入 updated_at 字段）。
7. **PII 检测**：用户在提交前可对叙事文本执行 PII 检测，确认是否存在个人信息。

### 1.7 状态的业务定义（如适用）

本功能点涉及表单状态和案例提交状态：

**表单业务状态**：
- **空闲**：表单已加载/已重置，用户尚未操作。无未保存的更改。
- **已变更**（dirty）：用户已修改字段值，有未保存的更改。若 30 秒内无新变更，自动保存草稿至本地存储。
- **提交中**：用户已触发提交操作，正在等待后端响应。提交完成后退出此状态。
- **草稿已保存**：表单字段已自动保存到本地存储。保存后回到"空闲"或"已变更"状态（如果保存期间用户继续编辑）。

**[推断]** [待确认] **案例业务状态**（由后端维护，本模块通过 submitCase 接口触发状态转换）：
- **草稿**（draft）：案例初始状态，表示案例尚未提交审核。
- **待审核**（pending_review）：案例已提交但尚未进入审核流程。
- **已通过**（approved）/ **已驳回**（rejected）：案例审核结果，驳回后返回草稿状态。

> 注：案例审核的完整状态机属于 CASE-03 案例审核工作流的范围。本模块仅涉及 draft → pending_review 的提交转换，以及列表查询时的状态筛选。

  3. [确定] Token 续期失败后自动跳转登录页（由 AUTH-06 的 httpClient 拦截器实现）
- **业务影响**：操作未完成，用户可稍后重试

- **业务处理策略**：
  1. [确定] 草稿保存失败时不阻断用户操作，用户可继续编辑
  2. [确定] 草稿加载失败时静默忽略，不影响表单初始化
  3. [推测] [待确认] 不向用户展示错误通知
- **业务影响**：草稿自动保存/恢复功能暂时不可用，用户需手动重新填写

2. **⚠️ 缺少 Hooks 层**：项目结构设计文档要求前端遵循 `views/ → Hooks → logics/` 的三层隔离架构，Hooks 是 `views/` 与 `logics/` 之间的唯一桥梁。当前 `logics/cases/` 目录下缺少 `hooks/` 子目录和对应的 Hook 文件（如 `useCases.ts`），可能导致 views 层直接引用 logics 层的 store 或 service，违反分层约束。

3. **⚠️ 代码路径与项目结构不一致**：实际文件位于 `logics/cases/services/caseApiService.ts` 和 `logics/cases/store/caseFormStore.ts`，而项目结构设计文档中预期的路径为 `logics/cases/services/caseApi.ts` 和 `logics/cases/store/caseStore.ts`。文件命名存在差异。

4. **⚠️ 草稿异常静默处理**：`loadDraft()` 和 `saveDraft()` 中的 try/catch 块均无日志输出——草稿保存/加载失败后静默忽略，用户和管理员无法感知数据丢失风险。

5. **⚠️ 缺少类型定义文件**：项目结构设计文档中预期 `logics/cases/types/index.ts` 存在，但当前未观察到该文件。本模块依赖的外部类型（CaseCreateRequest、CaseResponse 等）来自 `@campfire/ts-shared` 包，该包可能尚未创建或类型未完整迁移。
