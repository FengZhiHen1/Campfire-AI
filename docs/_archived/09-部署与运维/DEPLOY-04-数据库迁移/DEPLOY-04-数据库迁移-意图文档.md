### 1.1 项目定位

数据库迁移模块是篝火智答平台的**基础设施层核心组件**。它负责管理 PostgreSQL 数据库表结构的版本化演变，确保每次代码发布后数据库 Schema 与代码定义保持一致。在整体部署与运维流程中，它是一个**部署前置节点**——任何应用服务的启动都依赖迁移脚本先执行完毕，将数据库调整到目标版本后才允许应用容器对外提供服务。本模块不处理业务数据本身，只管理数据"容器"（表、索引、约束）的结构变更。

### 1.4 业务目标

1. 所有数据库 Schema 变更必须通过可追溯的版本化迁移脚本执行，禁止手动修改数据库结构，实现变更历史 100% 可审计
2. 每次部署前自动将数据库升级到最新版本，确保应用代码与数据库结构同步，消除因版本不一致导致的运行时错误
3. 每份迁移脚本支持正向升级和反向回滚双向操作，在变更出现问题时可在可接受时间内恢复到上一个已知正常版本
4. CI 流水线中自动验证迁移脚本在空数据库上的可执行性，阻断无法执行的迁移脚本进入生产环境
5. 迁移脚本的生成、执行、验证流程零人工干预，完全融入开发与部署流水线

### 1.5 用户旅程 / 业务流程

本模块的直接使用者为**开发人员**和**CI/CD 流水线**，终端用户（家属/老师/管理员）不直接感知数据库迁移过程。

1. **开发阶段：生成迁移脚本**：开发人员修改 ORM 模型定义后，执行生成命令自动创建一份新的迁移脚本。脚本以时间戳命名，确保迁移顺序可追溯。脚本包含升级和回滚两套操作。
2. **开发阶段：本地验证**：开发人员在本地的开发数据库中执行迁移，验证变更是否符合预期。若发现问题，执行回滚操作恢复到变更前状态。
3. **CI 阶段：自动化验证**：代码推送后，CI 流水线在一个空数据库上从头执行全部迁移脚本，验证所有脚本均可正常执行且无语法错误。验证不通过时流水线失败并通知开发人员。
4. **部署阶段：自动升级**：部署流程启动后，在应用容器启动之前，自动执行迁移命令将数据库升级到最新版本。若迁移失败，部署流程中断，应用容器不启动，系统仍运行在上一版本上。

### 1.6 输入/输出业务定义

#### 1.6.1 输入

| 字段名 | 业务含义 | 是否必填 | 业务约束 | 示例值 |
|--------|---------|---------|---------|--------|
| 迁移脚本 | 描述一次数据库结构变更的正向和回滚操作 | 是 | 以时间戳命名；必须包含升级操作和回滚操作两个部分；脚本通过 ORM 模型变更自动生成或手工编写 | `20260526_add_user_nickname.py` |
| 目标数据库连接 | 迁移操作的目标数据库地址与认证信息 | 是 | 由环境配置模块提供；不得硬编码在迁移脚本中 | 由环境变量 DATABASE_URL 注入 |
| 目标版本标识 | 要迁移到的目标版本标记 | 否 | 默认值为最新版本；用于指定回滚到特定历史版本 | `head`（最新）、`abc123`（指定版本哈希） |

#### 1.6.2 输出

| 字段名 | 业务含义 | 业务约束 | 示例值 |
|--------|---------|---------|--------|
| 迁移执行结果 | 迁移操作的成功或失败状态 | 成功时记录已执行的迁移版本号；失败时输出失败原因和失败的脚本文件名 | 成功：`INFO: Running upgrade 20260520_001 → 20260526_002`；失败：`ERROR: Migration 20260526_002 failed: column "nickname" already exists` |
| 当前数据库版本 | 迁移完成后数据库所处的版本号 | 必须与目标版本一致（正向迁移时等于 head，回滚时等于指定版本） | `20260526_002` |
| 变更历史记录 | 本次迁移执行的变更摘要列表 | 逐条列出每项 Schema 变更：新增表、新增列、修改列类型、新增索引、新增约束 | `新增表 users、新增列 users.nickname (VARCHAR 64, nullable)、新增索引 idx_users_email` |

### 1.7 状态的业务定义

数据库迁移涉及 3 个关键的业务节点状态：

- **待迁移**：数据库当前版本低于目标版本，存在未执行的迁移脚本
- **迁移执行中**：系统正在将数据库从一个版本升级到另一个版本，期间数据库表结构可能处于变更过程
- **已就绪**：数据库已处于目标版本，所有已记录的迁移脚本均已成功执行，数据库结构与应用代码一致

进入"迁移执行中"的条件：部署流程检测到存在未执行的迁移脚本，且目标数据库连接可用。
退出"迁移执行中"的条件：全部待执行迁移脚本成功执行完毕（进入"已就绪"），或任一脚本执行失败（回到"待迁移"，需人工介入）。

本模块不管理长时间运行的状态机——迁移是线性串行的版本化过程，单次部署中从"待迁移"到"已就绪"为一次性操作。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：迁移脚本执行失败

- **业务触发条件**：某份迁移脚本在数据库中执行时出错，常见原因包括：目标表已存在、列名冲突、约束冲突、数据类型不兼容
- **业务处理策略**：
  1. 立即中止后续迁移脚本的执行，数据库停留在失败脚本之前的版本
  2. 向部署流程报告失败的脚本文件名、失败原因及当前数据库版本号
  3. 应用容器不启动，保持系统在上一个正常版本运行
  4. 开发人员需排查失败原因，修复迁移脚本后重新触发部署
- **业务影响**：本次部署失败，但生产环境不受影响——应用服务仍运行在上一个正常版本，用户无感知

#### 1.8.2 异常 2：迁移脚本无法回滚

- **业务触发条件**：执行回滚操作时，目标版本不可达（迁移脚本缺少回滚操作或回滚操作与当前数据库状态不兼容）
- **业务处理策略**：
  1. 向部署流程明确报告回滚失败及原因
  2. 阻止继续降级，防止数据库进入不可预测的中间状态
  3. 要求开发人员编写修复迁移脚本，通过正向迁移（而非回滚）将数据库调整到正确状态
- **业务影响**：回滚路径不可用，但不影响正向迁移能力。开发团队需通过新增迁移脚本修正问题，而非强行回退

#### 1.8.3 异常 3：数据库连接不可用

- **业务触发条件**：迁移开始前无法连接目标数据库，或迁移执行过程中数据库连接中断
- **业务处理策略**：
  1. 迁移开始前检测到连接失败：立即报告并中止部署流程
  2. 迁移执行过程中连接中断：尝试恢复连接并确认已执行到哪个版本；若无法恢复，报告失败并中止
  3. 确保未完整执行的迁移不会留下残留变更（通过数据库事务保证原子性）
- **业务影响**：部署流程中止，等待数据库服务恢复后重新触发。已成功执行的迁移版本不受影响

### 1.9 验收标准

| 编号 | 验收条件 | 验收方式 | 通过标准 |
|------|---------|---------|---------|
| AC-01 | 迁移脚本生成 | 开发验证 | 修改 ORM 模型后可通过命令自动生成迁移脚本，脚本包含 upgrade 和 downgrade 两个函数 |
| AC-02 | 正向迁移 | 自动测试 | 在空数据库上执行全部迁移脚本后，数据库结构与应用 ORM 模型定义完全一致 |
| AC-03 | 回滚迁移 | 手动测试 | 执行回滚命令后，数据库成功恢复到上一个版本，被回滚的变更（表/列/索引/约束）均已撤销 |
| AC-04 | CI 空库验证 | CI 自动检测 | 每次代码推送后 CI 在空数据库上执行全部迁移脚本，任何脚本执行失败时流水线标红 |
| AC-05 | 部署前自动执行 | 集成测试 | 部署流程启动后自动执行迁移，迁移成功后才启动应用容器；迁移失败时容器不启动 |
| AC-06 | 禁止手动修改 | 审计检查 | 生产数据库的结构变更记录 100% 可通过 Alembic 迁移历史表追溯，无手动修改痕迹 |
| AC-07 | 变更历史可追溯 | 查询验证 | 数据库内置的迁移历史表完整记录了每次迁移的版本号和执行时间，可追溯到首次迁移 |

### 1.11 业务约束清单

1. **版本化唯一**：每份迁移脚本拥有全局唯一的版本标识（时间戳前缀），不可重复或跳过版本号
2. **双向操作强制**：每份迁移脚本必须同时提供升级操作和回滚操作，仅有升级操作的脚本不得进入版本库
3. **禁止手动改库**：任何对生产数据库结构的直接修改（绕过迁移脚本）均视为违规操作，必须在部署流水线层面阻断
4. **原子执行**：单份迁移脚本的升级操作必须在一次数据库事务中完成——要么全部成功，要么全部回滚，不留中间状态
5. **向后兼容**：迁移脚本不得包含破坏已有数据的操作（如 DROP TABLE 包含生产数据的表），除非同时提供数据迁移逻辑将数据转移到新结构中
6. **设计边界**：本模块不负责数据库查询性能优化（如索引策略）、不负责数据备份与恢复、不负责数据内容的校验——这些职责分别归属于 QUAL 质量保障模块和 SEC 安全模块

### 1.12 留给规范阶段的技术决策

| 序号 | 技术决策项 | 意图文档中的约束 | 留给规范阶段的理由 |
|------|-----------|-----------------|-------------------|
| 1 | Alembic 配置参数 | 迁移脚本需在事务中原子执行 | 具体 Alembic 配置（transaction_per_migration、compare_type 等参数）由 spec-writer 根据项目实际 ORM 模型复杂度确定 |
| 2 | 迁移脚本自动生成策略 | 修改 ORM 模型后可通过命令自动生成迁移脚本 | 自动检测的具体范围（是否包含列重命名检测、是否包含类型变更检测）以及手工补充的边界由 spec-writer 确定 |
| 3 | 执行超时与重试策略 | 迁移失败时部署流程中止 | 具体超时阈值和重试次数由 spec-writer 根据历史迁移脚本的平均执行时间确定 |
| 4 | 迁移锁机制 | 迁移执行期间数据库结构处于变更过程 | 是否需要分布式锁防止并发迁移执行、锁的实现方式由 spec-writer 根据部署架构确定 |
| 5 | 迁移脚本版本管理 | 迁移脚本存放在 packages/py-db/migrations/versions/，以时间戳命名 | 迁移脚本的版本分支策略（多分支并行开发时的迁移合并规则）由 spec-writer 确定 |
| 6 | CI 验证的环境隔离 | CI 中在空数据库上验证迁移脚本可执行性 | CI 中数据库容器的生命周期管理、清理策略由 spec-writer 确定 |
| 7 | 数据迁移策略 | 迁移脚本不得破坏已有数据 | 具体的数据迁移脚本编写规范（如 ALTER 操作的性能考量、大表变更的在线迁移策略）由 spec-writer 结合 PostgreSQL 特性确定 |
| 8 | 错误信息的技术格式 | 迁移失败时报告失败原因 | 具体日志级别、错误码定义、输出格式由 spec-writer 确定 |
