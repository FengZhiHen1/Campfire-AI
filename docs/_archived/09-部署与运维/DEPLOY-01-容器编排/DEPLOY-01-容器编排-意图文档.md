### 1.1 项目定位

容器编排是篝火智答项目的基础设施模块，位于工程支撑层（L3），负责定义全部后端服务的容器化构建方式与运行编排。它通过统一的编排配置，将 FastAPI API 服务、后台 Worker、PostgreSQL（含 pgvector 插件）、Redis 缓存队列、MinIO 对象存储和 Nginx 反向代理共 6 个服务容器组织为可一键启动的完整运行环境。在整体流程中，它是一个**部署执行节点**——输入代码与配置，输出可访问的完整服务集群，供所有上游业务模块运行其业务逻辑。本模块不产生业务数据，但它是所有业务功能在服务器上实际运行的物理基础。

### 1.4 业务目标

1. 执行一条命令即可启动全部 6 个服务容器（FastAPI、Worker、PostgreSQL、Redis、MinIO、Nginx），无需人工逐个启动
2. 当任一业务容器因异常退出时，系统在 30 秒内自动恢复运行，无需人工介入重启
3. 镜像构建采用多阶段构建策略，最终交付的 API 服务镜像体积不超过 500MB，Worker 服务镜像体积不超过 300MB
4. 生产与开发两套运行环境通过不同的编排文件区分（`docker-compose.yml` 用于开发、`docker-compose.prod.yml` 用于生产），切换环境只需切换编排文件，无需修改应用代码
5. 基础设施容器（PostgreSQL、Redis、MinIO）的数据目录通过宿主机绑定挂载实现持久化，容器销毁后数据不丢失

### 1.5 用户旅程 / 业务流程

1. **新开发者首次启动**：开发者从仓库获取代码后，准备 `.env` 文件配置环境变量，执行 `docker compose up -d` 命令，脚本自动构建 API 和 Worker 镜像、拉取 PostgreSQL、Redis、MinIO、Nginx 的基础镜像、按依赖顺序启动全部容器，开发者在一分钟内获得可访问的完整开发环境
2. **代码变更后重新构建**：开发者修改了 API 服务代码后，执行 `docker compose build api-server` 仅重新构建变更的容器镜像，再执行 `docker compose up -d` 更新运行中的容器，不影响其他正在运行的数据库和缓存容器
3. **生产部署**：运维在 CI/CD 流水线完成镜像构建和推送后，登录云服务器，拉取最新镜像，执行 `docker compose -f docker-compose.prod.yml up -d` 部署生产版本，Nginx 容器在启动时自动从 Let's Encrypt 获取或续期 SSL 证书
4. **服务异常自动恢复**：API 服务因某种原因进程崩溃，Docker 引擎检测到容器退出后，根据 `unless-stopped` 重启策略自动拉起新容器替换已退出的容器，应用在新容器中重新初始化连接池后继续对外提供服务
5. **日常运维操作**：运维需要查看各服务运行状态时，执行 `docker compose ps` 查看所有容器健康状态；需要查看日志时，执行 `docker compose logs -f api-server` 实时跟踪指定服务的日志输出；需要升级数据库版本时，修改编排文件中 PostgreSQL 镜像版本标签后执行 `docker compose up -d` 自动执行滚动升级

### 1.7 状态的业务定义

容器编排涉及 4 个业务状态：

- **启动中**：编排正在拉取镜像、创建并启动容器，部分容器尚未就绪。进入条件：执行 `docker compose up -d` 命令后。退出条件：全部容器通过健康检查（进入"运行中"）或任一关键容器启动失败（进入"异常"）。
- **运行中**：全部 6 个服务容器已启动并通过健康检查，系统对外正常提供服务。进入条件：所有容器的 HEALTHCHECK 返回 healthy。退出条件：任一容器健康检查失败或容器异常退出。
- **异常**：部分容器健康检查连续失败或容器异常退出，自动恢复机制已触发但尚未恢复。进入条件：健康检查连续失败达到重试次数（默认 3 次），或容器意外退出。退出条件：自动恢复成功（重新进入"运行中"）或人工干预停止服务（进入"已停止"）。
- **已停止**：全部容器已停止运行，系统不接受任何外部请求。进入条件：执行 `docker compose down` 命令。退出条件：执行 `docker compose up -d` 重新启动。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：单容器崩溃

- **业务触发条件**：运行中的 API 服务、Worker 或 Nginx 容器因进程崩溃、内存溢出或其他原因意外退出
- **业务处理策略**：
  1. Docker 引擎自动检测到容器退出，根据 `unless-stopped` 重启策略在 5 秒内自动拉起新容器
  2. 新容器启动后重新初始化连接池（数据库连接、Redis 连接、MinIO 连接）
  3. 容器恢复后 Nginx 自动将流量路由到新启动的 API 容器
  4. 如果容器连续崩溃 3 次，间隔时间逐步延长（防止无限快速重启导致日志爆炸）
- **业务影响**：服务短暂中断（通常不超过 30 秒），正在处理中的请求可能丢失（非幂等操作），用户需要重新提交请求。PostgreSQL 和 MinIO 的数据不受影响。

#### 1.8.2 异常 2：关键数据容器故障

- **业务触发条件**：PostgreSQL、Redis 或 MinIO 容器因磁盘故障、配置错误或资源耗尽而无法正常提供服务
- **业务处理策略**：
  1. PostgreSQL 故障时，依赖数据库的业务 API 全部返回服务不可用提示，不返回部分数据或错误的缓存数据
  2. Redis 故障时，API 限流和会话管理降级为无状态模式（所有请求暂时不受限流约束，用户会话回退到仅验证 JWT Token 签名）
  3. MinIO 故障时，附件上传和下载功能暂时不可用，其他业务功能正常
  4. 关键数据容器不宜配置自动重启（防止数据损坏），应等待运维人员确认数据完整性后手动恢复
- **业务影响**：部分或全部业务功能不可用，已持久化的数据不受影响。故障期间新产生的数据需在容器恢复后重新提交。

#### 1.8.3 异常 3：宿主机关机或 Docker 守护进程重启

- **业务触发条件**：云服务器因计划内维护或意外断电关机，Docker 守护进程停止运行后又重新启动
- **业务处理策略**：
  1. Docker 守护进程重启后，配置了 `restart: unless-stopped` 策略的容器自动跟随守护进程启动
  2. 各容器按依赖顺序依次启动：PostgreSQL、Redis、MinIO → API 服务、Worker → Nginx
  3. 自动启动后全部容器恢复至关机前的运行状态
  4. 如宿主机磁盘损坏导致数据丢失，需从最近的备份恢复数据（本模块不负责备份恢复，该职责归属数据备份恢复模块 QUAL-05）
- **业务影响**：服务从关机到完全恢复耗时通常不超过 3 分钟（取决于镜像拉取和数据库恢复速度），恢复后服务状态与关机前一致。

### 1.11 业务约束清单

1. **容器重启策略约束**：业务服务容器（API 服务、Worker、Nginx）必须配置 `restart: unless-stopped` 策略，确保异常退出后自动恢复；数据容器（PostgreSQL、Redis、MinIO）不宜配置自动重启，防止数据损坏风险
2. **多阶段构建要求**：所有应用镜像必须使用多阶段构建策略，第一阶段包含完整构建工具链和依赖安装，第二阶段仅含运行所需的最小文件集合，API 服务终态镜像不得超过 500MB
3. **网络隔离约束**：容器之间通过内部桥接网络（`internal network`）通信，禁止将数据容器端口暴露到宿主机外部（宿主机仅暴露 Nginx 的 80/443 端口）
4. **数据持久化红线**：PostgreSQL 数据目录、Redis 持久化文件、MinIO 存储桶必须通过绑定挂载映射到宿主机固定路径，容器销毁后再次挂载同一路径即可恢复数据。生产环境的数据目录必须配置定期快照或备份
5. **环境隔离约束**：开发环境和生产环境使用独立的编排文件，不同的数据持久化路径，不同的端口映射映射配置，避免开发数据污染生产数据库
6. **镜像版本标签规范**：生产环境部署必须使用明确的版本标签（如 `v1.2.3` 或构建日期 `20260526-01`），禁止使用 `latest` 标签指向生产镜像
7. **设计边界**：本模块不负责 SSL 证书的申请和续期（该职责归属反向代理路由模块 DEPLOY-02）；不负责容器级别的资源限制（CPU/内存配额留待规范阶段根据实际负载确定）；不负责镜像的构建时机（由 CI/CD 流水线 DEPLOY-03 触发构建）；不负责数据库的备份恢复操作（该职责归属数据备份恢复模块 QUAL-05）

### 1.12 留给规范阶段的技术决策

| 序号 | 技术决策项 | 意图文档中的约束 | 留给规范阶段的理由 |
|------|-----------|-----------------|-------------------|
| 1 | Docker Compose 文件的具体语法定制 | 必须使用 Docker Compose v2 格式，包含 6 个服务定义 | 具体的 Compose 文件 YAML 结构（services 定义、networks 配置、volumes 声明）由 spec-writer 根据项目根目录的 `docker-compose.yml` 和 `docker-compose.prod.yml` 骨架确定 |
