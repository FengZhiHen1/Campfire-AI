### 1.1 项目定位

环境配置管理是篝火智答项目的基础设施模块，位于整体系统架构的最底层（L1 基础层）。它负责统一管理开发、测试、生产三套运行环境所需的全部配置变量与密钥分发，确保所有业务模块（应急咨询、档案库、案例库、工单系统等）在启动时获得经过校验的、符合环境要求的配置集合。在整体流程中，它是一个**环境供应节点**——输入环境标识，输出经过校验的全局配置对象，无上游依赖，直接支撑 9 个以上的下游模块正常运行。

- **业务依赖的其他功能模块**：本模块为 L1 基础层模块，无上游业务依赖。下游共有 10 个模块依赖本模块提供的配置：容器编排（DEPLOY-01）、反向代理路由（DEPLOY-02）、数据库迁移（DEPLOY-04）、结构化日志（OBS-01）、指标监控（OBS-02）、告警通知（OBS-03）、健康检查（OBS-04）、数据备份恢复（QUAL-05）、RAG 检索与向量化服务（后续模块）

### 1.4 业务目标

1. 系统启动时自动完成全部环境变量的加载与校验，配置加载成功率达到 100%（任何必填配置项缺失或格式错误均阻断启动并给出明确信息）
2. 开发、测试、生产三套环境的配置切换仅需更换 `.env` 文件，无需修改代码
3. 生产环境密钥（数据库密码、API 密钥等）必须通过云厂商 KMS 注入，严禁以明文形式存储于服务器磁盘
4. `.env` 文件加入 `.gitignore` 清单，确保密钥永远不会被提交至版本仓库
5. 提供 `.env.example` 完整模板，新加入的开发者可直接复制并填入自有密钥后启动项目

### 1.5 用户旅程 / 业务流程

1. **新开发者加入项目**：从仓库获取 `.env.example` 模板，复制为 `.env`，填入自己的数据库连接串和 API 密钥，执行启动命令，系统自动校验所有配置项，校验通过后正常启动
2. **运维切换环境**：从开发环境切换到测试环境时，替换 `.env` 文件中对应的连接地址（如数据库 URL 从 `dev-postgres` 切换为 `test-postgres`），重启服务，系统自动加载新配置
3. **生产部署**：运维在云控制台通过 KMS 创建生产密钥，部署脚本在容器启动时将密钥注入环境变量，服务启动时自动从环境变量读取，密钥全程不落盘
4. **配置变更**：当业务需要调整限流阈值或 Token 过期时间时，修改对应环境变量值，重启服务后新配置生效

### 1.7 状态的业务定义

本功能点为纯配置加载模块，在服务启动阶段一次性完成配置加载与校验，不涉及运行时的状态流转，故无需状态机。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：必填配置项缺失

- **业务触发条件**：系统启动时发现任一必填环境变量未被设置或其值为空
- **业务处理策略**：
  1. 不启动服务，输出明确的错误信息，指出缺失的配置项名称
  2. 如有多项缺失，一次性全部列出，避免逐个报错导致反复重启
  3. 引导操作人员检查 `.env` 文件或 KMS 密钥注入状态
- **业务影响**：服务无法启动，正在进行的用户请求不受影响（服务尚未运行），需人工修正配置后重新启动

#### 1.8.2 异常 2：配置项格式错误

- **业务触发条件**：配置项已设置但值格式不符合业务约束（如连接串缺少端口、API 地址不是合法 URL、过期时间不是正整数）
- **业务处理策略**：
  1. 不启动服务，输出明确的错误信息，指出格式错误的配置项名称和期望的格式
  2. 区分"值缺失"与"格式错误"两种场景，提供差异化错误提示
- **业务影响**：服务无法启动，需人工修正配置后重新启动

#### 1.8.3 异常 3：生产环境密钥明文落盘

- **业务触发条件**：在生产环境中检测到密钥类配置项来源于本地 `.env` 文件而非 KMS 注入
- **业务处理策略**：
  1. 记录安全告警日志，标记为严重级别
  2. 允许服务继续启动（避免阻断生产业务）但输出明确的安全风险警告
  3. 建议运维团队将密钥迁移至云厂商 KMS
- **业务影响**：服务正常运行但存在安全隐患，需运维团队尽快完成密钥迁移

### 1.11 业务约束清单

1. **密钥安全红线**：任何包含密钥、密码、Token 的配置项，生产环境必须通过云厂商 KMS 注入，严禁以明文形式存储在服务器磁盘文件中
2. **版本控制隔离**：`.env` 文件必须加入 `.gitignore`，确保包含真实密钥的配置文件永远不会被提交至代码仓库
3. **启动阻断机制**：服务启动阶段，任何必填配置项缺失或格式不合法均须阻断启动，不得以降级模式运行（不符合"零运维负担"原则——带病运行会增加事故排查成本）
4. **环境隔离**：开发、测试、生产三套环境的配置必须彼此独立，切换环境仅需更换配置源（`.env` 文件或 KMS 环境变量），代码零修改
5. **模板完整性**：`.env.example` 模板必须包含全部 18 项配置的字段名、说明注释和示例值，任何新加入的开发者均可直接复制使用
6. **设计边界**：本模块仅负责配置的加载、校验与分发，不负责配置值的业务语义校验（如 "限流阈值是否合理" 由各消费模块自行判断）；不负责密钥的生成（密钥由运维团队在 KMS 控制台创建）；不负责配置变更后的热重载（变更需重启服务生效）
