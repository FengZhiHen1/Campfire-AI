### 1.1 技术实现思路

本模块采用**两阶段串行校验流水线**，而非并行评估。原因是两阶段之间存在严格的优先级关系——高危关键词检测必须在置信度计算之前执行。如果先算出置信度分数再发现关键词命中，已消耗了不必要的 LLM Token 成本和时间。

**第一阶段：关键词安检（零 LLM 调用，纯内存）**

关键词检测是安全红线，必须零延迟执行。采用 AC 自动机（Aho-Corasick）单次扫描方案，而非正则表达式逐词匹配。理由：高危关键词列表预计数百条（含语义变体），正则扫描是 O(kn) 复杂度（k=词数），AC 自动机是 O(n) 且与词数无关。CSLT-01 已实现完整的 AC 自动机引擎和 Redis Pub/Sub 热更新机制，CSLT-05 直接复用同一套词库加载逻辑——不独立维护词库确保三方（CSLT-01/SEC-02/CSLT-05）的关键词集合永远一致，避免新增高危词因遗漏同步而产生安全漏洞。

扫描对象有两层：（1）CSLT-01 传递的 `high_risk_keyword_hit` 标记（标记用户原文是否命中）——若命中则直接走阻断路径；（2）CSLT-05 自身对方案全文的独立扫描——这是安全纵深防御的第二层。LLM 可能在生成过程中引入高危相关内容（如建议药物剂量），即使用户原文不含高危词。两层扫描缺一不可。

**第二阶段：置信度复合评分（LLM + 规则双维度）**

置信度计算采用 `LLM 自评估 50% + 规则校验 50%` 的均衡加权。不以单一维度独立判定——意图文档对此有硬性约束。50/50 的权重选择理由：LLM 自评估覆盖定性维度（来源引用充分性、逻辑连贯性、无来源声明风险）但依赖外部 API，规则校验覆盖定量维度（结构完整性、来源引用覆盖率）且确定性强。均衡权重意味着二者互为制衡，任何一个维度的结果都不能单独决定最终结论。

权重的默认值 50/50 通过 `packages/py-config` 环境变量（`CONFIDENCE_LLM_WEIGHT`、`CONFIDENCE_RULE_WEIGHT`）配置，允许线上 A/B 测试调优而无需重新部署。

**数据流设计**

```
CSLT-03 GenerationResult ─┐
CSLT-01 CrisisJudgmentResult ─┤
CSLT-08 编排上下文 ─────────┘
       │
       ▼
   [前置检查：block_deep_response?]
       ├─ true → 跳过校验，输出 PASS
       │
       ▼
   [关键词检测]
       │
       ├─ high_risk_keyword_hit=true → FORCE_BLOCK → 替换方案+异步工单
       ├─ 方案全文独立扫描命中高危词 → FORCE_BLOCK
       │
       ▼
   [置信度计算]
       ├─ LLM 自评估（非流式，5s 超时）→ 成功：加权计算
       │         └─ 失败/超时：降级纯规则评分，记录 degradation_note
       │
       ├─ 规则校验（结构完整性 × 来源覆盖率）
       │
       └─ 复合评分
              ├─ >= 0.7 → PASS
              └─ < 0.7 → APPEND_WARNING + 异步工单（exponential backoff 3 次）
```

**工单创建的异步策略**

TICK-01 调用使用 FastAPI `BackgroundTasks` 投递到后台执行，主流程不阻塞等待。选择 BackgroundTasks 而非 Redis 队列的理由：CSLT-05 是请求路径上的同步校验环节，引入 Redis 队列会增加一个组件的故障面，且工单创建是低频操作（仅在置信度不足时触发），BackgroundTasks 的轻量级投递足以满足需求。但若未来工单创建频率显著上升（如产品上线后大量低质量生成），可平滑升级到 Redis 队列架构。

重试采用 `tenacity` 库的指数退避（1s/3s/5s，共 3 次），全部失败后标注 `ticket_creation_failed=true` 并记录 ERROR 日志，由 CSLT-08 编排层在前端展示手动联系专家的提示。

**降级策略**

LLM 自评估不可用时，自动降级为纯规则评分模式。降级后的置信度分数标注为「纯规则评分」，与正常复合评分通过 `degradation_note` 字段做区分。选择降级而非阻塞的理由：意图文档明确要求"不得因单点故障而中断整个校验流程"。纯规则评分的准确性虽低于复合评分，但至少能覆盖结构完整性和引用覆盖率两个硬性指标——这两项已经是方案质量的强信号。

校验流程整体超时（> 3s）时，默认按置信度不足处理——追加提示并触发工单。选择「宁可多触发工单也不放行低质量方案」的安全兜底策略，与意图文档 §1.8.2 的业务要求一致。

### 1.4 状态机设计（技术实现策略）

本模块为**无状态校验服务**，不维护跨咨询会话的持久化状态。单次校验的内部处理阶段为运行时内存流转：

```
等待输入 ──input_received──▶ 前置检查
                                 │
                    block_deep_response=true
                                 │
                                 └──▶ 跳过校验，输出 PASS ──▶ 已完成

                    block_deep_response=false
                                 │
                                 ▼
                           关键词检测
                                 │
                    high_risk_keyword_hit=true
                    OR 独立扫描命中
                                 │
                                 └──▶ FORCE_BLOCK，替换方案 ──▶ 已完成

                    两者均未命中
                                 │
                                 ▼
                           置信度计算
                                 │
                   ┌─────────────┼─────────────┐
                   │             │             │
            LLM 自评估成功  LLM 自评估失败  LLM 自评估超时
                   │             │             │
                   └──────┬──────┘             │
                          │                    │
                   复合评分计算           降级纯规则评分
                          │                    │
                   ┌──────┴──────┐             │
                 >= 0.7      < 0.7              │
                   │             │             │
              PASS       APPEND_WARNING    APPEND_WARNING
                          + 异步工单       + 异步工单
                          + 降级标注
```

**技术实现策略**：阶段流转通过 Python 协程的 `async/await` 自然表示，不引入显式状态机框架。理由是流程线性和无持久化需求——五个阶段严格串行，每个阶段只有一个入口和一个出口，用函数式的管线模式（输入 → 变换 → 输出）表达比状态机模式更简洁。若未来需要支持重试或分支（如同时执行 LLM 评估和规则校验的并行化），再引入显式状态机。

**幂等策略**：CSLT-05 是每条咨询的终点校验环节，同一条咨询不会重复请求校验。若编排层的重试机制导致重复输入（相同 request_id），CSLT-05 不检测幂等——信任上游 CSLT-08 的 dedup 逻辑。

### 1.6 架构权衡与备选方案

| 决策点 | 最终方案 | 备选方案 | 选择理由 |
|--------|---------|---------|---------|
| 关键词检测引擎 | AC 自动机单次扫描 | 正则逐词匹配 | 关键词列表达数百条时，AC 自动机 O(n) 优于正则 O(kn)。CSLT-01 已实现完整 AC 引擎，直接复用节省开发成本和维护风险 |
| 关键词词库维护 | 复用 CSLT-01 词库 + Redis Pub/Sub 热更新 | 独立维护词库 | 独立维护会导致三方词库不同步——CSLT-01 新增高危词后若 CSLT-05 未同步，产生遗漏阻断的安全漏洞。共享词库是安全一致性要求，非代码复用偏好 |
| LLM 自评估模式 | 非流式单次调用 + JSON 结构化输出 | 流式调用 + 自由文本 | 非流式适合短评估文本（<200 tokens），延迟更低且响应更稳定。JSON 结构化输出确保下游解析可靠性——自由文本需额外解析层，引入不确定性 |
| 语义连贯性评估 | 四段式段落结构完整性检查 | 基于嵌入向量的语义连贯性评估 | 向量方案需额外模型调用和推理延迟，不符合"3 秒完成"的性能约束。段落结构检查 O(n) 且结果 100% 确定性——CSLT-03 的 System Prompt 已约束四段式输出格式，结构缺失本身就是明确的质量问题信号。此为受意图文档约束的决策 |
| LLM vs 规则加权 | 50/50 均衡 + 环境变量可配 | 70/30 或 30/70 固定权重 | 意图文档要求"不可单一维度独立判定"，均衡权重符合意图。可配置化允许线上调优——在收集足够数据后可根据实际情况（LLM 评估准确率 vs 规则误判率）调整权重 |
| 工单创建异步 | FastAPI BackgroundTasks | Redis 队列投递 | BackgroundTasks 零额外组件依赖，适合低频工单创建（仅低置信度时触发）。若未来工单创建量上升可平滑升级到 Redis 队列。Simple is better |
| 超时降级策略 | 按置信度不足处理（安全兜底） | 直接抛错中止 | 意图文档 §1.8.2 明确要求"不得因超时而阻塞整体应急咨询流程"。宁可多触发工单也不放行低质量方案，安全兜底与业务约束一致。此为受意图文档约束的决策 |
| LLM 自评估重试策略 | 不重试，直接降级纯规则评分 | 重试 1-2 次后降级 | LLM 自评估耗时通常 < 1s，若失败则大概率是持续性故障（API 不可用 / 配额耗尽 / 网络分区）。重试增加延迟且大概率仍然失败，直接降级更快 |

### 1.7 注意事项与禁止行为（设计层面）

1. **[安全底线]** CSLT-05 必须对方案全文执行独立关键词扫描，不能仅依赖 CSLT-01 的 `high_risk_keyword_hit` 标记。CSLT-01 扫描的是用户输入的行为描述原文，CSLT-05 扫描的是 LLM 生成的方案全文——两者内容不同，两道安检独立执行是安全纵深防御的必要设计。

2. **[内容不可变性]** CSLT-05 只可在方案全文末尾追加警示文本或整体替换为安全提示，不得修改 CSLT-03 生成的方案正文的任何原始内容。前端需要能区分 AI 原始建议和系统追加的安全提示。

3. **[契约间隙]** CSLT-01 的 `CrisisJudgmentResult` 当前缺少 `high_risk_keyword_hit` 字段。若 s08 契约协调未补充此字段，降级方案为从 `judgment_sources` 数组反推（检查 `layer_name="RuleEngine"` 且 `trigger_rule_id` 以 `KW_` 开头）。此为技术债务，需跟踪解决。

4. **[禁止行为]** 禁止在置信度计算中只用 LLM 自评估或只用规则校验——必须两者同时参与。禁止在 LLM 自评估不可用时中断流程——必须降级纯规则评分。禁止在高危关键词扫描未完成前开始置信度计算。禁止从 CSLT-05 内部直接访问 CSLT-03 的生成过程——只能消费其输出的 GenerationResult。

5. **[性能边界]** 校验总耗时不得超过 3 秒（P95）。若规则校验 + LLM 自评估在 3 秒内无法完成，默认按置信度不足处理。工单创建为异步操作，不计入这 3 秒。

6. **[免责提示管理]** 追加的免责提示驳回（低置信度追加文案、高危阻断替换文案）通过 `packages/py-config` 环境变量配置，支持无代码修改的产品方文案微调。

7. **[设计边界]** 本模块不负责：
   - 工单的紧急分级（归属 TICK-02）
   - 工单的专家分配与接单（归属 TICK-03）
   - 案例的审核工作流（归属 CASE-03）
   - AI 输出安全护栏的前置注入（归属 SEC-02——SEC-02 负责 Prompt 层注入，CSLT-05 负责生成后校验）
   - 咨询编排的前端 UI 策略（归属 CSLT-08——CSLT-05 只输出数据和判定结论，UI 如何展示由编排层决定）
