## 1.1 项目定位

CSLT-08 咨询编排逻辑是智能应急咨询模块的前端流程编排枢纽。它负责管理家属从打开咨询页面到获得 AI 应急方案的完整交互过程：收集家属的应急输入、确保各处理环节按正确顺序流转、在异常时引导恢复、在必要时衔接人工专家通道。

在整体应急咨询体系中，它是连接家属操作与后端 AI 能力的桥梁——接收家属描述的患者当前行为表现，调度后端完成危机分级判定、相似案例检索、应急方案生成与质量校验，将结果以结构化、渐进式的方式呈现给家属，并在 AI 能力不济时无缝引导至人工兜底通道。

## 1.4 业务目标

1. 家属从进入咨询页面到看到首句应急建议，全流程的环节衔接顺畅、无跳过或卡死
2. 咨询会话的八种业务状态间转换正确率达到 100%，不存在非法跳转
3. SSE 流式文本到达后，首句内容 50 毫秒内可见于家属屏幕
4. 网络短暂中断时，3 次重连内成功续传 SSE 流的比例不低于 90%
5. 置信度不足或高风险场景下，人工专家入口（工单引导）的展示准确率达到 100%，不误触发、不漏触发

## 1.5 用户旅程 / 业务流程

1. **家属进入咨询页面**：家属在小程序中进入应急咨询 Tab 页。此时会话处于"空闲"状态，等待家属操作。
2. **选择行为类型**：家属在行为类型选择界面勾选当前观察到的患者行为类别，如同时勾选"情绪崩溃"和"自伤行为"。支持多选，至少选择一项。
3. **输入行为描述**：家属以文字输入框描述患者当前的详细表现、触发背景和情绪状态。无最小长度限制，鼓励尽可能详细。
4. **提交并等待**：家属点击提交按钮。系统校验输入（至少勾选一项行为类型且描述去除首尾空白后非空），校验通过后进入"提交中"状态等待后端响应。
5. **流式接收应急方案**：后端 SSE 流建立后，系统逐句接收并渲染四段式应急建议——即时安全干预动作、情绪安抚话术、后续观察指标、就医判断标准。家属可以实时阅读每个已完成的句子，同时看到段落标题的逐步展开。
6. **查看完整结果与置信度**：应急方案全部生成完毕后，系统展示本次咨询的 AI 置信度评分和危机等级判定结果。
7. **工单引导（按条件触发）**：若 AI 置信度不足（低于 0.7）或在咨询过程中检测到高风险关键词，页面在消息流底部自动展示一张工单引导卡片——"AI 建议仅供参考，如需专家介入帮助，可为您转接专业老师"。高风险场景下卡片升级为红色警示样式。家属可选择点击"联系专家"跳转工单模块，或忽略卡片继续浏览当前建议。
8. **查看历史**：家属可在咨询历史列表中浏览过往咨询记录，点击某一条进入后呈现只读浏览模式，不重新触发 AI 生成。

## 1.7 状态的业务定义

CSLT-08 管理咨询会话的下列八种业务状态。家属在任何时刻感知到的界面表现，由当前状态决定。

| 状态名 | 业务含义 | 进入条件 | 退出条件 |
|--------|---------|---------|---------|
| 空闲 | 尚未开始咨询，等待家属触发 | 进入咨询页面 | 家属点击"开始咨询"按钮 |
| 选择行为类型 | 家属正在勾选行为类别 | 家属点击"开始咨询" | 家属完成选择并进入描述输入步骤，或放弃并返回空闲 |
| 提交中 | 咨询请求已发送至后端，等待 SSE 流建立 | 家属点击提交且输入校验通过 | SSE 首个文本事件到达（进入流式接收中）或提交失败 |
| 流式接收中 | 正在逐句接收并渲染 AI 生成的应急建议 | SSE 流首个文本事件到达 | 流正常结束（进入已完成）或连接中断且重连耗尽（进入流传输失败） |
| 已完成 | 应急方案已完整生成并展示，家属可自由浏览 | 流式接收正常完成且收到流结束事件 | 家属发起新一轮咨询（进入选择行为类型）或触发工单引导（进入工单引导） |
| 工单引导 | 在已完成的应急方案下方展示人工专家入口卡片 | 置信度不足（低于 0.7）或高风险关键词命中 | 家属点击"联系专家"跳转工单模块，或家属忽略卡片并继续浏览 |
| 提交失败 | 咨询请求未能成功送达后端 | 提交请求阶段发生网络错误或后端不可用 | 家属点击"重试"（返回提交中）或点击"返回"（返回空闲） |
| 流传输失败 | SSE 连接中断且自动重连也已耗尽 | 流式接收中连接中断且 3 次重连均失败 | 家属点击"重新生成"（返回提交中发起新请求）或退出咨询 |

**状态转换的法定路径**（仅列出允许的转换，未列出即视为非法）：

- 空闲 → 选择行为类型
- 选择行为类型 → 空闲、提交中
- 提交中 → 流式接收中、提交失败
- 流式接收中 → 已完成、流传输失败
- 已完成 → 工单引导（条件触发）、选择行为类型
- 工单引导 → 选择行为类型
- 提交失败 → 空闲、提交中
- 流传输失败 → 空闲、提交中

**咨询互斥规则**：处于"提交中"或"流式接收中"状态时，禁止发起新的咨询请求。家属在此时尝试提交将被阻止并获得明确提示。

## 1.8 异常的业务策略

### 1.8.1 异常 1：提交前输入校验不通过

- **业务触发条件**：家属未勾选任何行为类型，或行为描述文本去除首尾空白后为空
- **业务处理策略**：
  1. 提交按钮保持不可点击状态（视觉效果上置灰），防止无效提交
  2. 在按钮附近以文字提示当前缺失的必填项（如"请至少选择一种行为类型"）
  3. 家属补充完整后，按钮自动恢复可点击状态
- **业务影响**：本次提交被阻止，不产生任何业务结果，家属已输入的内容不丢失

### 1.8.2 异常 2：SSE 连接中断

- **业务触发条件**：流式接收过程中与后端的 SSE 连接因网络波动而意外断开
- **业务处理策略**：
  1. 系统自动尝试重新建立 SSE 连接，从已接收内容的末尾位置续传
  2. 最多尝试 3 次自动重连
  3. 若在 3 次内恢复——继续流式接收和渲染，家属对中断无感知
  4. 若 3 次均失败：
     a. 已接收的部分内容保留在对话流中，以"草稿/不完整"状态呈现
     b. 在对话流中插入一条明确提示："生成中断，以下为不完整建议，可能缺失部分段落"
     c. 提供"重新生成"按钮供家属发起新一轮完整咨询
  5. 若中断时尚未收到任何内容——展示"生成失败，请稍后重试"提示并提供重试入口
- **业务影响**：家属体验受损，但已生成的部分内容仍可能有参考价值（如"即时安全干预动作"段落即使在后续段落缺失的情况下对家属仍有帮助）

### 1.8.3 异常 3：重复提交防护

- **业务触发条件**：家属在当前会话处于"提交中"或"流式接收中"时尝试再次点击提交
- **业务处理策略**：
  1. 新提交操作被系统阻止，不发送请求
  2. 向家属展示提示——"当前正在生成建议，请等待完成后再发起新的咨询"
- **业务影响**：防止重复请求导致后端资源浪费、数据重复记录和界面混乱

## 1.9 验收标准

| 编号 | 验收条件 | 验收方式 | 通过标准 |
|------|---------|---------|---------|
| AC-01 | 状态流转正确性 | 自动化测试（遍历全部合法与非法转换） | 八种状态间的法定转换 100% 按 §1.7 定义执行；任何非法转换均被阻止（包括并发操作压力下）；状态变更后界面表现与状态语义一致 |
| AC-02 | 流式渲染响应速度 | 性能测试（时间戳打点测量） | SSE 首个文本增量事件到达至家属屏幕上首句文本可见，间隔不超过 50 毫秒 |
| AC-03 | SSE 断点续传成功率 | 网络模拟测试（注入可控中断） | 网络中断 5 秒内恢复的场景下，3 次重连内成功续传 SSE 流且后续无数据丢失的比例不低于 90% |
| AC-04 | 消息列表容量控制 | 自动化测试（超限压力注入） | 消息数达到 200 条时自动触发裁剪；裁剪最早 50 条后列表条目数不超过 150 条；裁剪操作不影响正在进行的流式渲染 |
| AC-05 | 工单引导触发准确性 | 自动化测试（模拟不同置信度与关键词组合） | 置信度分数低于 0.7 或高风险关键词命中时，工单引导标记 100% 置为展示且风险等级正确；其余情况工单引导标记置为不展示 |
| AC-06 | 并发提交防护 | 自动化测试（快速连续点击提交） | 处于提交中或流式接收中状态时，任何新的提交操作均被阻止且向家属展示明确的不可提交提示 |
| AC-07 | 输入校验完整性 | 测试用例（覆盖全部缺失组合） | 零项行为类型勾选时提交被阻止；行为描述为空或仅含空白字符时提交被阻止；提示文案准确指明当前缺失的必填项名称 |

| AUTH-06 认证会话管理 | 横向依赖 | 依赖前端认证状态（登录状态、Token 有效性和自动续期机制），确保所有后端 API 调用携带有效认证凭证 |

## 1.11 业务约束清单

1. **咨询互斥**：同一家属在同一时刻最多存在一个活跃咨询会话。处于"提交中"或"流式接收中"状态时，禁止发起新的咨询请求。
2. **消息列表容量上限**：单次会话的消息列表最多保留 200 条消息。达到上限时自动裁剪时间最早的前 50 条消息，裁剪后列表条目数不超过 150 条。
3. **历史只读**：家属从历史列表点击进入往期咨询详情时，为只读浏览模式——不重新建立 SSE 连接、不触发后端 API 调用、不允许修改或重新生成。
4. **重试不覆盖历史**：家属点击"重新生成"时，新生成的应急方案作为新的消息追加到当前列表中，不覆盖之前已生成的方案内容或更早的历史消息。
5. **用户输入不丢失**：在提交失败、流传输失败等任何异常场景下，家属已勾选的行为类型和已输入的描述文本均被保留，不因异常而清空。
6. **设计边界**：CSLT-08 不负责后端 RAG 检索、LLM 生成、危机判定和置信度校验的具体业务逻辑——这些职责分别归属 CSLT-02、CSLT-03、CSLT-01 和 CSLT-05。CSLT-08 仅负责前端的流程编排和状态管理，通过调用后端 API 间接协调上述模块的协作。
