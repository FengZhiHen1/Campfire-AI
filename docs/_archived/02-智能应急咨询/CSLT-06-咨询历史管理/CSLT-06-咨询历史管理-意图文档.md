### 1.1 项目定位

咨询历史管理是篝火智答应急咨询业务流程中的记录存档节点。每次应急咨询完成后，它将咨询的完整上下文——用户的行为描述、检索到的参考案例、AI 生成的四段式应急方案、置信度评估分数和来源引用——作为一条历史记录归档存储。家属可随时回溯过往咨询记录，查看每一次的应急方案内容和参考来源。

在整体业务流程中，咨询历史管理处于应急咨询的最末端：位于方案生成与置信度校验之后，作为咨询环节的收尾动作。它向上游模块提供历史数据的查询接口，支撑工单上下文继承、用户反馈收集与服务质量分析。

### 1.4 业务目标

1. 每次应急咨询完成后，系统的完整上下文数据必须被完整归档存储，核心字段不丢失
2. 家属可随时查看自己的咨询历史列表，列表按时间倒序排列且每页最多显示 20 条记录
3. 家属可点击任意一条历史记录查看本次咨询的完整详情，包括原始输入、参考案例、生成方案和置信度评估
4. 单次咨询详情查询的响应时间不超过 200 毫秒，确保回溯体验流畅
5. 咨询历史记录完整保留，不因系统升级或数据迁移而丢失

### 1.5 用户旅程 / 业务流程

本模块为数据归档与查询模块，以下从两个主要使用场景描述其业务流程：

**场景一：咨询完成后的自动归档**

1. **咨询结束触发**：家属在应急咨询界面完成一次完整的咨询流程——系统完成危机分级判定、案例检索、方案生成和置信度校验后，前端编排层（CSLT-08）将本次咨询的完整上下文数据传递给本模块
2. **数据组装与归档**：本模块将接收到的用户行为描述、检索案例标识列表、生成的四段式应急方案全文、AI 置信度评估分数、来源引用清单和 Token 消耗数据，组装为一条完整的咨询历史记录并归档存储
3. **归档确认**：存储完成后，本模块向前端编排层返回归档确认，允许前端进入后续流程（如弹出用户反馈问卷）

**场景二：用户回溯历史咨询**

1. **打开历史列表**：家属在应用内进入「咨询历史」页面，系统展示该用户所有历史咨询的摘要列表，每页最多 20 条，按时间从最新到最早排列
2. **浏览列表**：每条列表摘要展示咨询时间、行为描述开头文字（前 50 字以内）、危机等级标签和方案是否有帮助的反馈标记
3. **查看详情**：家属点击任意一条记录，系统加载该次咨询的完整详情——包括完整的行为描述文本、四段式应急方案全文（即时安全干预动作、情绪安抚话术、后续观察指标、就医判断标准）、参考案例来源清单、AI 置信度评估结果和免责声明
4. **返回列表**：家属可从详情页返回历史列表，继续浏览其他记录

### 1.7 状态的业务定义

本功能点不涉及状态流转，故无需状态机。

咨询历史记录一旦归档存储即为只读，不支持用户自行修改或删除。每条记录的业务生命周期为：归档存储后可供查询，直到系统按数据保留策略进行周期性归档清理（具体保留周期属于运维层面的决策，不在本模块业务范围内定义）。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：用户无咨询历史记录

- **业务触发条件**：用户首次使用应急咨询功能，尚未产生任何咨询记录；或者系统中该用户的历史记录已被周期性清理
- **业务处理策略**：
  1. 历史列表接口返回空列表，同时返回总记录数为零
  2. 前端应展示引导性空状态页面，提示用户「暂无咨询历史，开始一次应急咨询即可在此查看记录」
  3. 不返回错误提示，空结果是正常的业务状态而非系统异常
- **业务影响**：新用户无法看到过往咨询参考，但不影响正常使用应急咨询功能

#### 1.8.2 异常 2：请求查看的咨询记录不存在或不属于当前用户

- **业务触发条件**：家属请求查看的咨询记录标识在系统中不存在（如已被清理），或该记录属于其他用户（如家属试图查看不属于自己账号下的记录）
- **业务处理策略**：
  1. 明确告知用户「该咨询记录不存在或无权查看」
  2. 引导用户返回历史列表页面重新选择
  3. 对外统一返回相同的模糊提示，不区分「记录不存在」和「无权查看」两种情况，以保护其他用户的隐私
  4. 在系统内部日志中记录实际的拒绝原因，供运维排查
- **业务影响**：用户的回溯请求被拒绝，不影响系统其他功能

#### 1.8.3 异常 3：归档写入时上游数据不完整

- **业务触发条件**：CSLT-08 编排层在咨询流程异常中断（如大模型生成超时但已产出部分内容）后仍触发了归档请求，导致传入数据中某些必填字段缺失或不满足业务约束
- **业务处理策略**：
  1. 若必填字段（行为描述文本、生成方案全文、来源引用清单、置信度评估分数、免责声明、危机等级）中任意一项缺失，拒绝写入并向前端编排层返回明确的字段缺失提示
  2. 若生成方案全文仅包含部分段落（如大模型超时时只产出了前两个段落），仍允许归档，但在记录中额外标注「部分生成」，供前端展示时向用户说明
  3. 前端编排层收到拒绝提示后，可选择补全数据后重新提交归档，或放弃本次归档
- **业务影响**：少量异常咨询可能无法归档，但不影响其他正常咨询的历史记录完整性

### 1.11 业务约束清单

1. **【数据完整性约束】** 每次应急咨询正常完成后，其完整上下文数据必须被归档存储，核心字段（行为描述、生成方案、来源引用、置信度分数、免责声明、危机等级）不得因系统异常而丢失。
2. **【隐私约束】** 家属仅能查看本人账号下的咨询历史记录。任何跨用户的咨询数据访问必须被拒绝，且对外统一返回模糊提示（不区分「不存在」与「无权查看」），以保护用户隐私。
3. **【只读约束】** 咨询历史记录一旦归档存储即为只读，不支持用户自行修改或删除。用户不能编辑历史方案内容、不能删除某条咨询记录。如需修正归档数据，仅限系统管理员通过运维通道操作。
4. **【性能约束】** 单次咨询详情查询的响应时间不得超过 200 毫秒；历史列表查询的响应时间不得超过 500 毫秒。确保用户在浏览历史记录时体验流畅，不出现可感知的等待。
5. **【数据保留约束】** 咨询历史记录必须可长期追溯。具体的数据保留周期（如保留 180 天、1 年或永久保留）属于运维和合规层面的决策，由 QUAL-05（数据备份恢复）模块的数据保留策略统一管理，本模块不单独定义清理规则。
6. **【设计边界】** 本模块不负责应急方案的生成（归属 CSLT-03）、不负责咨询流程编排（归属 CSLT-08）、不负责用户反馈的发起逻辑（归属 QUAL-03）、不负责 Token 成本的聚合统计（归属 QUAL-04）、不负责工单内容的生成逻辑（归属 TICK-01）。本模块聚焦于咨询数据的归档存储与查询回溯这一核心职责。
