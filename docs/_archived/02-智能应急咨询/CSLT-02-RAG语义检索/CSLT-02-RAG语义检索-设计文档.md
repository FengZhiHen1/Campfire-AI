### 1.1 技术实现思路

**顶层设计**：CSLT-02 作为智能应急咨询链路中的检索增强节点，采用"标签精确过滤 + 向量语义排序"的混合检索策略。核心目标是保证检索精度（过滤条件 100% 命中）的同时控制响应延迟（500ms 内返回结果）。

**为什么选择混合检索而非纯向量检索**：纯向量检索虽然能捕捉语义相关性，但无法保证 100% 满足档案标签约束（如"行为类型=情绪崩溃，年龄段=学龄儿童"）。项目中档案标签是结构化维度，与向量相似度是正交的筛选维度——前者保证"对的人"，后者保证"对的内容"。因此采用两阶段策略：先用 SQL WHERE 精确过滤候选集，再用向量距离对候选集排序。pgvector 的 SQL + 向量混合查询天然支持此模式，无需引入独立的向量数据库，运维复杂度可控。

**两阶段检索管道**：
1. **标签过滤阶段（SQL WHERE）**：将上游 PROF-02 传入的标签过滤条件（年龄段、行为类型、情绪等级等）转换为 pgvector 的 metadata JSONB 字段过滤条件。此阶段直接利用 PostgreSQL B-tree 索引，在百万级案例切片中快速收敛候选集。
2. **向量排序阶段（ORDER BY embedding distance）**：将用户行为描述通过 text-embedding-v4 编码为 1024 维向量，在筛选后的候选集上执行 HNSW 索引近似最近邻搜索，按余弦距离从小到大排序。

**降级策略的技术实现**：三层降级放宽是在标签过滤阶段的回退机制。当精确过滤结果不足 Top-K 时，逐层放宽 WHERE 条件而非一次性放宽全部——每层放宽后重新计数，达到目标即停止。这种"最小放宽原则"在召回率和精度之间做最优权衡：先用最小的条件妥协换取足够的候选量，而非一次放开所有约束导致精度崩溃。

**超时保护**：500ms 硬超时通过 asyncio.wait_for 实现。超时发生时，返回已检索到的结果并标记"是否完整检索=false"。选择部分返回而非抛错的原因与意图文档中的业务要求一致——下游 CSLT-03 在收到部分结果时仍可基于已有案例生成方案，只是可参考的背景减少；若直接抛错，用户将完全无法获得任何帮助。

**排序公式设计思路**：综合排序分数的三因子（语义相似度 + 时效衰减 + 循证加权）中，语义相似度是主导因子（权重最高），因为检索的首要目标是"内容匹配"。时效衰减采用阶梯函数（0-1 年权重 1.0 / 1-3 年权重 0.7 / 3 年以上权重 0.5），而非连续衰减函数——阶梯函数的计算成本极低（查表即可），且意图文档已明确三档衰减系数，无需对衰减粒度过度投入。循证等级加权采用固定权重（NCAEP=1.0 / 机构经验=0.8 / 个案观察=0.6），以专业证据强度做适度提权，但不喧宾夺主——当两条切片语义相似度相近时，循证等级更高的排在前面，这符合"优先参考最新专业意见"的业务目标。

**失效案例排除机制**：被 CASE-06 标记为失效（过时/有误/争议/强制下架）的案例切片，在 SQL WHERE 阶段通过 `metadata->>'status' NOT IN (...)` 排除。此操作在数据库层面完成，利用过滤索引确保排除逻辑在候选集收敛之前就生效，而非在应用层遍历结果集时再剔除——后者可能在候选集很大的情况下浪费向量排序的计算资源。

**缓存决策**：不对检索结果做缓存。原因有三：(1) 应急咨询的用户行为描述高度个性化（每次问题文本不同），缓存命中率极低；(2) 案例库是动态变化的（新案例入库、旧案例淘汰），缓存同步复杂度高；(3) 500ms 时延目标在 pgvector HNSW 方案下可通过纯计算达成，无需缓存提速。如果未来发现热点查询重复率较高（如常见行为类型的固定查询），再考虑引入 Redis 缓存层，但当前阶段保持简洁。

### 1.4 状态机设计

本功能点不涉及状态流转，故无需状态机。每次检索为独立的同步请求-响应操作。检索过程中的内部状态（检索中 → 完成/部分完成）通过回应字段隐式传递，不持久化。下游通过 `is_complete` 布尔值判断是否需要降级处理。

### 1.5 设计原则兑现清单（技术视角）

| 原则编号 | 原则名称 | 技术响应 |
|----------|----------|----------|
| 单一职责 | 模块只做检索 | 本模块仅负责"接收查询文本 + 标签条件，返回排序后的案例切片列表"。不负责标签提取（PROF-02）、不负责向量索引维护（CASE-04）、不负责方案生成（CSLT-03）。检索策略变更不影响上下游的接口契约 |
| 失败透明 | 异常信息充分暴露 | 检索超时返回部分结果并标记 is_complete=false；案例库为空返回空列表并标记 reason="case_library_empty"；嵌入 API 调用失败向上游抛出明确异常，由统一异常处理中间件转换为标准错误响应 |
| 性能优先 | 500ms 硬约束 | HNSW 索引提供亚毫秒级近似最近邻查询；SQL WHERE 过滤利用 JSONB GIN 索引加速标签筛选；asyncio 并发处理嵌入向量编码和数据库查询；超时机制确保不阻塞上游请求链路 |
| 可观测性 | 检索过程可追溯 | 每次检索记录结构化日志，包含 trace_id、查询文本长度、标签条件摘要、候选集大小、最终返回数量、检索耗时、是否触发降级、是否触发超时。关键指标（检索耗时 P50/P95/P99、降级触发率、超时触发率）通过 /metrics 暴露 |
| 安全红线 | 失效案例零泄露 | 被 CASE-06 标记为失效的案例切片在 SQL 层面通过 WHERE 条件杜绝。此逻辑不可通过应用层后置过滤实现——应用层后置过滤存在"先查询后过滤"的时间窗口，在极端高并发场景下可能存在脏读。数据库 WHERE 条件保证原子性 |
| 隐私保护 | PII 不进入检索 | 嵌入 API 调用的输入文本在上游 SEC-03 已脱敏。本模块不做额外隐私处理，但在日志中不记录完整的用户查询文本（仅记录长度和指纹），避免日志泄露 PII |

### 1.6 架构权衡与备选方案

| 决策点 | 最终方案 | 备选方案 | 选择理由 |
|--------|---------|---------|---------|
| 向量存储 | pgvector（PostgreSQL 插件） | 独立向量库（Milvus/Qdrant） | 与 PostgreSQL 统一数据层，减少运维组件；初期 <10 万切片规模下 HNSW 完全满足性能要求；SQL + 向量混合查询天然支持标签过滤式检索。受项目 ADR-003 约束 |
| 嵌入模型 | text-embedding-v4（阿里 DashScope） | text-embedding-v3 / bge-large-zh | ADR 已选定 text-embedding-v4 为项目标准嵌入模型；1024 维在精度和性能间取得平衡（维度更高精度提升但性能下降）。受技术栈设计约束 |
| 索引类型 | HNSW（m=16, ef_construction=200） | IVFFlat | HNSW 在内存占用和查询速度间取得更优平衡；10 万级数据规模下查询延迟显著低于 IVFFlat；参数 m=16/ef_construction=200 是 pgvector 官方推荐的高召回率配置 |
| 超时策略 | 返回部分结果 + 标记未完成 | 直接抛错 / 同步等待 | 与意图文档中"超时仍返回已有结果供下游使用"的业务要求一致。受意图文档约束 |
| 降级策略 | 逐层放宽（情绪等级→行为类型→移除全部标签） | 一次性放宽全部条件 / 逐步降低相似度阈值 | 逐层最小放宽原则在召回率和精度间最优权衡；意图文档已明确三层放宽顺序。受意图文档约束 |
| 时效衰减 | 阶梯函数（3 档权重） | 连续指数衰减 / 多段线性衰减 | 意图文档已明确三档衰减系数（1.0/0.7/0.5），阶梯函数计算成本为零，且与业务要求精确对应。受意图文档约束 |
| 缓存 | 不做检索结果缓存 | Redis 缓存热点查询 | 个性化查询命中率极低；缓存同步复杂度高（案例库动态变化）；500ms 目标可在无缓存下达标。未来如确需缓存，再引入 Redis 层 |
| 循证加权 | 固定权重映射（NCAEP=1.0, 机构=0.8, 个案=0.6） | 动态加权 / 调参 | 三档固定权重足以区分不同证据强度；权重值选择保守（最低仍有 0.6，不给任何等级过度惩罚），保留未来通过实验调优的空间 |
| 综合排序公式 | composite = similarity × 0.5 + time_decay × 0.25 + evidence_weight × 0.25 | 各因子权重不同 / 乘法模型 | 相似度为主导因子（0.5），因为内容匹配是检索第一目标；时效和循证各 0.25 作为调节项，在相似度接近时打破平局。权重比可通过环境变量调优。实际上线前需线下验证排序效果 |
| 异常处理 | 空库返回空列表 + 状态标记（不抛异常） | 空库返回 404 / 空列表 | 意图文档明确要求空库时下游使用通用策略生成方案，不应阻断流程。返回空列表 + reason 标记让下游有自主决策空间。受意图文档约束 |

### 1.7 注意事项与禁止行为（设计层面）

1. **标签过滤条件必须在 SQL 层执行，不可在应用层后置过滤**：SQL WHERE 利用索引快速收敛候选集，应用层后置过滤会导致向量排序在过大候选集上执行，严重浪费计算资源。更重要的是，应用层过滤存在"先查询后过滤"的并发窗口——在高并发场景下，从查询到过滤之间的时间窗口内，如果有新的失效标记写入，后置过滤可能遗漏。

2. **embedding 向量编码和数据库查询应并发执行**：在获取查询向量后，数据库查询和后续的缓存查询可以并发。但 embedding 编码本身依赖外部 API 调用，其延迟（通常 20-80ms）在检索总耗时中占比不可忽略。如果 embedding API 不可用，应直接抛出异常而非降级为纯标签检索——因为语义匹配是本模块的核心价值。

3. **降级放宽操作必须按序逐层执行**：不可跳过某层或同时放宽多层。每层放宽后执行独立查询（含 COUNT），达到目标数量即停止。此顺序在意图文档中已明确（情绪等级→行为类型→移除标签），不可自行调整。

4. **用户查询文本不得在日志中完整记录**：仅记录文本长度和 SHA256 指纹用于问题排查。查询文本可能包含敏感的行为描述信息，完整记录违反隐私保护原则。

5. **pgvector HNSW 索引的参数（m、ef_construction）不可在生产环境随意修改**：修改 m 和 ef_construction 需要重建索引（REINDEX），在数据量大的情况下耗时较长，会导致检索服务中断。如需调整，应在运维窗口内操作，并通过 A/B 测试验证效果。

6. **禁止绕过 CASE-06 的失效标记直接查询 case_chunks**：所有检索查询必须在 WHERE 条件中包含失效状态过滤。即使 CASE-06 的 metadata 写入存在短暂延迟，也宁可多漏（过滤条件稍宽后应用层校验）不能少漏。但最终上线前应确保 CASE-06 的写入为同步或近实时（延迟 <100ms）。

7. **禁止在检索模块中直接操作案例元数据表**：CSLT-02 只读 case_chunks 表（视图），不执行任何 INSERT/UPDATE/DELETE。案例的创建、审核、淘汰均由对应模块负责。此隔离保证各模块的数据主权清晰，避免跨模块事务。
