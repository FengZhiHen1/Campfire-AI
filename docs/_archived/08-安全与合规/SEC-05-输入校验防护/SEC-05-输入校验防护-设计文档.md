### 1.1 技术实现思路

输入校验防护模块的核心设计理念是**声明式零侵入校验**——校验规则以 Pydantic v2 BaseModel 声明式定义，通过 FastAPI 依赖注入系统在路由执行前自动生效，开发者无需在每个路由中手动编写校验逻辑，也不会遗漏新增路由的校验注册。

**数据流向**：API 请求进入后，依次经过认证中间件（JWT 校验）→ FastAPI 路由匹配 → Pydantic Schema 校验（类型/必填/范围/格式）→ 业务逻辑执行前二次安全检测（SQL 注入特征、XSS 载荷）→ 参数化查询执行 → 内容安全清洗（html.escape）→ 响应返回。若任一步骤失败，请求在当前位置被拦截并返回字段级错误说明。

**三条职责的技术实现策略**：

1. **请求数据校验**：采用 FastAPI 的 `Depends()` 声明式注入，而非独立的 HTTP 中间件。具体而言，每个 API 路由的参数类型注解为 Pydantic Schema 类的 `Annotated[T, Depends()]` 形式，FastAPI 在路由执行前自动调用 Pydantic 的校验引擎并对不合法请求返回 422 响应。选择路由依赖注入而非独立中间件的原因：(a) FastAPI 原生范式，无需额外中间件逻辑；(b) 校验规则绑定在路由参数类型上，逻辑原子性高，不会因中间件全局注册导致误拦截；(c) 与项目结构设计 §5.2（L1-sub 接口层：请求校验、路由分发、响应封装）的职责描述一致。

2. **数据库安全**：所有数据访问通过 `packages/py-db/repositories/` 中的 Repository 类执行，使用 SQLAlchemy 2.0 ORM 的参数化查询。Repository 模式作为数据访问的唯一入口，从框架层面确保用户输入值通过 DBAPI 的 `bindparam()` 参数绑定传递，而非字符串拼接。设计上禁止在 Service 层出现任何 `session.execute()` 调用（即使使用 `text()` 配合参数绑定也禁止），确保所有数据库操作的参数化路径可追踪、可审计。

3. **内容安全**：用户提交的文本内容在返回给前端之前，通过 Python 标准库 `html.escape()` 进行 OWASP 必修字符转义（`&` → `&amp;`、`<` → `&lt;`、`>` → `&gt;`、`"` → `&quot;`、`'` → `&#x27;`）。转义在后端 API 响应构造阶段执行，而非依赖前端 Taro/React 的 JSX 自动转义机制。此决策的理由：(a) 后端转义提供统一的 XSS 防护基线，不受前端框架差异影响（Taro 小程序 `text` 组件与 Web 标准 HTML 渲染的行为差异存在不确定性）；(b) 安全防护下沉到服务端符合纵深防御原则，即使前端转义机制被绕过或被错误配置，后端仍提供最后一道防线。

**无状态设计**：输入校验是同步的无状态操作。每次请求独立完成校验决策（放行/拒绝），不依赖历史请求的上下文，不维护会话级校验状态。这一设计与意图文档 §1.7 的声明完全一致，同时天然支持水平扩展——任意 API 实例均可独立处理校验请求，无需共享状态。

**8 项业务矛盾的处理方式**：技术预研报告（s06）将意图文档 §1.12 的 8 项"留给规范阶段的技术决策"标记为业务矛盾，并通过建议问法给出了推荐方向。用户已确认采纳报告的完整分析和建议。本设计文档基于报告推荐方向做出以下设计决策（各决策的具体理由见 §1.6 架构权衡表）：

| # | 矛盾点 | 设计决策 | 报告依据 |
|---|--------|---------|---------|
| 1 | 校验规则载体与定义方式 | 独立 Schema 文件（路径 A）—— `packages/py-schemas/` 中为各业务模块定义 `*-validation.py` Schema 文件 | §5 决策 1 分析（利于跨模块复用和集中管理） |
| 2 | 参数化查询严格度 | 严格模式——所有 DB 操作必须通过 Repository 类，Service 层禁止任何 `session.execute()` 调用 | §5 决策 2 分析（严格模式约束力强，防止滥用） |
| 3 | XSS 转义执行位置 | 后端转义——在 API 响应构造阶段通过 `html.escape()` 转义后返回前端 | §6 风险评估（规避 Taro 小程序 text 组件与 Web 标准差异的 XSS 盲区） |
| 4 | 文件类型白名单 | 采纳推荐列表：PDF/JPEG/PNG/WebP/MP4/DOC/DOCX | §5 决策 4 推荐候选 |
| 5 | 文件大小上限 | 通用上限 10MB，各业务场景可覆写（如头像 2MB） | §5 决策 5 分析（层级化策略） |
| 6 | 安全审计日志存储 | 结构化日志 + 日志驱动分流（路径 A）——通过 `py-logger` 独立 Logger 实例写入 stdout，由日志驱动按 180 天策略分流 | §6 风险评估（避免审计表成为性能瓶颈） |
| 7 | 错误响应协议格式 | 自定义 Exception Handler，覆盖 FastAPI 默认 422 格式，输出 `{errors: [{field, reason, constraint}]}` 格式 | §5 决策 7 分析（对齐意图文档 §1.6.2 错误明细结构） |
| 8 | 认证模块集成方式 | 路由依赖注入（路径 B）——无独立校验中间件，校验通过 Pydantic Schema 的 `Depends()` 注入在认证之后、路由处理之前自动执行 | §6 风险评估（与项目结构 §5.2 一致，轻量且自然融入 FastAPI 范式） |

### 1.3 依赖关系概述（技术层面）

| 依赖方 | 关系类型 | 技术交互说明 |
|--------|----------|-------------|
| FastAPI 依赖注入系统 | 框架依赖 | 通过 `Depends()` 将 Pydantic Schema 校验注入路由参数链，FastAPI 自动处理校验失败时的 422 响应生成 |
| Pydantic v2 | 框架依赖 | `BaseModel` 子类定义各接口的请求/响应 Schema，`Field()` 声明逐字段类型、范围、格式约束 |
| SQLAlchemy 2.0 Async | 框架依赖 | `AsyncSession` + ORM 模型的参数化查询，通过 Repository 模式统一调用，禁止裸 `session.execute()` |
| Python `html` 标准库 | 库依赖 | `html.escape(s, quote=True)` 执行 OWASP 五字符转义（`& < > " '`），零外部依赖 |
| `packages/py-logger` | 调用依赖 | 通过独立 Logger 实例（`logger_name="security_audit"`）写入结构化 JSON 日志，包含 `trace_id`、`event_type`、`detection_detail` 字段 |
| AUTH-04（五级 RBAC 鉴权） | 上游数据来源 | 认证中间件将解析后的 JWT payload（含 `user_id`、`roles`）注入 `request.state`；校验模块通过 `Depends(get_current_user)` 获取认证身份，校验逻辑在认证之后执行 |
| 所有 API 业务模块 | 下游服务消费 | 各 API 模块的路由通过 Pydantic Schema `Depends()` 隐式调用校验逻辑。校验通过后，路由处理函数收到的参数已是经过验证的类型安全数据 |
| CASE-02（案例附件上传） | 下游服务消费 | 文件上传请求通过 SEC-05 的独立文件校验函数验证文件类型白名单（MIME + 扩展名双重验证）和大小上限，校验通过后才将 `UploadFile` 对象传递给 CASE-02 的存储逻辑 |

### 1.4 状态机设计

本功能点不涉及状态流转，故无需状态机。输入校验是同步的无状态操作——输入数据进入，校验后放行或拒绝，不维护业务状态。此设计确保了模块的天然水平扩展能力。

### 1.5 设计原则兑现清单（技术视角）

| 原则 | 来源 | 技术响应 |
|------|------|----------|
| 厚 package、薄 app | 项目结构 §三 | 校验逻辑的核心——Schema 定义（`packages/py-schemas/`）、安全转义函数（`packages/py-utils/`）、文件校验服务——均放置在 `packages/` 共享层。`apps/api-server/app/` 中仅保留路由参数的类型注解和依赖注入声明，不包含校验具体实现 |
| 单向依赖（应用层 → 共享层） | 项目结构 §三 | SEC-05 作为 L1 基础层，本身无上游业务模块依赖。校验 Schema 定义在 `packages/py-schemas/`（共享层），路由层（`apps/api-server/app/api/`）向下依赖 Schema 模块，方向符合单向依赖约束 |
| 安全性优先 | 技术栈 §5 | 三条核心职责——请求校验、SQL 注入防护、XSS 防护——均采用白名单式安全策略：校验规则只放行符合约束的数据；数据库操作只接受参数化查询路径；文本输出只允许经过实体转义的内容 |
| 零运维负担 | 技术栈 §1.3 | 依赖全部来自标准库或已在技术栈中确定的组件（Pydantic、SQLAlchemy、py-logger），零新增外部依赖。`html.escape()` 使用标准库，无额外维护成本 |
| 可观测性 | 技术栈 §6.3 | 校验拦截、安全特征检测、文件校验失败三类安全事件通过 `py-logger` 写入结构化 JSON 日志，携带 `trace_id` 实现全链路追踪。审计日志与业务日志共享同一基础设施，按 `logger_name` 分流 |

### 1.6 架构权衡与备选方案

| 决策点 | 最终方案 | 备选方案 | 选择理由 |
|--------|---------|---------|---------|
| 校验实现方式 | FastAPI 路由依赖注入（`Depends()` + Pydantic Schema） | 独立 HTTP 中间件（Starlette `BaseHTTPMiddleware`） | 路由依赖注入是 FastAPI 原生范式，校验规则绑定在路由参数类型上逻辑原子性高；独立中间件会导致校验逻辑重复注册、错误响应格式需要额外定制。与项目结构 §5.2（接口层 = 请求校验 + 路由分发）的定位一致 |
| Schema 组织方式 | `packages/py-schemas/` 独立文件（路径 A） | 路由模块内联 `Annotated` 类型（路径 B） | 独立文件利于跨模块复用（如 CASE-02 的文件校验 Schema 可被 SEC-05 引用）、集中管理校验规则版本、支持单测覆盖 |
| 参数化查询严格度 | 严格模式——Repository 独占 DB 访问 | 宽松模式——Service 层允许 `text()` + `bindparams()` | 严格模式从架构层面杜绝字符串拼接的可能性，所有 DB 操作的参数化路径可追踪、可审计；宽松模式依赖开发者自觉，在团队规模增长时约束力下降 |
| XSS 转义位置 | 后端执行 `html.escape()` | 前端依赖 Taro/React JSX 自动转义 | 后端转义提供统一的 XSS 防护基线，不受前端框架差异影响。风险评估指出 Taro 小程序 `text` 组件的渲染行为与 Web 标准存在差异，前端方案存在 XSS 防护盲区风险 |
| 安全审计日志存储 | `py-logger` 独立 Logger + 日志驱动分流（路径 A） | PostgreSQL 独立审计表（路径 B） | 审计日志写入数据库会增加写入压力（每次安全事件触发一次 INSERT），在平台请求量增长后可能成为性能瓶颈；日志驱动分流方案运维简单、不影响数据库性能、通过 loki 或其他日志平台仍可查询 |
| 错误响应格式 | 自定义 Exception Handler 覆盖 FastAPI 默认 422 格式 | 保留 FastAPI 默认格式，前端自行转换 | 意图文档 §1.6.2 要求字段级错误说明格式为 `{field, reason, constraint}`，与 FastAPI 默认的 `{detail: [{loc, msg, type}]}` 不兼容。自定义 Handler 将 Pydantic `ValidationError` 映射为目标格式，前端无需做格式转换，接口契约统一 |
| 认证集成方式 | 路由依赖注入（路径 B）——`Depends(get_current_user)` + Pydantic Schema `Depends()` 串行 | 独立校验中间件注册（路径 A） | 路径 B 更轻量、自然融入 FastAPI 范式；校验不在中间件层意味着不会因中间件全局注册导致非 API 路由（如 health check）被误拦截；与项目结构 §5.2 接口层职责描述一致 |
| 文件校验架构 | 独立校验函数，在路由处理器中作为业务前置步骤调用 | Pydantic `field_validator` 内联文件校验 | `UploadFile` 对象的元数据（文件名、MIME 类型、文件大小）无法通过 Pydantic Schema 的 `field_validator` 高效访问；独立校验函数可以同时执行 MIME 类型检测（通过 `python-magic` 或文件头字节检测）和大小校验，提供更强的安全保障 |

> 标注"受意图文档约束"的决策点：错误响应格式（意图文档 §1.6.2 明确要求字段级错误明细）、参数化查询方式（意图文档业务约束 3 明确禁止 SQL 拼接）。

### 1.7 注意事项与禁止行为（设计层面）

1. **[设计边界]** 本模块不负责：
   - 用户身份认证（由 AUTH 系列模块负责）
   - 访问权限鉴权（由 AUTH-04 五级 RBAC 负责）
   - 敏感数据的加密存储（由 SEC-01 传输存储安全负责）
   - PII 数据的检测与脱敏（由 SEC-03 负责）
   - API 限流（由 RateLimit 中间件负责）

2. **[校验顺序硬约束]** 校验必须严格在认证之后、业务逻辑之前执行。这是一条架构级硬约束——校验依赖 `Depends(get_current_user)` 提供的认证上下文进行日志关联，但不得在同一层与认证逻辑耦合（两者通过 Depends 链串行，但属于不同的依赖注入实例）。

3. **[禁止绕过校验]** 任何新增 API 路由必须包含对应的 Pydantic Schema 校验注解。设计上通过 Repository 接口约定和代码审查流程保证，而非依赖开发者自觉。落地规范中将定义模板化的路由注册模式，降低遗漏风险。

4. **[禁止直接拼接 SQL]** `apps/api-server/` 下的所有模块禁止出现 `session.execute(text(...))`、`session.execute("SELECT ...")` 或任何形式的字符串拼接 SQL。所有数据库操作必须通过 `packages/py-db/repositories/` 中的 Repository 类访问。此约束为架构级硬性规定，代码审查时必须强制执行。

5. **[错误响应不得泄露内部细节]** 安全检测（SQL 注入特征、XSS 载荷）拦截请求时，返回给客户端的错误信息必须是通用的"数据包含不安全内容"，不得透露检测到何种特征、使用了何种检测规则。详细的检测信息仅写入安全审计日志（`logger_name="security_audit"`）。

6. **[安全日志保留期]** 安全审计日志的保留期不得低于 180 天（意图文档业务约束 6）。日志驱动（如 Loki）的保留策略需要明确配置此期限。

7. **[文件校验双重验证]** 文件类型校验不得仅依赖文件扩展名或 HTTP `Content-Type` 头（两者均可伪造）。应结合文件头魔数（magic bytes）检测实现双重验证——扩展名 + 文件真实类型同时匹配白名单方可通过。

8. **[禁止校验逻辑与业务逻辑耦合]** 校验 Schema 的 `Field()` 约束仅声明数据规则（类型、范围、格式），不得嵌入业务判断逻辑或调用外部服务。校验层保持纯函数特征，确保单元测试可独立覆盖。
