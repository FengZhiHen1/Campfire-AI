### 1.1 项目定位

防刷限流模块是篝火智答平台的**流量控制闸门**，位于所有 API 请求的最前端、身份认证之前执行。它通过限制单个用户和单个 IP 地址的请求频率，防止恶意攻击、爬虫滥用和意外流量峰值导致的服务过载，确保平台资源被所有用户公平使用。

本模块不参与任何业务逻辑处理，也不感知请求的业务内容。它在每次 API 调用到达服务器时执行一次快速检查——如果请求频率在正常范围内则放行；如果超出限制则告知客户端等待。作为平台安全纵深防御体系的第一道关卡，它为后续所有业务模块提供稳定的运行环境。

### 1.4 业务目标

1. 每个已登录用户在 1 分钟窗口内的 API 调用次数不超过 30 次，超出后拒绝并提示用户等待
2. 每个未登录 IP 地址在 1 分钟窗口内的 API 调用次数不超过 100 次，超出后拒绝并提示用户等待
3. 用户级限流和 IP 级限流独立计数、独立执行——同一用户的两次请求，即使 IP 不同，用户级计数器也正常累加
4. 限流触发后用户在合理等待时间后自动恢复访问能力，不永久封禁
5. 限流服务自身发生故障时不影响主业务流程——即使限流功能临时失效，平台业务仍可继续运行

### 1.5 用户旅程 / 业务流程

本模块为基础设施级防护机制，用户不会直接感知其存在，但在以下场景中间接体验其效果：

1. **正常请求处理**：用户通过小程序发起任意 API 请求（如查看科普文章、发起应急咨询）→ 请求到达服务器时先经过限流检查 → 检查通过后继续处理 → 用户无感知地获得正常服务
2. **用户级超限拦截**：同一用户短时间内快速重复操作（如频繁刷新、反复提交表单）→ 限流检查发现该用户 1 分钟内请求超过 30 次 → 请求被拦截 → 用户收到"请求过于频繁，请在 60 秒后重试"的提示 → 用户等待后自动恢复使用
3. **IP 级超限拦截**：未登录用户通过同一 IP 地址高频访问（如爬虫抓取、恶意攻击）→ 限流检查发现该 IP 1 分钟内请求超过 100 次 → 请求被拦截 → 该 IP 地址的请求被暂时限制 → 等待窗口滑动后自动恢复
4. **限流服务降级**：Redis 缓存服务发生故障 → 限流检查无法正常执行 → 系统自动放行所有请求（不再执行限流检查）→ 运维人员在后台收到告警通知 → 修复 Redis 后限流功能自动恢复

| 字段名 | 业务含义 | 业务约束 | 示例值 |
|--------|---------|---------|--------|

### 1.7 状态的业务定义

本功能点不涉及状态流转，故无需状态机。限流检查是同步的无状态操作——请求到达，检查计数，放行或拒绝，不维护请求间的状态依赖。

### 1.8 异常的业务策略

#### 1.8.1 异常 1：用户级请求频率超出限制

- **业务触发条件**：同一已登录用户（按用户身份标识区分）在 1 分钟滑动窗口内的 API 请求总次数超过 30 次
- **业务处理策略**：
  1. 拒绝本次请求，不进入任何业务处理逻辑
  2. 告知用户"请求过于频繁，请在 60 秒后重试"
  3. 不计入永久封禁或导致账号异常，滑动窗口自动恢复后第 31 次请求可正常通过
  4. 同一用户在 5 分钟内累计触发限流 3 次时，在后台记录为潜在异常行为（用于后续安全分析，不触发额外处理动作）
- **业务影响**：用户需等待最多 60 秒后重新发起请求；已登录的其他用户不受影响；用户账号不受任何惩罚性限制

#### 1.8.2 异常 2：IP 级请求频率超出限制

- **业务触发条件**：同一 IP 地址（未登录场景或已登录场景的补充检查）在 1 分钟滑动窗口内的 API 请求总次数超过 100 次
- **业务处理策略**：
  1. 拒绝本次请求，不进入任何业务处理逻辑
  2. 告知用户"请求过于频繁，请在 60 秒后重试"
  3. 窗口滑动自动恢复后该 IP 的正常请求可正常通过
  4. 用户级限流和 IP 级限流独立判断，其中之一超限即触发拒绝
- **业务影响**：同一 IP 地址下的所有用户（如公司出口 IP 共用场景）均暂时受限；同一 IP 下的不同用户可通过登录行为获得用户级独立配额（已登录用户同时受用户级和 IP 级限制，但用户级配额独立核算）

#### 1.8.3 异常 3：限流服务不可用

- **业务触发条件**：Redis 服务发生故障（连接超时、连接拒绝、服务宕机），限流检查无法正常执行
- **业务处理策略**：
  1. 临时放行所有请求，不执行任何限流检查
  2. 在后台立即触发告警通知运维人员
  3. 限流功能在 Redis 恢复后自动恢复正常，无需人工介入
  4. 此期间不记录限流相关的审计事件（因为限流服务未运行）
- **业务影响**：限流保护暂时失效，平台在短时间内缺少请求频率控制；由于这是临时降级策略，平台需要尽快恢复 Redis 服务的可用性以重新获得保护

### 1.9 验收标准

| 编号 | 验收条件 | 验收方式 | 通过标准 |
|------|---------|---------|---------|
| AC-01 | 用户级限流生效 | 压力测试 | 同一用户在第 30 次请求通过，第 31 次请求被拒绝并返回"请求过于频繁"提示，响应中包含等待时长 |
| AC-02 | 用户级限流自动恢复 | 压力测试 | 超限用户等待窗口滑动到期后，再次发起请求正常通过，不再返回限流提示 |
| AC-03 | IP 级限流生效 | 压力测试 | 同一 IP 在第 100 次请求通过，第 101 次请求被拒绝并返回限流提示（无用户身份标识的场景） |
| AC-04 | IP 级限流自动恢复 | 压力测试 | 超限 IP 等待窗口滑动到期后，再次发起请求正常通过 |
| AC-05 | 用户级与 IP 级独立限流 | 集成测试 | 同一用户从两个不同 IP 发起请求，两个 IP 的计数器各自独立，用户级计数器在所有 IP 上累加；用户级和 IP 级任一项超限均触发拒绝 |
| AC-06 | 限流服务降级可用 | 故障注入 | Redis 服务不可用时，所有请求正常通过（不抛异常、不阻塞），且后台有告警记录 |
| AC-07 | 健康检查端点豁免 | 接口测试 | 对 `/health` 端点的高频请求不被限流拦截，其他 API 端点正常受限流保护 |
| AC-08 | 限流拒绝的响应格式 | 接口测试 | 限流拒绝返回的响应中包含明确的业务提示文字和等待时长（秒数），不包含限流阈值、算法细节等内部信息 |

### 1.11 业务约束清单

1. **【请求频率公平使用】** 每个已登录用户在 1 分钟内最多调用 30 次 API，每个未登录 IP 在 1 分钟内最多调用 100 次。用户级与 IP 级限流独立核算，二者满足其一即触发拒绝。
2. **【限流不可绕过】** 所有 API 端点必须经过限流检查，不存在绕过限流的请求通道。仅 `/health` 健康检查端点可通过白名单豁免限流。
3. **【自动恢复原则】** 限流触发后不永久封禁用户或 IP，限流窗口滑动重置后自动恢复访问。禁止引入用户黑名单、IP 黑名单等永久封禁机制（此类功能不属于本模块职责范围）。
4. **【降级可用原则】** 限流服务自身故障（Redis 不可用）时自动降级放行所有请求，不得因限流故障导致全服务不可用。降级的同时触发后台告警。
5. **【信息隐藏原则】** 限流被拒绝时返回的提示不得包含限流阈值的具体数值、算法实现细节、Redis 服务信息等内部细节。用户仅需知道"请求过于频繁"和等待时长。
6. **【异常行为标记】** 同一用户 5 分钟内累计触发限流 3 次时，应被标记为潜在异常行为并记录，但不触发额外处理动作（如冻结账号、人工审核）。异常行为数据留作安全分析使用。
7. **【设计边界】** 本模块不负责身份认证（由 AUTH 系列模块负责）；不负责传输加密（由 SEC-01 负责）；不负责请求内容的 Schema 校验（由 SEC-05 负责）；不负责 IP 黑名单或用户黑名单管理。本模块仅执行频率维度的限流检查。

| 1 | 限流算法的精确实现 | 使用 Redis 滑动窗口，用户级 30 次/分钟，IP 级 100 次/分钟，窗口 60 秒 | 滑动窗口的具体粒度（秒级 key 分片 vs 分钟级单 key + 计数器）、计数聚合方式（LUA 脚本 vs 应用层聚合）由 spec-writer 根据性能基准确定 |

| 3 | Redis 故障告警的阈值与通道 | Redis 不可用时触发告警通知运维人员 | 告警的具体发送通道（钉钉/企业微信 Webhook）、重复告警的间隔频率、CRITICAL 日志的结构化字段定义由 spec-writer 根据运维体系确定 |

| 5 | 限流配置的运行时可调性 | 用户级 30 次/分钟、IP 级 100 次/分钟、窗口 60 秒 | 限流阈值是重启生效（环境变量/配置文件）还是支持运行时动态调整（配置中心/Redis 热更新），由 spec-writer 根据运维能力确定 |
| 6 | 限流拒绝响应的 HTTP 状态码 | 超限时返回状态码 429，包含提示文字和等待时长 | 429 响应体的精确 JSON 结构、是否包含 request_id/trace_id 等追踪字段、错误码编号体系由 spec-writer 确定 |
| 7 | 用户级限流与 IP 级限流的执行顺序 | 两者独立核算，任一项超限即拒绝 | 执行顺序策略（先检查用户级再检查 IP 级、或两级全量检查后聚合结果）由 spec-writer 根据性能优化需求确定 |
| 8 | 限流 key 的内存管理策略 | 每个 key 设置独立 TTL，防止 Redis 内存泄漏 | key 的精确 TTL 值（窗口+缓冲的具体秒数）、是否引入 key 的定期清理策略、限流 key 的内存预警阈值由 spec-writer 根据 Redis 容量规划确定 |
